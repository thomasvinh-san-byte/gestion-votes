## Failure Analysis

All failures are pre-existing and unrelated to the htmx 2.0.6 upgrade. The htmx surface area is minimal: 2 HTML pages (vote, postsession) with `hx-ext` body attribute, 3 DELETE JS callers using query params, 3 PHP DELETE handlers using `api_query()`. None of the failing tests touch these files or features.

### Category 1: Login form timing (all browsers)

- `auth.spec.js:19` -- login timeout (form submit + redirect)
- `workflow-meeting.spec.js:34` -- invalid credentials error display timing
- `workflow-meeting.spec.js:46` -- operator login redirect timeout
- `workflow-meeting.spec.js:54` -- admin login redirect timeout
- `ux-interactions.spec.js:87` -- Enter key form submission

**Root cause:** Login page uses vanilla JS (zero htmx). Timeouts caused by rate-limiting or Docker container startup latency during parallel test execution.

### Category 2: Operator mode-switch visibility (all browsers)

- `critical-path-operator.spec.js:19` -- `#btnModeSetup` hidden
- `critical-path-president.spec.js:18` -- mode switch hidden
- `page-interactions.spec.js:78` -- refresh button hidden
- `page-interactions.spec.js:94` -- mode switch aria-pressed
- `workflow-meeting.spec.js:121` -- setup tabs visibility
- `operator-e2e.spec.js:31` -- full workflow (cascading from mode switch)

**Root cause:** Operator page uses zero htmx attributes. The `#btnModeSetup` button resolves to a DOM element with `class="mode-switch-btn active"` but Playwright reports it as visually hidden. This is a CSS viewport/overflow issue, not an htmx regression.

### Category 3: Modal/overlay visibility (all browsers)

- `critical-path-archives.spec.js:22` -- `#exportsModal` has `hidden=""` attribute

**Root cause:** The `[hidden]` attribute CSS rule from Phase 02 (overlay hittest). Modal toggle JS does not remove `hidden` fast enough or test does not wait. Not htmx-related.

### Category 4: WebKit-specific (webkit only)

- `accessibility.spec.js:92` -- axe violations on archives, help, hub, report pages
- `audit-regression.spec.js:19,153` -- sidebar overlap, report empty state
- `critical-path-analytics.spec.js:26` -- analytics page (WebKit JS execution)
- `critical-path-meetings.spec.js:19` -- meetings page race condition
- `critical-path-vote.spec.js:20` -- vote page selectors
- `mobile-viewport.spec.js:138` -- tablet members page
- `navigation.spec.js:32` -- admin page load timeout
- `ux-interactions.spec.js:351` -- page load performance (2 tests)

**Root cause:** WebKit has documented slower JS execution and resource pressure issues (see v1.3 15-CROSS-BROWSER-REPORT.md). Pre-existing, not htmx-related.

### Category 5: Firefox-specific (firefox only, beyond common)

- `accessibility.spec.js:38` -- keyboard navigation
- `accessibility.spec.js:92` -- axe help page
- `hidden-attr.spec.js:35` -- settings panel hidden attr

**Root cause:** Firefox-specific timing for keyboard events and axe-core analysis. Pre-existing.

### Category 6: Mobile-chrome-specific (beyond common)

Additional 23 failures on mobile-chrome (35 total - 12 common) are viewport-dependent tests written for desktop layouts, consistent with v1.3 baseline where mobile-chrome had the most failures (21/25 in critical-path subset).

## v1.3 Baseline Comparison

The v1.3 cross-browser report used `--grep @critical-path` (25 specs). This run used the full suite (212 specs). Direct numerical comparison is not meaningful. However:

- v1.3 critical-path failures were in webkit (2/25) and mobile-chrome (4/25)
- v1.4 critical-path failures follow the same pattern (operator mode-switch, archives modal)
- The failure categories are identical: timing, CSS visibility, viewport

**No new failure category introduced by htmx 2.0.6.**
