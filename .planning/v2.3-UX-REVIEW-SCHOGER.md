## 2. Critiques (9 trous identifiés)

### S-1. Cockpit opérateur — 70 boutons en mode exec

*Refactoring UI Ch.1 :* "The most common reason an interface looks busy is treating everything as equally important."

`operator.htmx.html` exec mode totalise **70 boutons** : `btn-primary` ×20, `btn-secondary` ×35, `btn-success` ×5, `btn-ghost` ×7, `btn-danger` ×3. La Phase 1 (`<ag-health-bar>` + raccourcis clavier) a ajouté une couche de hiérarchie mais **n'a rien retiré**.

**Cible :** ≤ 25 boutons visibles permanents. Le reste va en menu contextuel (`...`), drawer, ou hover-revealed. Hors scope strict v2.3 — à backlog v2.4 ou phase dédiée "cockpit declutter".

### S-2. 73 box-shadow + 57 border distinctes — système échappé

*Refactoring UI :* "Just because you can pick any value doesn't mean you should." Schoger recommande **5-6 niveaux d'élévation** consolidés en tokens.

73 box-shadow uniques signifient qu'aucune carte ne ressemble à aucune autre carte au même niveau d'élévation. Bruit visuel difficile à voir consciemment mais qui rend une UI "amateure" en aveugle. Idem 57 border variantes.

**Cible :** ≤ 6 shadow tokens + ≤ 8 border tokens, mappage automatique des occurrences existantes vers ces tokens. Quick task ou phase backlog.

### S-3. 77 padding/margin hardcodés dans login/operator/audit

*Refactoring UI :* "If you don't use the system everywhere, you don't have a system — you have a suggestion."

Top offenders : `operator.css` (73 hardcodes), `pages.css` (42), `login.css`, `audit.css`. Les tokens `--space-*`, `--pad-*`, `--gap-*` existent mais sont contournés.

**Cible :** Audit + remplacement page par page. Naturel à intégrer aux phases v2.3 qui touchent ces pages : Phase 2 → `audit.css`, Phase 3 → `login.css` + `pages.css`.

### S-4. 0 `display: grid` — flex-first dépassé en 2026

*Refactoring UI moderne :* "Grid for layout, flex for content alignment within layout cells."

26 instances `flex`, **0 instance `grid`**. Marche mais demande des hacks (`min-width: 0`, `flex: 1 1 auto`) et empêche les vraies grilles éditoriales (Phase 2 PV) ou tabulaires (audit, archives).

**Cible :** Introduire `display: grid` pour `.ag-editorial` (Phase 2 — colonne contenu + colonne hash/intégrité) et hero+KPI grid (Phase 3 — alignement parfait sans flex-basis games).

### S-5. Login — orbe + grid pattern + 3 features = décoration sur outil

LOGIN-01 (Phase 3) prévoit la suppression de l'orbe ✅. **Mais le grid pattern (`login-brand-grid`) reste non flaggé.**

*Refactoring UI :* "Patterns and textures belong on marketing pages, not on tool entry points where the user wants to do their job."

**Cible :** Amender LOGIN-01 pour supprimer aussi `login-brand-grid` (background-image gradient pattern). Garder le `login-brand-glow` radial atténué (single subtle gradient ≠ pattern).

### S-6. Operator — 6 personas × 5 sémantiques = 11 couleurs sur la même page

*Refactoring UI Ch.5 :* "Use color to draw attention, not to label everything."

11 couleurs distinctes coexistent sur `operator.htmx.html` (6 personas + 5 sémantiques success/danger/warning/info/primary). L'œil ne sait plus ce qui est urgent.

**Cible :** Confiner les couleurs persona à 1-2 surfaces d'identité (badge sidebar, bordure left du nom utilisateur) — pas sur composants fonctionnels (boutons, panneaux). Hors scope strict v2.3 — backlog v2.4.

### S-7. Audit page — 5 filter tabs + table+timeline toggle + drawer

Phase 2 va ajouter Fraunces sur le contenu. Mais les **contrôles** saturent le haut de page (5 filter tabs + toggle table/timeline + drawer + 2 export + 1 help).

*Refactoring UI Ch.10 :* "Every tab is a tax — if not all tabs are commonly used, dropdown them."

**Cible :** Amender EDITORIAL-02 pour auditer les 5 filter tabs. Si 2-3 sont utilisés <5% du temps, les déplacer dans `<details>` "Plus de filtres". Le contenu prend la lumière, pas la chrome.

### S-8. Dashboard — 22 surfaces visuelles (4 KPI + ~~15~~ **3** shortcut-cards + 7 boutons)

> **CORRECTION 2026-04-30 (post-Plan 03.2 audit factuel) :** le chiffre "15 shortcut-cards" dans cette revue est **erroné**. Le scan initial via Explore agent comptait les occurrences du substring `shortcut-card`, qui matchaient aussi les classes BEM enfants (`shortcut-card__icon`, `__text`, `__title`, `__sub`). La réalité du code : **3 shortcut-cards** dans `<aside class="dashboard-aside">` (Créer une séance, Piloter un vote, Consulter le suivi). **S-8 est de facto résolu** par le code existant — DASHBOARD-05 marqué Done sans diff (cf REQUIREMENTS.md ligne 47).

Phase 3 prévoit 4→3 KPI + sunken actions. ~~Mais les 15 shortcut-cards ne sont pas dans le scope.~~ **Mais les 3 shortcut-cards sont déjà sous le seuil de 5 — rien à plier.**

*Refactoring UI :* "Reduce by half, see what breaks. Almost nothing breaks." — appliqué *avant* cette phase, possiblement par v1.9 ou v2.0 lors de précédentes refontes.

**Cible :** Amender DASHBOARD-03 pour auditer les 15 shortcut-cards : grouper par persona, top 5 utilisés en avant, le reste derrière "Toutes les actions".

### S-9. Pages éditoriales (Phase 2) — risque `text-align: center` non verrouillé

*Refactoring UI :* "Centered long-form text is a readability anti-pattern."

Phase 2 prévoit `max-width: 720px` avec Fraunces. **Aucun requirement n'interdit `text-align: center` dans `.ag-editorial`.** Risque : un dev futur centrera "pour rendre joli", massacrant la lecture d'un PV.

**Cible :** Amender EDITORIAL-01 pour exiger `text-align: left` explicite sur `.ag-editorial > *` et le verrouiller par test (CSS lint ou PHPUnit qui scanne les templates).
