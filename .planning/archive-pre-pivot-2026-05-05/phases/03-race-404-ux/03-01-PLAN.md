## ag-empty-state existant (rétrocompat à préserver)

Attributs supportés actuellement : `icon`, `title`, `description`, `action-label`, `action-href`.
Icon values : `meetings | members | votes | archives | generic | none`.
Le composant utilise light DOM (pas de shadow DOM). Le rendu utilise les classes CSS `.empty-state`, `.empty-state-icon`, `.empty-state-title`, `.empty-state-description` déjà définies dans `design-system.css` (lignes 2693-2738).

NOUVEL attribut à ajouter : `variant` (valeurs `resource-deleted | no-data-yet | error`). Quand `variant` est présent, il fournit des **valeurs par défaut** pour `icon` et applique une classe modificateur CSS (`.empty-state--resource-deleted`, etc.) — mais les attributs explicites (`title`, `description`, `action-label`, `action-href`, `icon`) restent toujours prioritaires.

Mapping variant → defaults :
- `resource-deleted` → icon par défaut `generic`, classe modificateur `.empty-state--resource-deleted`
- `no-data-yet` → icon par défaut selon contexte (laissé `generic` par défaut), classe modificateur `.empty-state--no-data-yet`
- `error` → icon par défaut `generic`, classe modificateur `.empty-state--error`

Quand `variant` est absent → comportement actuel inchangé.

## API JSON erreur AgVote (déjà en place)

Toutes les erreurs API renvoient un body JSON avec la forme :
```json
{ "ok": false, "error": "<code_snake_case>", "message": "<français traduit>" }
```
Codes pertinents pour ce plan (déjà définis dans `ErrorDictionary.php`) :
- `meeting_not_found` → "Cette séance est introuvable, vérifiez le lien d'accès ou retournez à la liste des séances."
- `motion_not_found` → "Cette résolution est introuvable, vérifiez l'ordre du jour de la séance."

NE PAS modifier ces messages — le 404-handler doit les **lire et utiliser** comme source de vérité du libellé.

## Hook HTMX à utiliser : `htmx:beforeOnLoad`

Préférer `htmx:beforeOnLoad` à `htmx:responseError` car :
1. `beforeOnLoad` se déclenche AVANT que HTMX décide quoi faire avec la réponse → on peut substituer `e.detail.shouldSwap = true` et `e.detail.serverResponse` pour que HTMX fasse le swap normalement avec notre HTML d'empty-state.
2. `responseError` se déclenche pour status >= 400 mais HTMX skip déjà le swap → il faudrait faire le swap manuellement.
3. Le handler générique existant dans `utils.js` (lignes 339-345) écoute `htmx:responseError` → en interceptant sur `beforeOnLoad` et en absorbant l'event, on évite que le toast générique se déclenche.

Pattern :
```js
document.body.addEventListener('htmx:beforeOnLoad', function(e) {
  const xhr = e.detail.xhr;
  if (xhr.status !== 404) return;
  let body = {};
  try { body = JSON.parse(xhr.responseText); } catch (_) {}
  const code = body && body.error;
  const known = { meeting_not_found: { ... }, motion_not_found: { ... } };
  if (!known[code]) return; // unknown 404 → laisser le handler générique gérer
  // Substituer la réponse pour que HTMX swap notre empty-state
  e.detail.shouldSwap = true;
  e.detail.isError = false; // empêche le passage en responseError + toast
  e.detail.serverResponse = renderEmptyStateHtml(code, body.message);
});
```
Le `isError = false` empêche le déclenchement subséquent de `htmx:responseError`.

    ```js
    /**
     * htmx-404-handler.js — RACE-V27-01
     *
     * Listener global qui intercepte les réponses HTMX 404 dont le body JSON
     * contient un code d'erreur reconnu (meeting_not_found, motion_not_found)
     * et substitue le contenu cible par un &lt;ag-empty-state variant="resource-deleted"&gt;
     * au lieu du toast d'erreur rouge générique.
     *
     * Utilise htmx:beforeOnLoad (avant decision swap/error) plutôt que
     * htmx:responseError pour empêcher le toast générique de se déclencher
     * (e.detail.isError = false).
     */
    (function () {
      'use strict';
      
      var KNOWN_404_CODES = {
        'meeting_not_found': {
          fallbackMessage: 'Cette séance n\'existe plus.',
          ctaLabel: 'Retour aux séances',
          ctaHref: '/dashboard.htmx.html',
        },
        'motion_not_found': {
          fallbackMessage: 'Cette résolution n\'existe plus.',
          ctaLabel: 'Retour à la séance',
          ctaHref: '/operator',
        },
      };
      
      function escapeHtml(s) {
        return String(s == null ? '' : s)
          .replace(/&amp;/g, '&amp;amp;').replace(/&lt;/g, '&amp;lt;').replace(/&gt;/g, '&amp;gt;')
          .replace(/"/g, '&amp;quot;').replace(/'/g, '&amp;#39;');
      }
      
      function renderEmptyStateHtml(code, message) {
        var cfg = KNOWN_404_CODES[code];
        var msg = message || cfg.fallbackMessage;
        return '&lt;ag-empty-state variant="resource-deleted"'
          + ' title="' + escapeHtml(msg) + '"'
          + ' description=""'
          + ' action-label="' + escapeHtml(cfg.ctaLabel) + '"'
          + ' action-href="' + escapeHtml(cfg.ctaHref) + '"&gt;'
          + '&lt;/ag-empty-state&gt;';
      }
      
      document.body.addEventListener('htmx:beforeOnLoad', function (e) {
        var xhr = e.detail &amp;&amp; e.detail.xhr;
        if (!xhr || xhr.status !== 404) return;
        
        var body = null;
        try { body = JSON.parse(xhr.responseText); } catch (_) { return; }
        
        var code = body &amp;&amp; body.error;
        if (!code || !KNOWN_404_CODES[code]) return;
        
        // Hijack the response : tell HTMX to swap our empty-state instead
        e.detail.shouldSwap = true;
        e.detail.isError = false; // empêche htmx:responseError + toast générique
        e.detail.serverResponse = renderEmptyStateHtml(code, body.message);
      });
    })();
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → API HTMX | Le 404-handler JS lit `xhr.responseText` (déjà parsé serveur, mais le JSON peut contenir n'importe quoi côté code-attaquant). On échappe via `escapeHtml()` avant injection dans `serverResponse`. |
| Browser → Custom element | `<ag-empty-state>` reçoit `title`/`description`/`action-href` qui sont des attributs HTML. Le composant utilise `_esc()` (déjà en place ligne 38) pour échapper avant innerHTML. |
| Test endpoint → DB | `/api/v1/test/delete-meeting` est dev-only triple-guarded. En prod = 404. En dev, l'audit_log trace l'invocation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering / XSS | htmx-404-handler.js renderEmptyStateHtml | mitigate | Échappement HTML systématique de `body.message` via `escapeHtml()` avant injection dans `e.detail.serverResponse`. Le composant `ag-empty-state` ré-échappe ensuite via `_esc()` interne (defense in depth). |
| T-03-02 | Information Disclosure | xhr.responseText body JSON | accept | Le body JSON est déjà filtré par `api_fail()` côté serveur — il ne contient que `{ok, error, message, detail?}`. Pas de stack trace en prod (controllée par `APP_ENV`). |
| T-03-03 | Elevation of Privilege | TestSeedController::deleteMeeting | mitigate | Triple-guard : (1) route registration conditionnelle dans `routes.php` sous `if (!in_array($appEnv, ['production', 'prod']))` ; (2) `EnvGuardMiddleware` via `['env_guard' =&gt; true]` ; (3) `$this-&gt;guardProduction()` interne. Audit log obligatoire. Pattern identique à `seedMeeting()`. |
| T-03-04 | Denial of Service | deleteForTest sans limit rate | accept | Endpoint dev-only inaccessible en prod (404 garanti par triple-guard). En dev, attaque triviale impossible — environnement maîtrisé. |
| T-03-05 | Spoofing | Faux 404 envoyé par MITM avec code arbitraire | accept | TLS obligatoire en prod (HSTS). Si MITM réussit le TLS, l'attaque est largement plus grave qu'un swap d'empty-state. La whitelist `KNOWN_404_CODES` (2 codes seulement) limite la surface — un code inconnu fait fall-through vers le handler générique (toast neutre). |
