**Critical:** `api_ok()` throws — does NOT return. The `set_exception_handler` in `app/api.php` line 19-33 catches `ApiResponseException` and calls `$e->getResponse()->send()`. So `HttpCache::respondWithEtag()` must follow the same pattern — throw an `ApiResponseException` with a 304 JsonResponse.

Pattern to implement in HttpCache:
```php
namespace AgVote\Core\Http;

final class HttpCache {
    /**
     * Deterministic ETag for an arbitrary payload. Stable across requests with same data.
     * Uses md5(json_encode(...)) — fast, no crypto need (ETag is a cache key, not a signature).
     * Sorted-key serialization NOT required if PHP's json_encode produces stable order
     * for associative arrays (it does, since PHP arrays are ordered).
     */
    public static function etagFor(array $payload): string {
        return '"' . md5(json_encode($payload, JSON_UNESCAPED_UNICODE)) . '"';
    }

    /**
     * If client If-None-Match matches the computed ETag for $payload, throw a 304 response.
     * Otherwise return — caller proceeds to api_ok($payload), which will (separately) emit
     * ETag + Cache-Control headers via emitHeaders() called from api_ok wrapper or via
     * controller adding headers manually.
     *
     * Recommended use: call BEFORE api_ok() so 304 short-circuits before any extra work.
     *   $payload = [...computed data...];
     *   HttpCache::respondWithEtag($payload);  // throws 304 OR returns
     *   api_ok($payload);                      // 200 OK with ETag header
     *
     * For the 200 case, we also need to set the ETag header on the JsonResponse.
     * Use HttpCache::sendOk($payload) as a wrapper that does both:
     *   - calls etagFor + checks If-None-Match → throws 304 if match
     *   - else emits ApiResponseException(JsonResponse::ok($payload, 200, headers))
     */
    public static function sendOk(array $payload): never {
        $etag = self::etagFor($payload);
        $clientEtag = trim((string) ($_SERVER['HTTP_IF_NONE_MATCH'] ?? ''));

        if ($clientEtag !== '' && $clientEtag === $etag) {
            // 304 Not Modified — empty body, must include ETag + Cache-Control per RFC 7232
            throw new ApiResponseException(new JsonResponse(
                304,
                [],   // empty body — but JsonResponse::send echoes json_encode([]) = "[]" ; that's fine for HTMX
                [
                    'ETag' => $etag,
                    'Cache-Control' => 'private, must-revalidate',
                ],
            ));
        }

        // Otherwise 200 OK with ETag headers
        throw new ApiResponseException(new JsonResponse(
            200,
            ['ok' => true, 'data' => $payload],
            [
                'ETag' => $etag,
                'Cache-Control' => 'private, must-revalidate',
            ],
        ));
    }
}
```

**Note on JsonResponse::send() for 304:** the current implementation always echoes `json_encode($body)`. For 304 RFC compliance, body MUST be empty. Either:
- (a) Patch JsonResponse::send() to skip body for status 304, OR
- (b) Accept that JsonResponse::send echoes "[]" for our 304 (negligible — HTMX ignores body on 304).
Choose (a) — small targeted patch in JsonResponse::send to skip body+content-type for 304/204. Add a test for this.
