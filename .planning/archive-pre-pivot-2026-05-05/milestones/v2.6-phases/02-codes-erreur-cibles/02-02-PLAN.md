<objective>
Lever ERR-V26-02 : rendre `ErrorEventsRepository::capture()` idempotent dans le scope d une requete HTTP, pour que les rafales d events SSE empty-state (un meme handler peut etre invoque 2-3 fois sur des reconnexions / retries) n inserent qu une seule ligne en base par (request_id, error_code, route).

**Contexte concret.** Le ROADMAP Phase 2 success-criteria #2 demande :
> "Un test cible (PHPUnit ou spec Playwright) emet 2 requetes back-to-back sur la meme ressource vide et verifie : meme code de retour, meme payload, aucun etat corrompu (compteur d audit incremente <=1 fois, pas de double-event SSE)."

Le code generique `business_error` n est pas le seul concerne — tout `api_fail()` declenche `ErrorEventsRepository::capture()` (cf. `app/api.php:48-68`), et un meme handler appele plusieurs fois dans la meme requete (ex: middleware retry, SSE handler invoque par event-bus + par catch-all) inscrit une ligne par appel. Le dashboard `/admin/error-stats` voit alors un compte gonfle artificiellement.

**Strategie.** Guard in-memory pur, pas de Redis, pas de DB :

1. Le repo expose une property statique private `$seenKeys` (array de hash) initialisee vide a chaque construction de RepositoryFactory (qui est singleton-per-request via `RepositoryFactory::getInstance()` — donc la duree de vie est strictement la duree de la requete HTTP).
2. `capture()` calcule une cle `md5($requestId . '|' . $errorCode . '|' . $route)` au debut. Si cette cle existe deja dans `$seenKeys`, on SKIP l INSERT (early return apres un Logger::debug optionnel). Sinon on l ajoute puis on insere.
3. Si `$requestId` est null OU si `$route` est null, on SKIP le guard (insert direct comme avant) — preserve le comportement pour les invocations CLI / jobs / bootstrap qui n ont pas de request id.

**Pourquoi pas un guard DB (UNIQUE constraint).** Un index unique sur `(request_id, error_code, route)` aurait l avantage d etre robuste cross-process, mais (a) `error_events` est best-effort (cf. comment ligne 45-47), (b) la table prend des inserts haute frequence et un UNIQUE constraint avec retry serait couteux, (c) les tests existants ne s y attendent pas. In-memory satisfait le success criteria (back-to-back same request) et zero risque de regression en prod.

Purpose: Lever ERR-V26-02 sans modifier le schema BDD ni introduire une dependance Redis specifique. Test PHPUnit dedicacé verifie le contrat.

Modifier la methode `capture()` pour la rendre idempotente. Voici le corps cible (remplacer le body actuel) :

```
public function capture(
    string $errorCode,
    int $httpStatus,
    ?string $tenantId,
    ?string $userId,
    ?string $route,
    ?string $method,
    ?string $requestId,
    array $payload,
): void {
    // Idempotency guard (ERR-V26-02): skip if same (request_id, error_code, route)
    // already captured in this HTTP request. Skip the guard entirely if we don't
    // have both a request_id and a route — in that case dedup isn't well-defined
    // (CLI jobs, bootstrap hooks, etc.).
    if ($requestId !== null && $requestId !== '' && $route !== null && $route !== '') {
        $key = md5($requestId . '|' . $errorCode . '|' . $route);
        if (isset(self::$captureSeenKeys[$key])) {
            return;
        }
        self::$captureSeenKeys[$key] = true;
    }

    try {
        $this->execute(
            'INSERT INTO error_events
                (request_id, tenant_id, user_id, error_code, http_status, route, method, payload)
                VALUES (:rid, :tid, :uid, :code, :status, :route, :method, :payload::jsonb)',
            [
                ':rid' => $requestId,
                ':tid' => $tenantId,
                ':uid' => $userId,
                ':code' => $errorCode,
                ':status' => $httpStatus,
                ':route' => $route,
                ':method' => $method,
                ':payload' => json_encode($payload, JSON_UNESCAPED_UNICODE | JSON_UNESCAPED_SLASHES),
            ],
        );
    } catch (Throwable $e) {
        // Capture must never break api_fail. Log the failure and move on —
        // missing one row is acceptable, breaking the user's request is not.
        Logger::warning('error_events capture failed', [
            'exception' => $e->getMessage(),
            'error_code' => $errorCode,
        ]);
    }
}
```

namespace Tests\Unit;

use AgVote\Repository\ErrorEventsRepository;
use PDO;
use PDOStatement;
use PHPUnit\Framework\TestCase;

final class ErrorEventsCaptureIdempotencyTest extends TestCase {
    private PDO $pdo;
    private PDOStatement $stmt;
    private int $executeCount = 0;
    private ErrorEventsRepository $repo;

    protected function setUp(): void {
        $this->pdo = $this->createMock(PDO::class);
        $this->stmt = $this->createMock(PDOStatement::class);

        // Compter les invocations execute()
        $this->executeCount = 0;
        $this->stmt->method('execute')->willReturnCallback(function () {
            $this->executeCount++;
            return true;
        });

        $this->pdo->method('prepare')->willReturn($this->stmt);

        // Adapter selon la signature reelle d AbstractRepository
        $this->repo = new ErrorEventsRepository($this->pdo);

        // Reset du cache statique pour eviter les fuites entre tests
        ErrorEventsRepository::resetIdempotencyCache();
    }

    protected function tearDown(): void {
        ErrorEventsRepository::resetIdempotencyCache();
    }

    public function test_back_to_back_same_key_inserts_once(): void {
        $this->repo->capture('archived_meeting_locked', 400, null, null, '/api/v1/foo', 'POST', 'req-abc', []);
        $this->repo->capture('archived_meeting_locked', 400, null, null, '/api/v1/foo', 'POST', 'req-abc', []);
        $this->assertSame(1, $this->executeCount, '2 captures back-to-back avec meme (rid, code, route) doivent inserer 1 seule fois');
