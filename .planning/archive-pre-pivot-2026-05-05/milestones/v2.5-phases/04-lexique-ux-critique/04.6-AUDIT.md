## Cadre Norman appliqué

Pour chaque code, la question :

> *"Peut-on faire disparaître cette erreur par contrainte UI plutôt que de la rattraper ?"*

> Norman : *constraints > error messages*. Désactiver un bouton ou masquer un champ supprime la classe d'erreur entière, plutôt qu'afficher un message après coup.

Quatre verdicts possibles :
- **Prévenu en v2.3** : un fix UI minimal ELIMINE la classe d'erreur. Fix livré dans le commit.
- **Prévenu par design v2.3** : la prévention est déjà en place dans le codebase v2.3 (audit le confirme). Le rattrapage serveur reste comme filet de sécurité. Rationale écrite.
- **Déféré v2.4 (race fondamentale)** : aucune contrainte UI ne peut éliminer la classe (race condition serveur, état distribué). Rattrapage serveur indispensable.
- **Déféré v2.4 (refonte structurelle)** : prévention possible mais hors-scope v2.3 (refonte architecturale, décomposition en sous-codes, audit business à faire).

## 1. `meeting_not_found` (49 émissions)

**Cause racine.** Émis quand un `meeting_id` UUID syntaxiquement valide pointe vers une séance inexistante, supprimée, ou hors-tenant courant. Causes typiques :
- Lien stale (l'utilisateur clique sur un lien stocké d'une séance entretemps supprimée).
- Bascule de tenant : l'utilisateur a changé de tenant courant et l'UUID en cache est devenu hors-périmètre.
- Course entre affichage de liste et clic : un autre administrateur supprime la séance.
- URL trafiquée à la main (UUID fabriqué).

**Sites d'émission notables (échantillon).**
- `app/Controller/AgendaController.php` (4 sites — agenda, présidents, suppléants)
- `app/Controller/AuditController.php` (6 sites — accès aux events de séance)
- `app/Controller/QuorumController.php` (2 sites)
- `app/Controller/TrustController.php` (2 sites)
- `app/Controller/MeetingWorkflowController.php` (3 sites — ouvrir/clôturer/valider)
- `app/Controller/SpeechController.php`, `app/Controller/VoteTokenController.php`, `app/api.php`
- (Liste complète : `grep -rn "api_fail(['\"]meeting_not_found['\"]" app/ public/`.)

**Préventible UI ?** **Non — race condition fondamentale.**

Une séance peut être supprimée par un autre opérateur entre le moment où la liste est affichée à l'utilisateur et celui où il clique. Aucune contrainte UI ne peut éliminer cette classe : l'état "séance existe" est partagé, distribué, et changeable par un autre acteur.

**Décision : déféré v2.4 (race fondamentale).** Pas de fix UI possible. Le message enrichi par 04.4 ("Cette séance est introuvable, vérifiez le lien d'accès ou retournez à la liste des séances.") guide correctement vers la résolution. Le filet serveur fait son travail.

**Optimisation possible v2.4** (hors scope ici) : SSE `meeting:deleted` event qui retire en temps réel les liens vers la séance supprimée dans la liste de l'utilisateur. Réduit la fenêtre de race mais ne l'élimine pas (l'utilisateur peut avoir un onglet inactif).

## 2. `missing_meeting_id` (35 émissions)

**Cause racine.** Paramètre `meeting_id` totalement absent de la requête HTTP. Vient théoriquement d'un trigger UI qui ne passe pas le contexte (bouton sans data-attribute), d'un lien malformé, ou d'un appel programmatique direct (curl, test).

**Sites d'émission notables (échantillon).**
- `app/Controller/AuditController.php:16, 199, 222, 287` (4 sites)
- `app/Controller/MeetingsController.php` (5 sites — list/get/update/delete)
- `app/Controller/EmailController.php` (3 sites)
- `app/Controller/MeetingAttachmentController.php` (3 sites)
- `app/Controller/ReminderController.php`, `InvitationsController`, `ProxiesController`, `AttendancesController`, `OperatorController`, `TrustController`, `ExportController`, `MotionsController`, `ImportController`, `MeetingReportsController`, `ResolutionDocumentController`, `MeetingWorkflowController`, `ProcurationPdfController`, `DashboardController` (1-2 sites chacun)
- (Liste complète : `grep -rn "api_fail(['\"]missing_meeting_id['\"]" app/ public/`.)

**Préventible UI ?** **Oui — et déjà prévenu par design v2.3.**

L'audit du codebase frontend montre qu'une **architecture de prévention est déjà en place** :

1. **`public/assets/js/services/meeting-context.js`** — singleton `MeetingContext` cristallise un meeting_id unique pour tout l'écran. Sources de résolution (priorité descendante) : URL param → URL clean path UUID → sessionStorage → `<input name="meeting_id">` caché. Si aucune source ne fournit, `MeetingContext.get()` retourne `null` et **les pages-clients refusent de procéder**.

2. **Garde précoce dans les pages sensibles** :
   - `validate.js:7-10` : `if (!currentMeetingId) { setNotif('error', 'Aucune séance sélectionnée'); setTimeout(() => window.location.href = '/meetings', 2000); }` — redirige automatiquement vers la liste de séances.
   - `vote.js`, `report.js`, `trust.js`, `audit.js`, `postsession.js`, `archives.js`, `hub.js` — pattern systématique `if (!meetingId) return;` avant tout fetch API.

3. **Liens et boutons HTMX** — la majorité des CTA ouvrant une page contextuelle naviguent via URL (`/audit?meeting_id=...`) construite à partir de `MeetingContext` ou d'une row de liste. Pas de bouton qui POST `/api/v1/...` sans contexte préalable.

**Conséquence :** les 35 émissions serveur subsistent comme **filet de sécurité défensif** pour :
- Appels programmatiques (curl, scripts internes, tests d'intégration).
- Race condition early-call : un script autoload se déclenche avant que `MeetingContext.init()` ait fini.
- Anciens links externes vers `/api/v1/...` sans paramètre.
- Tentatives malveillantes (URLs forgées).

Aucun de ces 4 cas n'est un trigger UI normal en v2.3. Un utilisateur cliquant sur l'UI ne déclenche **jamais** ce code organiquement (sauf bug à corriger via Rule 1).

**Changement UI minimal envisageable.** Aucun n'est nécessaire — l'architecture est déjà optimale pour ERR-04. Si un jour un nouveau bouton est ajouté sans data-attribute meeting_id, la guideline est explicite (`MeetingContext.get()` doit être consulté avant tout appel API).

**Décision : prévenu par design v2.3 (rationale écrite + audit codebase).** Pas de fix livré dans le commit — la prévention est déjà cristallisée dans `MeetingContext` (architecture singleton) et dans le pattern systematic `if (!meetingId) return;` documenté ci-dessus. ERR-04 satisfait par construction. Le filet serveur reste comme garde-fou défensif (comportement souhaitable).

## 3. `motion_not_found` (7 émissions)

**Cause racine.** Analogue à `meeting_not_found` mais pour une résolution. Émis quand un `motion_id` UUID valide pointe vers une motion inexistante, supprimée, ou non rattachée à la séance courante.

**Sites d'émission.**
- `app/Controller/BallotsController.php:32, 157, 279` (3 sites — coeur du flow vote)
- `app/Controller/MotionsController.php:110, 305` (2 sites)
- `app/Controller/VoteTokenController.php:37`
- `app/Controller/ResolutionDocumentController.php:44`

**Préventible UI ?** **Non — même race condition que `meeting_not_found`.**

Une motion peut être supprimée par l'opérateur entre l'affichage du bulletin de vote et la soumission par le votant. Race fondamentale entre l'état affiché et l'état serveur, partageable par plusieurs acteurs concurrents.

**Décision : déféré v2.4 (race fondamentale).** Le message enrichi 04.4 ("Résolution introuvable, vérifiez la liste des résolutions ou demandez à un administrateur de la rouvrir.") guide correctement. Pas de fix UI possible.

**Note prospective v2.4** (hors scope) : SSE `motion:deleted` retirerait en live les bulletins ouverts. Réduit la fenêtre de race.

## 4. `invalid_meeting_id` (11 émissions)

**Cause racine.** Le paramètre `meeting_id` est présent mais n'est pas un UUID syntaxiquement valide. Causes possibles :
- Saisie manuelle dans une URL trafiquée (`?meeting_id=abc`).
- Trigger UI cassé qui passe une string brute au lieu d'un UUID.
- Test d'intégration incorrect.
- Appel programmatique avec mauvais format.

**Sites d'émission.**
- `app/Controller/OperatorController.php:43, 112` (2 sites)
- `app/Controller/MeetingReportsController.php:68, 112, 150` (3 sites)
- `app/Controller/QuorumController.php:98`, `VoteTokenController.php:20`, `ProcurationPdfController.php:46`, `MeetingsController.php:265`, `InvitationsController.php:24`, `AttendancesController.php:48`

**Préventible UI ?** **Oui — et déjà prévenu par design v2.3.**

L'audit du codebase confirme qu'**aucune saisie manuelle d'un meeting_id n'est offerte dans l'UI v2.3** :

1. `grep -rn "input.*meeting_id\|name=\"meeting_id\"" public/*.html` retourne 0 hit (aucun `<input>` visible pour saisir un UUID).
2. Tous les meeting_id en circulation viennent d'une de ces 3 sources (cf. `meeting-context.js`) :
   - URL pré-construite à partir d'une row cliquée dans la liste serveur (UUID toujours valide).
   - sessionStorage (UUID hérité d'une page précédente, valide à la sauvegarde).
   - URL `clean` matchée par regex stricte `/[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}/i`.
3. Le seul `<input name="meeting_id">` du codebase est un champ caché peuplé par le serveur (cf. `meeting-context.js:40`), pas un champ utilisateur.

**Conséquence :** un utilisateur cliquant sur l'UI ne peut **pas** déclencher organiquement ce code en v2.3. Les 11 émissions serveur sont un filet de sécurité pour :
- URLs trafiquées à la main (recherche-engine bot, tentative malveillante, copy-paste manuel partiellement corrompu).
- Tests d'intégration intentionnels.
- Appels programmatiques externes.

**Décision : prévenu par design v2.3 (rationale écrite + audit codebase).** Pas de fix livré — l'absence totale de saisie UI manuelle d'UUID est confirmée par grep statique. ERR-04 satisfait par construction. Le filet serveur reste comme garde-fou défensif (comportement souhaitable).

## 5. `business_error` (11 émissions)

**Cause racine.** Code générique catch-all pour `RuntimeException` métier. Émis dans des contextes très hétérogènes :
- `MeetingWorkflowController` (3 sites — ouvrir/clôturer/valider une séance avec préconditions non remplies)
- `MeetingsController` (6 sites — create/update/delete avec contraintes business)
- `AbstractController:50` (catch-all final pour toute `RuntimeException` non gérée par un controller spécifique)

À chaque émission, le `detail` (message exception) varie — c'est un sac fourre-tout dont la sémantique dépend de chaque cas.

**Préventible UI ?** **Non — par nature.**

Le code `business_error` couvre une famille hétérogène d'erreurs dont chacune aurait sa propre stratégie de prévention :
- `business_error` "séance non en setup" → préventible par bouton désactivé sur les rows non-setup.
- `business_error` "déjà des votes lancés" → préventible par lock UI.
- `business_error` "tenant inactif" → préventible par middleware avant rendu.

Mais on ne peut pas faire un seul fix UI universel pour `business_error`. **La vraie action de prévention est de décomposer le code** en codes plus précis (`workflow_not_in_setup`, `votes_already_started`, `tenant_inactive`, etc.), puis pour chacun appliquer le cadre Norman individuellement.

**Décision : déféré v2.4 (refonte structurelle).** Action recommandée v2.4 :
1. Auditer les 11 sites d'émission de `business_error`.
2. Lister les ~5-8 sous-cas business distincts.
3. Créer un sous-code dédié pour chaque dans `ErrorDictionary` (avec next-step Norman).
4. Pour chaque sous-code, refaire un mini-audit ERR-04 individuel.

Cette décomposition est l'équivalent de ce que ERR-01 a fait sur le top 50 (enrichir avec next-step), mais cette fois en **divisant un code monolithique en sous-codes spécifiques**. Trop large pour le scope v2.3.

**Note transition.** Les 11 émissions actuelles renvoient déjà un `detail` (`['detail' => $e->getMessage()]`) qui donne du contexte. Le message FR du code générique reste générique mais le `detail` apporte la spécificité runtime. Comportement acceptable en attendant la décomposition.

### Critère ERR-04

Le ROADMAP §SC #7 exige : *"Au moins 2 sur 5 sont marqués 'prévenu en v2.3' (ou 'reporté v2.4 avec rationale écrite')."*

**Résultat : 5/5 satisfait.**
- 2/5 marqués "prévenu par design v2.3" avec rationale écrite + audit codebase prouvant le pattern (`missing_meeting_id`, `invalid_meeting_id`).
- 3/5 marqués "déféré v2.4" avec rationale claire (`meeting_not_found` race, `motion_not_found` race, `business_error` refonte structurelle).

Tous les codes ont une décision écrite avec justification — le seuil minimum (2/5) est dépassé.

### Fixes UI livrés v2.3

**Aucun fix UI à livrer dans le commit Plan 04.6.** Les 2 codes "prévenus par design v2.3" le sont par l'architecture déjà en place (cristallisée par 04.4 + travaux antérieurs : `MeetingContext` singleton, gardes systematic `if (!meetingId) return;`, absence d'`<input>` UUID manuel). L'audit Plan 04.6 documente cette prévention pour figer le pattern et orienter les futurs développements.

### Followups v2.4 capturés

1. **`business_error` decomposition** : auditer les 11 sites, créer ~5-8 sous-codes spécifiques, refaire mini-audit Norman par sous-code. Hors-scope v2.3 (refonte structurelle).
2. **SSE `meeting:deleted` / `motion:deleted`** : réduit la fenêtre de race condition pour `meeting_not_found` et `motion_not_found`. Coût modéré, bénéfice UX. Hors-scope v2.3 (event-bus à étendre).
3. **Threshold émission ERR-04** : le ranking statique grep est un proxy. Une stat runtime (compteur Redis ou logs sur 30j) donnerait un ranking plus fidèle. Hors-scope v2.3.
4. **Validation client UUID** : ajouter une validation côté `MeetingContext.set()` (regex UUID) qui refuse silencieusement un UUID malformé. Réduit le filet serveur `invalid_meeting_id` à 0 émission organique. Faisable en quick task post-Phase 4 si remontée terrain.

### Méthodologie pour audits ERR-04 futurs

1. Snapshot grep statique au début de chaque milestone : `grep -rn "api_fail(['\"]<CODE>['\"]" app/ public/` pour les top N codes ranking.
2. Pour chaque code, classer parmi les 4 verdicts (prévenu / prévenu par design / déféré race / déféré refonte).
3. Si "prévenu" : livrer fix UI minimal dans le même commit. Si "prévenu par design" : auditer le codebase pour prouver la prévention en place, écrire la rationale.
4. Si "déféré" : rationale écrite (race fondamentale ou refonte structurelle) + capture en followup pour le milestone suivant.
5. Cible minimum : ≥2/5 des codes top 5 marqués "prévenu" (au sens large : prévenu, prévenu par design, ou déféré avec rationale écrite).
