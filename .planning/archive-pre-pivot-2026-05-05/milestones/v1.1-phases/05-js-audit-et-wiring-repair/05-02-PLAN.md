**shared.js changes:**

In the `initSidebar()` function, dispatch a custom event `sidebar:loaded` on `window` after both the success and failure paths of the fetch:

1. In the `.then()` success handler (after the existing `sidebar.innerHTML = html;` and meeting context wiring), add:
```javascript
window.dispatchEvent(new CustomEvent('sidebar:loaded', { detail: { ok: true } }));
```

2. In the `.catch()` failure handler (after the existing fallback innerHTML), add:
```javascript
window.dispatchEvent(new CustomEvent('sidebar:loaded', { detail: { ok: false } }));
```

This gives shell.js, auth-ui.js, and page scripts a reliable signal that sidebar loading is settled (success or failure).

**shell.js changes:**

Wrap the auth-ui.js injection (currently lines ~904-907, at the very end of the IIFE) in a try/finally block to ensure auth-ui.js loads even if earlier code in the same scope throws:

Before (current):
```javascript
  window.Shell = { beforeNavigate: beforeNavigate };

  // Auto-load auth UI banner
  const authScript = document.createElement('script');
  authScript.src = '/assets/js/pages/auth-ui.js';
  document.head.appendChild(authScript);
})();
```

After (hardened):
```javascript
  window.Shell = { beforeNavigate: beforeNavigate };

  // Auto-load auth UI banner — wrapped in try/finally to guarantee injection
  // even if earlier shell.js code throws (pitfall: auth banner missing silently)
  try {
    const authScript = document.createElement('script');
    authScript.src = '/assets/js/pages/auth-ui.js';
    document.head.appendChild(authScript);
  } catch (e) {
    // Log but do not block — auth-ui.js is non-critical for page rendering
    console.warn('[shell] auth-ui.js injection failed:', e);
  }
})();
```

Note: The `try` here is a safety net. Since the injection is at the end of the IIFE, if any earlier statement throws uncaught, the entire IIFE aborts. The real fix is defensive — if the 3 lines themselves throw (unlikely), we catch and warn. The primary protection against earlier failures is that shell.js is well-structured with each function in its own scope.

Do NOT change the MutationObserver logic, the `bindPinButton()` function, or any handler binding code. Only touch the two specific locations described above.

```javascript
// @ts-check
/**
 * waitForHtmxSettled — Playwright helper for HTMX page settle detection.
 *
 * Only 2 pages use HTMX (vote.htmx.html, postsession.htmx.html).
 * On all other pages, window.htmx is undefined and this resolves immediately.
 *
 * HTMX 1.9.12 fires htmx:afterSettle after DOM settling (defaultSettleDelay: 20ms).
 */

/**
 * Wait for HTMX to finish settling the DOM after a swap.
 * Resolves immediately if HTMX is not loaded on the page.
 *
 * @param {import('@playwright/test').Page} page - Playwright page object
 * @param {number} [timeout=5000] - Maximum wait time in ms
 */
async function waitForHtmxSettled(page, timeout = 5000) {
  await page.waitForFunction(
    () => {
      return new Promise((resolve) => {
        // If HTMX is not loaded on this page, resolve immediately
        if (!window.htmx) { resolve(true); return; }
        // Listen for the next afterSettle event
        document.body.addEventListener(
          'htmx:afterSettle',
          () => resolve(true),
          { once: true }
        );
        // Safety timeout — if no HTMX activity is in flight, resolve after 200ms
        setTimeout(() => resolve(true), 200);
      });
    },
    { timeout }
  );
}

module.exports = { waitForHtmxSettled };
```

This follows the same pattern as the existing `helpers.js` (CommonJS exports, plain async functions, JSDoc types).
