## ❌ THEATRE — /settings (cas le plus grave)

**Verdict:** **Page a moitie un theatre.** Sur 14 fields visibles :
- ✅ **8 fields fonctionnent** : tous SMTP-related (`settSmtpHost`, `settSmtpPort`, `settSmtpUser`, `settSmtpPass`, `settSmtpTls`, `settSenderEmail`, `settSenderName`) + `settSessionTimeout`. Ces valeurs sont persistees ET lues par MailerService et la session middleware.
- ❌ **6 fields DEAD** : la valeur est sauvegardee en DB mais **personne ne la lit**. L'utilisateur clique Save, voit le toast vert, et l'application n'a pas change de comportement.

| Field DEAD | Section | Impact |
|-----------|---------|--------|
| `settVoteMode` | Regles vote | Mode public/secret reglable mais ignore par VoteEngine |
| `settQuorumThreshold` | Regles vote | Seuil de quorum ignore par QuorumEngine |
| `settMajority` | Regles vote | Type de majorite ignore par calcul de decision |
| `settMaxLoginAttempts` | Securite | Rate limit hardcode dans nginx, ignore le setting |
| `settPasswordMinLength` | Securite | Validation password sans contrainte |
| `settHighContrast` | Accessibilite | Toggle sans effet sur le CSS |

**Code path manquant** : `SettingsRepository::get($tenantId, 'settVoteMode')` n'est appele nulle part dans `app/Services/`. Le vote engine utilise des constantes hardcodees au lieu de lire la config tenant.

**Severity** : **CRITIQUE pour les regles de vote** (mode/quorum/majority sont des params metier essentiels). Mineur pour le reste.

**Boutons orphelins** : aucun (tous wires correctement). Le bug est dans le contrat backend, pas dans le wiring UI.

**Verdict:** Le shell + 95% des features marchent. Mais **3 boutons appelent des endpoints inexistants** :

| Feature | Bouton | Endpoint appele | Statut |
|---------|--------|----------------|--------|
| Telecharger PDF procuration | `<a download>` sur chaque proxy | `/api/v1/procuration_pdf.php` | **404** — fichier supprime ou jamais cree |

**Tous les autres operator features marchent** : creation reunion, ajout membres, mode switch (Setup/Exec), action bar refresh, vote launch, vote close, presence tracking, etc.

## Cross-Page Issues

### Hotfixes deja delivres dans v1.2 (qui empechaient le moindre test reel)

1. **`ddcf02f2`** — RateLimiter::configure() boot regression : app crashait sur chaque API call
2. **`2f441469`** — nginx clean URL routing : `/dashboard` etc servait `index.html` au lieu de `dashboard.htmx.html`

4. **Phase 9 cookie domain fix** : tests Playwright bloques par bug de domain
5. **Phase 9 HSTS preload `.app` fix** : Chromium force-upgradait `http://app:8080` en HTTPS

**Sans ces fixes, RIEN ne pouvait etre teste.** L'app etait inutilisable et les tests automatises ne pouvaient pas tourner.

### Problemes architecturaux releves

1. **Settings = key/value libre sans schema** : `tenant_settings` accepte n'importe quelle clef, n'a pas de allowlist, pas de validation. C'est ce qui permet aux fields DEAD d'avoir l'air de fonctionner (UPSERT reussit toujours).
2. **Pas de "is wired" enforcement** : aucun outil ne verifie automatiquement que chaque champ visible dans une UI est lu par le code metier.
3. **Endpoints fantomes** : 5 endpoints appeles par le frontend n'existent pas. Aucun lint backend ne catch ca.

## Reponse a la question "faut-il migrer en React/Laravel ?"

**NON.** Voici pourquoi :

1. **Le wiring UI casse n'est pas un probleme de framework.** Les 5 endpoints fantomes seraient toujours fantomes en React. Les 6 settings DEAD seraient toujours DEAD en Laravel (parce que l'application metier ne les lit pas, et aucun framework n'ecrit ce code automatiquement).

2. **Le backend est solide.** Voting engine, quorum, RBAC, audit trail, RGPD, SSE temps reel, exports XLSX, emails — tout fonctionne. Migrer en Laravel = reecrire 150+ tests PHPUnit + 50+ controllers + 30+ repositories pour ZERO gain.

3. **Le design system est OK.** `design-system.css` (5278 lignes, OKLCH, @layer) est mieux que ce que la plupart des projets React ont. Tailwind/Bootstrap ne te donnerait pas mieux.

4. **Le vrai probleme est la discipline d'execution** : "shows cards" a ete accepte comme "done" sans test E2E qui prouve que la save persiste ET impacte le comportement metier. Aucun framework ne corrige ca — seul un changement de definition de "done" le fait.
