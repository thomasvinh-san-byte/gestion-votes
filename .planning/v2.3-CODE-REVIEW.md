## 2. Warnings — 2

### WR-001 — `operator-keybindings.js` document-level listeners sans cleanup

**Fichier** : `public/assets/js/pages/operator-keybindings.js:115, 122, 152`

3 listeners attachés à `document` (keydown overlay Escape `:122`, keydown dispatcher `:152`, click overlay outside `:115`) sans `removeEventListener` correspondant. Pas de `disconnectedCallback`/cleanup symétrique.

**Risque** : si la page principale est rechargée via HTMX swap (`hx-target="body"` ou similaire) sans full page reload, le module se ré-évalue et ré-attache → duplications progressives → pression mémoire + double dispatch.

**Mitigation actuelle** : le projet utilise visiblement plein refresh sur la navigation principale (`window.location.href` dans plusieurs JS pages, `<a href>` natifs). Donc en pratique **0 leak observé**. Risque latent uniquement si on adopte un HTMX SPA-mode futur.

**Recommandation** : ajouter un sentinel `if (window.__agKeybindingsLoaded) return; window.__agKeybindingsLoaded = true;` au début de l'IIFE OU stocker les handlers dans des constantes pour pouvoir les retirer si jamais la page est swappée.

**Sévérité** : warning. Pas bloquant pour ship.

### WR-002 — `<ag-integrity-modal>` keydown listener pas explicitement nettoyé

**Fichier** : `public/assets/js/components/ag-integrity-modal.js`

Le `connectedCallback` appelle `_bindKeyboard()` qui ajoute un listener Escape sur `document`. Si `disconnectedCallback` ne retire pas ce listener (vérifier l'implémentation complète), il fuit lors d'un retrait DOM dynamique.

**Vérification** : si la modale n'est jamais détachée (markup HTML statique injecté via HtmlView::render et conservé toute la vie de la page), il n'y a pas de leak. Mais si HTMX swap remplace la zone contenant `<ag-integrity-modal>`, le listener orphelin reste actif.

**Mitigation** : pattern identique à WR-001 — projet en plein refresh, faible risque pratique. Future-proof : ajouter `disconnectedCallback() { document.removeEventListener('keydown', this._onKeyDown); }` si pas déjà présent.

**Sévérité** : warning. Pas bloquant pour ship.

## 3. Info — 6 observations positives

### IN-001 — `<ag-health-bar>` HTML construction XSS-safe

`public/assets/js/components/ag-health-bar.js:81-97` — toutes les chaînes user-injectées (quorumState, quorumRatio, sseState, sseLabel, votesRemaining, motionNumber, motionTitle) passent par `_esc()` avant string concat dans `innerHTML`. Pattern défensif modèle pour custom elements vanilla JS.

### IN-002 — `<ag-integrity-modal>` defense-in-depth XSS

`public/assets/js/components/ag-integrity-modal.js:74-119` — 4+ appels `_escape()` (date + ev.hash + ev.prev + ev.at) avant insertion innerHTML. Le `JSON.parse` sur `data-events` est intrinsèquement safe (pas d'eval). Pattern complet : data-attribute → JSON.parse → field-by-field escape → innerHTML. Pas de raccourci unsafe.

### IN-003 — Auto-hidden défensif (B-2 fix iter 2)

`public/assets/js/components/ag-integrity-modal.js:38-42` — `connectedCallback` force `hidden` au démarrage avec garde `data-keep-open` pour les overrides explicites. Empêche le flash visible avant que le CSS `[hidden]` s'applique. Bonne discipline.

### IN-004 — `ag-empty-state.js` extension rétrocompat préservée

`public/assets/js/components/ag-empty-state.js:54, 65-68` — l'extension F2 (`if (icon !== 'none')`) **préserve** le fallback `EMPTY_SVG[icon] || EMPTY_SVG.generic` pour tout icon nommé. Les usages existants (`icon="meetings"`, `members`, `votes`, `archives`, `generic`) continuent de rendre leur SVG. Aucune régression visuelle sur les pages déjà consommatrices du composant. Patch chirurgical idéal.

### IN-005 — `ErrorDictionary.php` baseline byte-stable

`app/Services/ErrorDictionary.php` — `grep -c "=>"` retourne 206 (count exact pré-Phase 4). 81 messages enrichis par 04.4 sans ajout/suppression de codes. Le contrat API (`getMessage(string $code)`, `getCodes()`, `hasMessage()`) reste strictement intact — aucun caller ne casse. `php -l` exit 0.

### IN-006 — Modal IDs uniques (post-04.2 migration)

7 instances `<ag-modal id="...">` dans les 6 fichiers HTML migrés, **toutes distinctes** (vérifié via `grep | sort | uniq -c`). Pas de collision DOM possible quand plusieurs pages partagent le même contexte (HTMX include scenarios).
