## 5. Process observations (Nyquist + workflow)

### 5.1 VERIFICATION.md absent
Les verifiers per-phase ont produit leurs outputs inline (dans la conversation) plutôt que de les écrire sur disque. Pas un blocker (les SUMMARY.md couvrent l'info), mais c'est une déviation par rapport au workflow standard. À documenter dans le runbook v2.4 si on veut ce filet.

### 5.2 Stream timeouts récurrents
**5 stream idle timeouts** sur cette session (planner Phase 1 revision, planner Phase 2 first try, planner TECH-01 first try, executor 04.3, executor 04.6, code reviewer cumulative). Pattern : agent passe le budget tool calls en lectures avant de produire output. **Mitigation appliquée** : prompts plus serrés au retry + finalisation orchestrator inline post-commit. **Followup v2.4** : configurer agents avec budget timeout plus large ou patterns "write-as-you-go".

### 5.3 Schoger scan erreur de comptage
Le scan Explore initial a compté `grep -c shortcut-card` = 15 (incluant enfants BEM `__icon`/`__text`/etc.) au lieu des 3 cards réelles. Erreur propagée dans REQUIREMENTS DASHBOARD-05 + plan + questionnaire. **Détection** : Rule 4 deviation Plan 03.2 par l'executor lors de l'audit factuel. **Résolution** : zero-diff + correction en cascade (REQUIREMENTS + Schoger review + plan SUMMARY).

### 5.4 Plan-checker iter cycles efficaces
Pattern systématique sur les 4 phases :
- Iter 1 trouve toujours 2-3 BLOCKERs + 4-6 FLAGs
- Iter 2 fixes appliqués en mode chirurgical (orchestrator inline)
- Iter 2 verification PASSED → execute-phase

C'est **le bon ROI** : sans plan-checker, les BLOCKERs auraient été détectés en exécution (coût 10× plus haut).
