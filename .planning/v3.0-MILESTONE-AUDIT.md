## Unsatisfied Requirements

### Critical — Integration Gaps

**HUB-01: Hub loads real session state** (Phase 16) — PARTIAL
- Hub correctly loads session data via `wizard_status` API
- **Gap:** Hub action buttons do not pass `meeting_id` to operator page
- Operator opens with no meeting pre-selected; user must manually pick from dropdown
- Fix: `hub.js` — call `MeetingContext.set(sessionId)` or append `?meeting_id=` to destination URLs

**VOT-01: Operator opens motion, voters see it** (Phase 20) — PARTIAL
- Motion opens correctly via `motions_open.php` and `motionOpened` SSE fires
- **Gap:** When meeting is `frozen`, `motions_open.php` does not transition to `live` or broadcast `meetingStatusChanged`
- `operator_open_vote.php` (which handles this) is registered in routes.php but never called by the frontend
- Fix: `operator-motions.js` — when `isFrozen`, call `operator_open_vote.php` instead of `motions_open.php`

**VOT-04: State machine transitions work E2E** (Phase 20) — PARTIAL
- Same root cause as VOT-01: frozen→live transition skipped for combined "start session + open vote" path
- The explicit "change meeting status" button via `doTransition('live')` works correctly

- Integration checker confirms wiring is correct (motions_for_meeting.php, consolidation, Dompdf, archive transition)
