# Plan 04.1 — Visual Validation Procedure (3 PVs longs)

## Sandbox status

- PDF binary generation IS exercised in CI (`testGeneratePdfReturnsNonEmptyBinary`, `testBuildPdfBytesReturnsPdfMagicHeader`) → confirms dompdf 3.1 accepts the new `@page` syntax and produces a valid `%PDF-` binary.
- HTML output assertions IS exercised in CI (`testBuildPdfHtmlContainsAtPageWithHeaderAndFooter`, `testRenderHtmlContainsAtPageBlock` & 4 others) → confirms `@page`, `@top-center`, `@bottom-center`, `counter(page)`, `counter(pages)`, header text + page-break preservation are present in emitted HTML.
- **Visual inspection of rendered pages** (header repeated each page, footer pagination evolves correctly) → cannot be done in sandbox without a PDF viewer. **Procedure below to be executed on dev-machine.**

## Procedure

### Pre-flight

1. Pull `feat/v2.4-cockpit-polish` and `composer install`.
2. Boot the app (`make dev` or `php -S 0.0.0.0:8080 -t public`).
3. Have at least one tenant with 3 séances available, ideally with varied content sizes.

### Test cases

| # | Cas | Source de génération | Pages attendues |
|---|-----|----------------------|-----------------|
| 1 | PV séance courte (1–2 résolutions, ≤10 présents) | `MeetingReportsService::buildPdfBytes` via `/api/v1/meetings/{id}/report.pdf` | 1–2 |
| 2 | PV séance moyenne (5–10 résolutions, ~30 présents) | idem | 3–5 |
| 3 | **PV séance longue (≥20 résolutions, ≥50 présents)** | idem | **≥10** |
| 4 | Procuration | `ProcurationPdfService::generatePdf` via `/api/v1/proxies/{id}.pdf` | 1 |

### Inspection points

Pour chaque PDF généré :

1. **Page 1** :
   - Header (haut centre) : `[Titre séance] — JJ/MM/YYYY`
   - Footer (bas centre) : `Page 1 sur N`
   - Aucun chevauchement avec le contenu (margin-top: 3cm respecté)

2. **Page médiane** (≥page 3 pour cas #3) :
   - Header : titre + date identiques à page 1 (répétition correcte)
   - Footer : `Page 3 sur N` (incrémenté)
   - Pas de bloc résolution coupé en deux pages (page-break-inside: avoid actif)

3. **Page finale** :
   - Header : toujours présent
   - Footer : `Page N sur N` (compteur final cohérent)
   - Signature + footer "Document généré..." présents et non tronqués

### Verdict acceptance

- ✅ AC remplis si : sur les 4 PDFs, header répété + footer paginé évoluent correctement, sans régression visuelle.
- ❌ Bloquant si : header absent / footer non paginé / texte tronqué / overflow / encoding accents cassé.

### Dispositif fallback

Si dompdf 3.1 affiche du texte vide pour `@top-center` (mode legacy) :
1. Vérifier `$options->set('isHtml5ParserEnabled', true)` (déjà actif dans les deux services ✓)
2. Tester en local en pré-rendant : `dompdf --no-php loadHtml | render` (CLI bin/dompdf if installed)
3. Si toujours vide → activer fallback `Dompdf::getCanvas()->page_text(...)` API en post-render hook (D-CONTEXT discrétion). Documenter deviation dans SUMMARY.
