## Deviations from Plan

### 1. [Rule 1 — Bug] Plan suggested `\b` word boundary for BEM tokens — actually unsafe in grep ERE

**Found during** : Task 3 (EXPLORE-PATTERNS.md) regex validation in bash.

**Issue** : The plan section suggested `grep -cE '\.shortcut-cards\b'` as the correct pattern. Bash test showed this STILL matches `.shortcut-cards-grid` and `.shortcut-cards--variant` because POSIX `\b` treats `-` as a non-word character — it's a boundary between `s` and `-` and again between `-` and `g`, so the match succeeds at the start.

**Fix** : Replaced `\b` recommendation for hyphenated tokens with the character-class pattern `<token>($|[^a-zA-Z_-])`. `\b` retained as correct only for ASCII-pure tokens (JS identifiers like `meeting`).

**Files modified** : `.planning/codebase/EXPLORE-PATTERNS.md` (Rule d'or section + Anti-pattern #1 fix + Generic pattern table).

**Validation** : All 6 regex examples self-tested in bash — outputs match documented counts on real `public/assets/css/operator.css` (naive=7, correct=3).

**Commit** : `738764c`

### 2. [Rule 2 — Critical doc accuracy] op-tab modifier check returned 0

**Found during** : Task 3 final validation pass.

**Issue** : Initial draft of EXPLORE-PATTERNS validation case asserted `grep -cE '\.op-tab--' operator.css` would show "N variantes modificateurs (n'apparaissent PAS dans le compte précédent)". Real check returned 0 — there are no `--` BEM modifiers in operator.css; the variants are `:hover`, `.active`, `-panel`, `s` (`.op-tabs`).

**Fix** : Rewrote the validation block with the actual variant list and explained why `[^a-zA-Z_-]` correctly admits `:` (pseudo-class) and `.` (combined state) while rejecting `s` and `-`.

**Files modified** : `.planning/codebase/EXPLORE-PATTERNS.md` (Cas d'usage validation v2.4 section).

**Commit** : Folded into `738764c` (single commit for EXPLORE-PATTERNS).
