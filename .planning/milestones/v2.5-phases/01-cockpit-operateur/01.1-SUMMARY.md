## Decisions Made

- **Light DOM vs shadow DOM:** chose light DOM because (a) design-system tokens must cascade in, (b) Plan 01.3 needs to style `#viewExec` (a sibling DOM node) from the same companion stylesheet, and (c) the existing `ag-quorum-bar.js` precedent uses light DOM for the same reasons. `ag-modal.js` keeps shadow DOM where input isolation matters; that does not apply here.
- **Pulse on the vote zone, not the bar:** ROADMAP SC #2 reads "le pulse entoure la zone vote". Animating only the bar would create a misplaced affordance (alert near the indicator, not where the action happens). The component therefore exposes `quorum-state` as a contract; Plan 01.3 will mirror it onto `#viewExec`. The bar still gets a static danger border (via `:has()`) so the indicator stays consistent.
- **Token substitutions:** `design-system.css` exposes `--color-surface` (not `--surface-base`), `--color-bg-subtle` (not `--surface-sunken`), and has no `--radius-pill`. Substitutions documented in the file header so future tooling and reviewers see the audit trail.
- **Defensive fallback to known enum values:** unknown `quorum-state` / `sse-state` inputs collapse to `met` / `offline` so the rendered `data-state` attribute is always one of the three CSS-known values — prevents broken state tints on bad upstream data.
