## Decisions Made

- **IIFE vs ES module:** matched the existing convention in `public/assets/js/pages/` (operator-realtime.js, operator-exec.js, operator.js) so the file ships via a plain `<script src="…">` tag in Plan 01.3 — no build step required.
- **Anti-trap fix on `isContentEditable`:** the existing handler in `operator-exec.js` (lines 417-436) only excluded `input/textarea/select`. Rich-text fields and any contenteditable surface would have leaked shortcut presses. The new module fixes this per COCKPIT-06; Plan 01.3 will remove the legacy handler and lean exclusively on this module.
- **`?` available in any mode:** the help overlay is passive documentation. Gating it behind `_isExecMode()` would mean a user browsing the cockpit pre-meeting cannot discover the shortcuts. The mode check applies to action shortcuts only (L/F/→/N).
- **Single toggle fallback (`#opBtnToggleVote`):** the current operator UI has one button that toggles open ↔ closed; Plan 01.3 will (or may) split it. To avoid coupling Plan 01.2's correctness to Plan 01.3's button rename, the module tries `#opBtnLaunchVote` / `#opBtnCloseVote` first and falls back to `#opBtnToggleVote`. This makes the module's behaviour robust during the migration window.
- **Lazy overlay construction:** building the overlay DOM on first `?` press (not on script load) keeps the cost zero on the (probably common) case where the user never opens the help. It also keeps the document body free of an extra hidden node until needed.
- **Two `keydown` listeners not one:** the dispatcher listens at `document` for the action shortcuts; `_ensureOverlay` adds a second `document` listener scoped to `Escape` while the overlay is visible. They don't conflict (`Escape` isn't a dispatch case) and keeping them separate makes the overlay self-contained — if Plan 01.3 ever decided to lift the overlay into its own module, only `_ensureOverlay` would migrate.

## Existing P/F handler in operator-exec.js — confirmation for Plan 01.3

The legacy keydown block in `public/assets/js/pages/operator-exec.js` (lines 417-436, with shortcuts `P` for proclaim and `F` for vote toggle) is **confirmed for removal in Plan 01.3**:

- `P` → no longer applicable. Per COCKPIT-06 the new verb is `L` (Lancer le vote actif). `Proclamer` was specific to results display and is not part of the v2.3 cockpit shortcut contract.
- `F` → covered by the new module's `F` binding (with the same `#opBtnCloseVote` / `#opBtnToggleVote` fallback chain as before).
- The new module also adds the `isContentEditable` check the legacy handler missed — leaving both handlers in place would actually weaken anti-trap, since whichever fired first would skip the missing exclusion.

Plan 01.2 deliberately does NOT delete that block (out of scope per the plan objective and to keep the diff atomic). Plan 01.3 owns the deletion as part of its operator.htmx.html wiring task.

## Final shortcut → button-id mapping (as shipped)

| Shortcut         | Action                                | Primary target              | Fallback target          |
|------------------|---------------------------------------|-----------------------------|--------------------------|
| `L`              | Lancer le vote actif                  | `#opBtnLaunchVote`          | `#opBtnToggleVote`       |
| `F`              | Fermer le scrutin actif               | `#opBtnCloseVote`           | `#opBtnToggleVote`       |
| `→` or `N` / `n` | Résolution suivante                   | `O.fn.selectNextMotion()`   | next `.op-agenda-item:not(.voted)` after `.current` |
| `?`              | Afficher / masquer l'aide             | `#agShortcutsOverlay` (lazy) | n/a                     |
| `Échap`          | Fermer l'overlay s'il est visible     | `#agShortcutsOverlay`       | n/a                      |

All five paths skipped if `metaKey || ctrlKey || altKey` is held, or if `_isTypingContext()` is true. Action shortcuts (L/F/→/N) additionally require `_isExecMode()` (the `?` overlay does not).
