# Plan 02.3 — Logger context + admin error-stats page

## Goal

Enrichir tous les call-sites `Logger::error()` / `::critical()` / `::alert()` avec un contexte standardisé (`request_id`, `user_id`, `tenant_id`, `error_code`, `caller`), et exposer une page admin HTMX `/admin/error-stats` affichant les codes d'erreur les plus émis sur 7/30 jours pour piloter l'observabilité produit.

Source : ERR-V24-03 (REQUIREMENTS.md), CONTEXT.md décisions D-08..D-12.

## Acceptance Criteria

1. **`Logger::errorContext()` helper** ajouté à `app/Core/Logger.php` qui auto-remplit `request_id`, `user_id`, `tenant_id` à partir du contexte session/request, fusionne avec context array fourni par caller (`error_code`, `caller`).
2. **Audit `02.3-AUDIT.md` livré** : grep complet `Logger::error|critical|alert` dans `app/` + `public/api/`. Pour chaque call-site : file:line, méthode, error_code émis (si déductible), classification (migrer / déjà-OK / hors-scope).
3. **Migration call-sites identifiés** : chaque call-site `Logger::error(`/`::critical(`/`::alert(` migré vers `Logger::errorContext('msg', ['error_code' => 'X', 'caller' => __METHOD__])` (ou équivalent). Cible : 80 %+ des sites identifiés migrés (out-of-scope sites documentés avec rationale).
4. **Page admin `/admin/error-stats.htmx.html`** créée :
   - Route déclarée dans `app/routes.php` avec middleware `RoleMiddleware: 'admin'`
   - Controller HTML utilisant `HtmlView::render()` (pattern login/setup, NE PAS étendre `AbstractController`)
   - Template affiche tableau : Code | Émissions 7j | Émissions 30j | Taux next-step
   - Filtre périodes (7j/30j/90j) via HTMX
5. **Stats query** : agrégation depuis `audit_events` table (filtrer par `event_type` correspondant à api_fail ou logger erreurs, à confirmer dans audit). Si la table ne contient pas ces events → fallback : compter via parsing fichier log structuré (less ideal, à documenter).
6. **Taux next-step** : placeholder N/A documenté dans le tableau si pas de tracking instrumenté (cas attendu — voir CONTEXT D-12). Colonne présente avec valeur "N/A — tracking instrumentation v2.5+".
7. **Test gardien** : `tests/Unit/Core/LoggerTest.php` créé ou enrichi, 2 tests minimum :
   - `testErrorContextAutoFillsRequestUserTenant` : appel sans context manuel → assertion log final contient les 3 IDs
   - `testErrorContextMergesWithCallerContext` : caller fournit `error_code` + `caller` → assertion fusion correcte
8. **RBAC enforcement test** : 1 test E2E ou unit qui vérifie qu'un user non-admin reçoit 403 sur `/admin/error-stats`.
9. **Régression nulle** : tests existants (Logger tests si présents, AuthMiddleware, Security suite) restent verts.

### Task 2 — Implémenter `Logger::errorContext()` helper

Modifier `app/Core/Logger.php` :

```php
public static function errorContext(string $message, array $context = []): void
{
    $autoContext = [
        'request_id' => self::getRequestId(),
        'user_id' => function_exists('api_current_user_id') ? api_current_user_id() : null,
        'tenant_id' => function_exists('api_current_tenant_id') ? api_current_tenant_id() : null,
    ];
    $merged = array_merge($autoContext, $context);
    self::error($message, $merged);
}

// Idem pour criticalContext, alertContext si scope étendu
```

`php -l app/Core/Logger.php` doit passer.

### Task 3 — Migrer call-sites

Atomic commit per service ou logical group. Pour chaque site :

```php
// AVANT

    'error_code' => 'vote_already_cast',

]);
```

| Migration 30-50 sites = scope creep | M | M | Cible 80 %+ — out-of-scope sites documentés. Atomic commit per service, revert facile si problème |
| RBAC mal câblé → fuite admin route | L | H | Test RBAC explicit (Task 5) + smoke test manuel pré-merge |
| `errorContext` change le format log → casse parsers existants | M | M | Test: format JSON identique à `Logger::error()` baseline (juste plus de clés). Pas de changement format wrapper. |
| Page admin sans tokens design v2.2 | L | L | Réutiliser classes existantes admin pages (audit pattern existing dans `public/admin/`) |
