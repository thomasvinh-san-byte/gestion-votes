**operator-tabs.js — 4 changes:**

1. **Add sendReminder() function** after the sendInvitations() function (~line 2928). Follow the exact sendInvitations() pattern:
   - Guard: check `currentMeetingId` exists
   - Guard: check `membersCache.filter(m => m.email).length > 0`
   - Show confirmation modal via `O.confirmModal()` with title "Envoyer un rappel", body showing member count, confirmText "Envoyer le rappel", confirmClass "btn-warning"
   - Call `api('/api/v1/invitations_send_reminder.php', { meeting_id: currentMeetingId })`
   - On success: `setNotif('success', \`${scheduled} rappel${scheduled > 1 ? 's' : ''} envoye${scheduled > 1 ? 's' : ''}\`)`
   - On error: handle `smtp_not_configured` specifically, fallback to generic error message
   - Use `Shared.btnLoading(btn, true/false)` on `btnSendReminder` during request
   - After success, call `loadInvitationStats()` to refresh counts

2. **Add event listener** for btnSendReminder after the btnSendInvitations listener (~line 2929):
   ```javascript
   document.getElementById('btnSendReminder')?.addEventListener('click', sendReminder);
   ```

3. **Add invitation status badge update in loadInvitationStats()** (~line 2812). At the end of the function, after stats are updated, add badge logic:
   ```javascript
   const invBtn = document.getElementById('btnSendInvitations');
   if (invBtn) {
     let badge = invBtn.querySelector('.inv-status-badge');
     if (inv.sent > 0 || inv.total > 0) {
       if (!badge) {
         badge = document.createElement('span');
         badge.className = 'inv-status-badge badge badge-sm ml-1';
         invBtn.appendChild(badge);
       }
       badge.textContent = `${inv.sent}/${inv.total}`;
     } else if (badge) {
       badge.remove();
     }
   }
   ```
   Where `inv` is the stats object from the API response (find the variable name used in loadInvitationStats — likely the destructured response body).

4. **Add results email count to close-transition success toast.** Find the close-transition handler in operator-tabs.js (where `toStatus === 'closed'` triggers a success toast/notification). Update the toast message to include results email count. Two approaches depending on what the transition API returns:

   **(a) If transition response includes `results_emails` key** (check MeetingWorkflowController — if Plan 01 added it to api_ok payload): Use `data.results_emails` in the toast, e.g.:
   ```javascript
   const emailMsg = data.results_emails > 0
     ? ` — ${data.results_emails} email${data.results_emails > 1 ? 's' : ''} de resultats programme${data.results_emails > 1 ? 's' : ''}`
     : '';
   setNotif('success', `Seance cloturee${emailMsg}`);
   ```

   **(b) If transition response does NOT include results_emails:** First update `MeetingWorkflowController::transition()` to capture the scheduleResults() return value and include it in the api_ok() payload:
   ```php
   $resultsEmailCount = 0;
   if ($toStatus === 'closed') {
       try {
           global $config;
           $emailQueue = new EmailQueueService($config ?? []);
           $result = $emailQueue->scheduleResults($tenantId, $meetingId);
           $resultsEmailCount = $result['scheduled'] ?? 0;
       } catch (Throwable $e) {
           error_log('[Email] Results email scheduling failed: ' . $e->getMessage());
       }
   }
   ```
   Then include `'results_emails' => $resultsEmailCount` in the api_ok() data array. Then use approach (a) in the JS.

   NOTE: If MeetingWorkflowController.php is already modified per Plan 01 Task 2, you may need to adjust the existing code to capture the return value. This is a minor change to the try/catch block.
