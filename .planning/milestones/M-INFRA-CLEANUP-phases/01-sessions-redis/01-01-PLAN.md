# Plan 01-01 — Sessions PHP fichier → Redis (P0 dogfood-blocker)

## Objective

Migrer le storage des sessions PHP de `/tmp` (fichier, tmpfs read-only 128M, mort à chaque restart container) vers Redis DB 1 isolée. **Critère :** un `docker compose down && up` (sans `-v`) doit préserver la session de l'utilisateur connecté.

## Context

- Stack actuelle : `deploy/php.ini` ligne 22 : `session.save_path = "/tmp"`. Container app monte tmpfs 128M sur `/tmp` (docker-compose.yml). À chaque arrêt/démarrage : `/tmp` réinitialisé → toutes les sessions perdues.
- Phpredis déjà chargé (vérifié AUDIT-STACK-05). Service `redis` déjà up dans docker-compose avec password `${REDIS_PASSWORD}`.
- DB 0 utilisée par RedisProvider (cache app + rate-limit + SSE queue). On prend **DB 1** pour les sessions = isolation, pas d'interférence avec FLUSHDB côté app.
- Pattern injection runtime ini : `deploy/entrypoint.sh` génère déjà `/tmp/php-runtime/zz-runtime.ini` (cookie_secure, opcache). On suit le même pattern pour le password Redis (qui vient de l'env, pas de la build).
- Sessions accédées via `AgVote\Core\Security\SessionHelper::start()` — aucun changement applicatif requis, uniquement infra PHP.

## Boundary

Modifications attendues :

- `deploy/php.ini` — `session.save_handler = redis` + `session.save_path = "tcp://redis:6379?database=1&prefix=agvote-sess:"` (sans auth — auth injecté runtime).
- `deploy/entrypoint.sh` — append `session.save_path` (avec `?auth=PASSWORD`) au runtime ini quand `REDIS_PASSWORD` est set.
- `.planning/intel/SESSIONS-MIGRATION.md` — doc opérationnelle (TTL, flush, hard-cutover).
- `tests/Unit/SessionRedisConfigTest.php` (NEW) — valide que `ini_get('session.save_handler') === 'redis'` après chargement.
- `tests/e2e/specs/session-persistence.spec.js` (NEW) — login → restart container app (Redis stays up) → vérifie session active.
- `composer.lock` : intouché (pas de nouvelle dépendance, phpredis = ext native).

## Tasks

### Task 01 — `CLEANUP-SESSIONS-01` : Configuration PHP

1. **Ajouter au `deploy/php.ini`** sous la section `; Session` :
   ```
   session.save_handler = redis
   session.save_path = "tcp://redis:6379?database=1&prefix=agvote-sess:"
   ```
   Note : le `auth=` est ajouté dynamiquement par entrypoint.sh car le password est runtime.

2. **Modifier `deploy/entrypoint.sh`** : dans le bloc qui écrit `zz-runtime.ini` (lignes ~146-161), append après le branch dev/prod :
   ```bash
   if [ -n "${REDIS_PASSWORD}" ]; then
     {
       echo "session.save_path = \"tcp://${REDIS_HOST:-redis}:${REDIS_PORT:-6379}?auth=${REDIS_PASSWORD}&database=1&prefix=agvote-sess:\""
     } >> /tmp/php-runtime/zz-runtime.ini
     echo "Sessions: Redis ${REDIS_HOST:-redis}:${REDIS_PORT:-6379} DB1"
   else
     echo "Sessions: Redis (sans auth) — production exige REDIS_PASSWORD"
   fi
   ```
   Cela override le `save_path` de php.ini avec la version authentifiée.

3. **Vérifier extension phpredis chargée** : pas d'action — déjà garantie par fail-fast Dockerfile.

**Commit :** `feat(01-01): configure PHP sessions to use Redis DB1`

### Task 02 — `CLEANUP-SESSIONS-02` : Documentation migration

Créer `.planning/intel/SESSIONS-MIGRATION.md` avec sections :

- **TL;DR** : Sessions sur Redis DB 1, hard-cutover (tous logged out à upgrade), TTL = `session.gc_maxlifetime` PHP (1440s défaut).
- **Configuration** : ENV requis (`REDIS_PASSWORD`), DB partitioning (DB0 cache app, DB1 sessions), prefix `agvote-sess:`.
- **TTL & garbage collection** : phpredis utilise `EXPIRE` natif Redis = pas de cron PHP gc nécessaire. `gc_maxlifetime` traduit en TTL Redis sur chaque write.
- **Migration opérationnelle** : impact deploy = utilisateurs déconnectés une fois (pas d'erreur visible, juste re-login). Pas de double-write nécessaire (pas de prod live).
- **Rollback** : commenter les 2 lignes `session.save_handler` / `session.save_path` dans php.ini = retour fichier `/tmp`.
- **Monitoring** : `redis-cli -n 1 DBSIZE` pour compter les sessions actives. `redis-cli -n 1 KEYS 'agvote-sess:*' | wc -l` pour audit ponctuel.
- **Failure mode** : Redis down → PHP `session_start()` échoue → user voit erreur 500. Acceptable car sans Redis le rate-limit + SSE sont déjà cassés (Redis = dépendance dure post-pivot).

**Commit :** `docs(01-01): SESSIONS-MIGRATION operational guide`

## Risks

- **R1 — phpredis session handler unauthenticated** : si `auth=` mal échappé, échec silencieux + fallback file impossible (handler = redis hard). **Mitigation** : test PHPUnit + verif `php -i` après deploy.
- **R2 — auth password contient char spécial URL** : `?` `&` `=` dans REDIS_PASSWORD casse l'URI. **Mitigation** : entrypoint encode via `printf '%s' | jq -sRr @uri` si nécessaire (seulement si REDIS_PASSWORD contient un de ces chars).
- **R3 — tmpfs `/tmp` toujours read-only** : pas un risque, on ne stocke plus rien dans `/tmp` côté sessions.

## Notes

- Migration **hard-cutover acceptée** (DECISION.md ouvre cette option, pas de prod live).
- Pas de double-write file+redis pendant transition.
- `session.save_path` overridé par `zz-runtime.ini` car PHP_INI_SCAN_DIR scanne `conf.d` PUIS `php-runtime` → la dernière déclaration gagne.
