## Animation Timing & Easing

All durations and easings reuse existing `design-system.css` tokens. No new tokens introduced.

| Animation                  | Duration                         | Easing                                              | Property animated         | Rationale |
|----------------------------|----------------------------------|-----------------------------------------------------|---------------------------|-----------|
| Counter numeric tween      | **400ms** (`--duration-elaborate`) | `--ease-out` (`cubic-bezier(0, 0, 0.2, 1)`)       | `textContent` (RAF)       | Long enough that humans *see* values changing (not a snap), short enough to finish before the next SSE arrival in a busy session (typical: ‚â•1 vote/sec at peak). Ease-out lands gently on the final value ‚Äî no overshoot (overshoot on counters reads as "wrong number"). |
| Counter cell bump pulse    | **300ms** (`--duration-deliberate`) | `--ease-spring` (`cubic-bezier(0.34, 1.56, 0.64, 1)`) | `transform: scale()`   | Single pulse 1 ‚Üí 1.06 ‚Üí 1. Spring easing gives a slight overshoot that reads as "kinetic" without being cartoonish. Triggered on the cell that increased, not on all three. Confirms *which* counter received the vote. |
| Bar fill width             | **400ms** (`--duration-elaborate`) | `--ease-out`                                       | `width` (via `--bar-pct`) | Existing CSS already has `transition: width 0.4s var(--ease-default)` ‚Äî Phase 3 standardizes this to `var(--duration-elaborate) var(--ease-out)` for consistency with counter tween. Duration matched to counter so bar and number arrive together. |
| Bar percentage tween       | **400ms** (`--duration-elaborate`) | `--ease-out`                                       | `textContent` (RAF)       | Tied to the bar width ‚Äî finishing in lockstep avoids the bar landing while the % label still reads stale. |
| Participation bar          | **400ms** (`--duration-elaborate`) | `--ease-out`                                       | `width` (via inline style) | Same family ‚Äî coherent timing across all motion in the card. |

**Why 400ms and not 200ms or 600ms:**
- < 250ms reads as a snap; counter values flash by unread.
- > 500ms makes a busy AG (multiple votes/sec) feel laggy ‚Äî animations queue up.
- 400ms is the project's `--duration-elaborate` token already used elsewhere (analytics width transitions, vote bar legacy code).

**Why ease-out and not spring/overshoot for counters/bars:**
- A counter that lands on 47 should not overshoot to 49 then come back ‚Äî even visually, this reads as data corruption.
- Bars overshooting their final width look toy-ish in a governance tool.
- Spring/overshoot is reserved for the cell *bump pulse* (transform-only), which never represents a numeric value.

## Counter Animation Behavior

### Tween mechanics

When `#execVoteFor` (or any counter) receives a new target value via SSE-driven update:

1. **Read** current rendered integer from `textContent` (parse, fallback 0).
2. **Compute** delta = `target - current`.
3. **If delta === 0**: no animation, no work.
4. **If `prefers-reduced-motion: reduce`**: write `target` directly. Skip RAF entirely.
5. **Else**: run a `requestAnimationFrame` tween over 400ms with ease-out easing (`1 - Math.pow(1 - t, 3)`), writing `Math.round(start + delta * eased)` each frame.
6. **On completion**: write the final integer exactly once (defensive ‚Äî guarantees `textContent === target`).

**Key constraints:**

- **Integer ticking, not chunked jumps.** A 0 ‚Üí 47 transition over 400ms at 60fps = 24 frames; ~2 integer increments per frame. The user sees values like `‚Ä¶3, 5, 8, 11, 15, 19, 23, 27, 31, 35, 39, 42, 45, 47` ‚Äî feels live. NEVER chunk to "0 ‚Üí 24 ‚Üí 47" ‚Äî that destroys the "live tally" feeling.
- **Cancel pending tween** when a new value arrives mid-tween. Store the active RAF id on the element (`el._tweenRaf`) and cancel it before starting a new one. The new tween starts from whatever the displayed value was at cancellation.
- **No leading zero / padding.** Values render as `0`, `7`, `47` (not `00`, `07`, `47`). `font-variant-numeric: tabular-nums` (already on `.op-vote-count`) keeps the column width stable as digit count grows.
- **Initial render after hidden/show.** When the vote card transitions from `hidden` to visible, counters should NOT tween from 0 ‚Äî they should appear at the current authoritative value instantly. Tweening only fires on subsequent updates.
- **Decrement support.** If a vote is reverted (rare, but possible via existing audit operations), the tween runs in reverse identically. ease-out works in both directions.

### Cell bump pulse

When a counter value *increases*, the parent `.op-vote-counter` cell (e.g. `.op-vote-for`) gets a one-shot `op-vote-counter--bump` class for 300ms.

- CSS rule: `transform: scale(1.06)` at 50% keyframe, returning to `scale(1)` at 100%.
- Easing: `--ease-spring` (subtle overshoot at the bottom of the return).
- Class is removed on `animationend` event.
- **Decrement does NOT trigger the bump** ‚Äî the bump means "a vote landed here". Reverts are silent.
- **Equal target** (no change) does NOT trigger the bump ‚Äî nothing landed.
- **Multiple counters incrementing in the same SSE batch** (e.g. tally correction): each gets its own bump. Bumps don't compose into a louder effect ‚Äî visually they're independent.

### Color flash on increment? Bounce?

**No color flash.** The counter is already a saturated color (success green, danger red, warning amber) at all times ‚Äî a flash would be redundant and noisy. The bump pulse alone is the increment cue.

**No bounce.** Spring easing on the *transform pulse* gives 6% overshoot peak which reads as kinetic. Spring on the *number itself* (overshoot value) is forbidden ‚Äî see "ease-out and not spring" above.

**No tick / sound.** Out of scope per project memory (no audio cues in operator console).

## Progress Bar Animation Behavior

The bar fills already have `transition: width 0.4s var(--ease-default)` since v1.x. Phase 3 standardizes this and ensures the participation bar shares the same family.

### Property choice: `width` (not `transform: scaleX`)

Trade-off considered:
- **`transform: scaleX`** is GPU-composited (avoids layout thrashing). However scaleX from a left origin distorts the `border-radius` of `.op-bar-fill` (the fill has `border-radius: 4px`) ‚Äî corners would smear during transition.
- **`width`** triggers paint, but paint cost on a 7px-tall √ó ~200px-wide bar with 3 instances is trivial (< 1ms per frame on commodity hardware). Border-radius stays crisp.

**Decision: `width`.** Already in production. Visual quality wins over micro-optimization for a 3-bar element that updates ‚â§ 1√ó per second in normal load.

If profiling at 1080p ever shows layout thrash here, the swap to `transform: scaleX` + a wrapper that masks border-radius is a future optimization ‚Äî not Phase 3 scope.

### Easing choice: cubic-bezier (ease-out), not linear

**`linear`** would feel mechanical ‚Äî bars would slide at constant speed, then stop dead. Reads as a lab instrument, not a live tally.

**`--ease-out` (`cubic-bezier(0, 0, 0.2, 1)`)** decelerates as the bar approaches its target. Reads as "settling in" ‚Äî appropriate for a value that just changed and is now stable.

Cross-checked with existing `analytics.css:655` (`transition: width var(--duration-normal)`) ‚Äî same family, just longer duration here because vote results are the focal point and earn a more prominent transition.

### Bar fill direction

Bars fill left-to-right (`transform-origin` not used; default `width: var(--bar-pct)` from `0%` baseline). When percentage *decreases* (rare ‚Äî e.g. participation drops if eligible voter count grows mid-vote), bars retreat using the same transition. No special "retreat" animation.

### % label sync

The `.op-bar-pct` text label tweens its integer value over the same 400ms ease-out window as the bar width. Implementation: same RAF tween primitive as counters, just with `'%'` suffix appended to `textContent`.

If RAF and CSS transitions drift by a frame or two due to JS event loop pressure, that's acceptable ‚Äî the eye reads "bar and label both moving toward the same end state" not "bar leads label by 16ms".

## Reduced-Motion Semantics (ANIM-03)

When `(prefers-reduced-motion: reduce)` is active:

| Animation                  | Reduced-motion behavior                                          |
|----------------------------|------------------------------------------------------------------|
| Counter numeric tween      | **Hard cut.** Value written directly to `textContent`. No RAF.   |
| Counter cell bump pulse    | **Suppressed.** No `op-vote-counter--bump` class added. Cell stays at scale 1. |
| Bar fill width             | **Hard cut.** CSS rule overrides `transition: none`.             |
| Bar percentage tween       | **Hard cut.** Value written directly.                            |
| Participation bar / %      | **Hard cut.** Same.                                              |

**Implementation pattern (CSS):**

```css
@media (prefers-reduced-motion: reduce) {
  .op-bar-fill,
  .progress-bar {
    transition: none !important;
  }
  .op-vote-counter--bump {
    animation: none !important;
  }
}
```

**Implementation pattern (JS):**

```js
const reducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)');
function tweenCounter(el, target) {
  if (reducedMotion.matches) {
    el.textContent = String(target);
    return;
  }
  // ... RAF tween
}
// Listen for runtime preference change:
reducedMotion.addEventListener('change', () => { /* future writes use new mode */ });
```

**Why hard cut and not "keep color flash but no movement":**
- There is no color flash in the design (see "No color flash" above).
- The bump pulse IS the increment cue, and it IS movement ‚Äî suppressing it leaves the increment unmarked, which is the correct WCAG outcome (silent update, screen reader handles it via the existing `aria-live="polite"` on `.op-vote-counters`).
- Keeping a residual cue (e.g. brief background tint) violates the spirit of `reduce`: users who set this prefer no animation, full stop. The semantic announcement via aria-live is sufficient.

**Verification target (for ui-checker / executor):**
- Toggle OS reduce-motion preference ‚Üí fire SSE event ‚Üí assert no transform applied, no transition delay observable, value writes synchronously in one frame.

## Performance & Batching

### SSE event volume

Worst case observed: cluster vote of ~120 voters voting within a 5-second window during a roll call. Peak rate: ~5 votes/sec across 1 motion. Each `vote.cast` event triggers a counter recompute.

### Batching strategy

**Rule: 1 RAF tween per counter per SSE event.** Do NOT batch multiple SSE events into one tween ‚Äî each vote should produce a visible bump, even at 5 votes/sec.

At 5 votes/sec with 400ms tweens, each tween will be cancelled mid-way by the next one. This is **correct behavior** ‚Äî the bump pulse fires for each, so the operator sees a steady drumbeat of "vote landed" cues, while the underlying number tweens smoothly toward the latest value.

**Coalescing rule for the counter tween:** if a new SSE event arrives while a tween is in flight, cancel the in-flight RAF (`cancelAnimationFrame`) and start a new tween from the *currently displayed* integer toward the new target. This avoids:
- Stale tween "racing" to an outdated target.
- Multiple RAFs on the same element causing flicker.

**Coalescing rule for the bump pulse:** if a new bump fires while the previous bump is still in its 300ms window, remove and re-add the class with a 1-frame gap (`requestAnimationFrame` to clear, then add). This ensures the second bump is visible ‚Äî without the gap, the browser collapses identical class adds and the second pulse is silently absorbed.

### Layout thrashing avoidance

- **Read once, write once per frame.** The RAF callback reads `el.textContent` only at start (cached in tween state). Subsequent frames write only.
- **No `getBoundingClientRect`** in the animation hot path. None needed.
- **Bar `width` updates via CSS variable** (`--bar-pct`). Single style write per bar. Browser handles transition off the main thread for the actual width change.
- **`will-change` not used** ‚Äî modern browsers handle `transition: width` and short `transform` animations without it. Adding `will-change` permanently would consume layer memory unnecessarily on a 3-instance element.

### Frame budget

At 60fps, frame budget is 16.7ms. RAF tween cost per element: ~0.1ms (one parseInt, one math op, one `textContent` write). Three counters + three bar % labels = 6 RAF callbacks per frame = ~0.6ms total. Well under budget.

**Aria-live considerations:** the existing `aria-live="polite" aria-atomic="true"` on `.op-vote-counters` (line 1340 of `operator.htmx.html`) already announces the full counters block on update. Phase 3 must NOT add additional `aria-live` regions ‚Äî that would cause double announcements. The animation tween writes to `textContent`; `aria-live` polite reads the final composite at its own cadence (browser debounces).

If counter tween writes 24 intermediate values per second to `textContent`, screen readers may read intermediate values. To avoid this:
- Counter tween writes intermediate values to `textContent`.
- The RAF tween, on completion, leaves the final value in `textContent` (so screen reader reads stable end-state when its polite-region debounce settles).
- **Risk acceptance:** if a screen reader user has the operator console open AND a vote is in progress, they may hear "forty-two‚Ä¶ forty-five‚Ä¶ forty-seven" rather than just "forty-seven". This is acceptable: (a) operators using screen readers are an edge case in the live-session UI, (b) hearing the running tally is arguably *useful*, (c) when reduced motion is on (which screen reader users often have), animations are off ‚Üí only final values are announced.

If post-implementation testing shows screen-reader noise is unacceptable, the fix is to write intermediate values to a `data-tween-value` attribute and only update `textContent` on tween completion. This is a Phase 4 (Validation Gate) follow-up if it surfaces, not a Phase 3 spec change.

## Interaction Contract

### SSE event ‚Üí animation pipeline

```
SSE 'vote.cast' event arrives
  ‚Üí existing handler in operator-realtime.js fires
  ‚Üí existing handler calls refreshExecVote() in operator-exec.js
  ‚Üí refreshExecVote() reads ballot stats, computes new counts and percentages
  ‚Üí NEW: refreshExecVote() calls tweenCounter() for each of the 3 counters
  ‚Üí NEW: refreshExecVote() detects which counter increased, adds .op-vote-counter--bump to that cell
  ‚Üí NEW: refreshExecVote() calls tweenCounter() for each of the 3 bar % labels
  ‚Üí existing: refreshExecVote() writes --bar-pct CSS variable on each bar fill
  ‚Üí CSS transition handles the visual width slide (existing behavior, now using updated tokens)
```

**Existing handler is untouched in flow** ‚Äî only the assignment statements that set `textContent` and `--bar-pct` are routed through `tweenCounter()`.

### State transitions

| Vote card state                       | Animation behavior                                              |
|---------------------------------------|-----------------------------------------------------------------|
| Card hidden ‚Üí shown (vote opens)      | Counters render at 0 instantly. No tween from previous values. Bars at 0%. |
| Vote in progress, vote arrives        | Tween + bump pulse (full Phase 3 contract).                    |
| Vote closed (no more votes accepted)  | Final tally already displayed. No animation triggered.         |
| Card shown ‚Üí hidden (vote closes / motion changes) | No exit animation for this phase. Card disappears via existing display logic. |
| Vote reset / corrected (admin action) | Tween runs in reverse if values decrease. No bump pulse on decrement. |

### Race conditions

| Scenario                                                | Behavior                                                                  |
|---------------------------------------------------------|---------------------------------------------------------------------------|
| Two SSE `vote.cast` events arrive in same frame         | Each calls `tweenCounter()` ‚Äî second cancels first via RAF id. One bump per event. |
| SSE batched as single `meeting.snapshot` (catch-up)     | Snapshot delivers final counts in one update ‚Äî single tween from displayed ‚Üí snapshot target. **No multiple bumps for catch-up snapshots.** Detection: caller passes `{ silent: true }` option to suppress bump for snapshot/catch-up updates. |
| Reduced-motion preference toggled mid-session           | `matchMedia` listener flips; subsequent tweens use new mode. In-flight tweens complete normally. |
| User exits focus mode mid-tween                         | Vote card DOM is unchanged ‚Äî tween continues. Layout shift on `body.focus-mode` removal does not cancel RAF. |
