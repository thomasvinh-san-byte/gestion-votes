## Summary

Cette phase est une refonte CSS/HTML pure : pas de nouveau backend, pas de nouvelle API, pas de nouveaux composants JS. Le travail consiste à aligner la **composition visuelle** de 28 pages `.htmx.html` avec le wireframe v3.19.2 (`ag_vote_wireframe.html`) et à corriger le FOUC.

La source de vérité est le wireframe React dans `ag_vote_wireframe.html`. Les tokens CSS (`design-system.css`) sont déjà alignés (Phase 4), mais la **mise en page, la hiérarchie, et la densité** sont en retard. Les quatre zones prioritaires sont : (1) page votant tablette, (2) projection salle, (3) dashboard, (4) console opérateur.

Le FOUC vient de la chaîne `@import url("/assets/css/design-system.css")` dans `app.css`. La correction consiste à inliner un bloc `<style>` minimal (tokens critiques : `--color-bg`, `--color-text`, `--color-surface`) dans le `<head>` de chaque page HTML avant tout chargement externe. `theme-init.js` est déjà inline et synchrone — il fait le bon travail pour le thème, mais le flash vient des tokens `body { background: var(--color-bg) }` qui ne sont pas encore résolus au moment du premier paint.

**Recommandation principale :** Travailler fichier par fichier. Pour chaque page : (1) modifier le HTML pour la composition/hiérarchie, (2) modifier le CSS de la page pour les styles spécifiques, (3) ne jamais modifier `design-system.css` (tokens déjà alignés). La projection (public.css/public.htmx.html) reçoit le gradient background via CSS, pas via JS.

## Architecture Patterns

### Pattern 1 : Inline CSS critique dans `<head>` pour éliminer le FOUC

**Problème actuel :** `app.css` fait `@import url("/assets/css/design-system.css")`. Le navigateur parse app.css, trouve l'import, fait une requête réseau, et pendant ce temps le `body { background: var(--color-bg) }` n'est pas résolu — flash blanc/non-stylé visible pendant ~100-300ms.

**Solution :** Ajouter un bloc `<style>` dans le `<head>` de chaque page HTML **avant** les `<link rel="stylesheet">` :

```html
<!-- FOUC Prevention: inline critical tokens -->
<style>
  :root {
    --color-bg: #EDECE6;
    --color-surface: #FAFAF7;
    --color-text: #151510;
  }
  [data-theme="dark"] {
    --color-bg: #0B0D10;
    --color-surface: #141820;
    --color-text: #ECF0FA;
  }
  body { background: var(--color-bg); color: var(--color-text); }
</style>
```

Ce bloc est synchrone — pas de requête réseau. `theme-init.js` (déjà inline et synchrone) a déjà posé `data-theme` sur `<html>` avant ce style, donc le sélecteur `[data-theme="dark"]` s'applique immédiatement.

**Pages concernées :** Toutes les pages qui chargent `app.css`. La page `public.htmx.html` charge `design-system.css` directement — même correction nécessaire.

### Pattern 2 : Grille 2×2 votant (CSS grid)

**Wireframe :** Toujours 2×2 (Pour/Contre haut, Abstention/NSP bas). Les boutons sont plus hauts et ont un `background` coloré semi-transparent même sans sélection.

```css
/* vote.css — remplacement */
.vote-buttons {

  grid-template-columns: 1fr 1fr;  /* TOUJOURS 2x2 */
  grid-template-rows: 1fr 1fr;
  gap: 10px;
  /* supprimer le max-width: min(1100px, 100%) */
}
```

Les 4 choix dans le wireframe sont : Pour, Contre, Abstention, NSP (et non "Blanc"). Vérifier avec le HTML existant (`btnBlanc` → à renommer ou supprimer selon le métier).

### Pattern 6 : Dashboard — lanceur d'actions

**Actuel :** Dashboard dense avec "Action urgente", KPI row, "Prochaines séances" + "Tâches en attente" (2 colonnes), "Accès rapides".

**Wireframe PageDashboard :** Même structure, mais la hiérarchie est plus aérée. Les 3 raccourcis en bas sont les CTAs primaires. Les KPIs sont des liens, pas de l'information passive.

La structure actuelle HTML (`dashboard.htmx.html`) est déjà très proche du wireframe — elle a les 3 raccourcis, les 4 KPIs, les 2 panels. Le travail est plutôt de réduire la densité visuelle (espacement, réduire les éléments visuels parasites) que de restructurer.

### Pattern 7 : Hiérarchie typographique Fraunces vs Bricolage

**Règle wireframe :**
- `font-family: var(--font-display)` (Fraunces) → titres de pages, titres de résolutions en projection, logo
- `font-family: var(--font-sans)` (Bricolage Grotesque) → corps, labels, boutons, métadonnées

**Actuel :** `.page-title` dans `design-system.css` utilise `font-family: var(--font-display)` ✓. Mais les titres de résolutions dans la projection et le header de projection n'utilisent pas Fraunces.

**Application dans public.css :**
```css
.motion-title {  /* dans projection */
  font-family: var(--font-display);
}
```

### Anti-Patterns à éviter

- **Ne pas modifier `design-system.css`** : les tokens sont verrouillés depuis Phase 4
- **Ne pas ajouter de nouvelles dépendances JS** : stack vanilla
- **Ne pas casser les IDs et classes JS** : les JS de pages (`vote.js`, `public.js`, etc.) référencent des IDs spécifiques — les renommer casserait la fonctionnalité
- **Ne pas utiliser `!important`** : surcharger proprement par spécificité ou ordre dans le fichier
- **Ne pas inliner de styles dans le HTML** si une classe CSS suffit

## Don't Hand-Roll

| Problème | Ne pas construire | Utiliser à la place | Pourquoi |
|----------|-------------------|---------------------|----------|
| Thème clair/sombre projection | Un toggle JS custom | `document.documentElement.setAttribute('data-theme', ...)` + tokens CSS | Déjà en place dans theme-init.js + design-system.css |
| Responsive clamp | Calculs JS de taille | `clamp(min, preferred, max)` CSS natif | Zéro JS, adaptatif de 320px à 4K |
| FOUC prevention | Polyfill ou détection complexe | Inline `<style>` dans `<head>` | Standard industry — < 200 bytes |
| État de vote (couleur bouton confirm) | Composant overlay | Bouton inline + injection CSS via JS | Le wireframe montre explicitement ce pattern |
| Timer countdown | Une lib externe | `setInterval` + format JS (déjà en place dans vote.js) | Pas de nouvelle dépendance nécessaire |

## Common Pitfalls

### Pitfall 1 : Casser les IDs référencés par JS

**Ce qui va mal :** Renommer `id="btnFor"`, `id="confirmationOverlay"`, `id="motionTitle"`, etc. dans le HTML.

**Pourquoi :** `vote.js` et `vote-ui.js` font des `document.getElementById()` sur ces IDs. Les modifier sans mettre à jour le JS casse silencieusement la fonctionnalité.

**Comment éviter :** Garder tous les IDs existants. Ajouter des classes CSS pour le style, modifier la structure HTML (wrapping, affichage/masquage) sans toucher aux IDs fonctionnels.

### Pitfall 2 : Le fond gradient sur projection brise le thème switcher

**Ce qui va mal :** Hardcoder `background: linear-gradient(170deg, #0B0F1A, #111827)` sur `body.projection-body` sans gérer le mode clair.

**Pourquoi :** La page `public.htmx.html` force dark au chargement mais a un bouton thème toggle. Le JS appelle `document.documentElement.setAttribute('data-theme', 'light')` et les tokens changent, mais si le gradient est hardcodé il reste sombre.

**Comment éviter :**
```css
body.projection-body {
  background: var(--projection-bg, linear-gradient(170deg, #0B0F1A, #111827));
}
[data-theme="light"] body.projection-body,
body.projection-body.light-mode {
  background: linear-gradient(170deg, #F8FAFC, #EEF2FF);
}
```

Ou mieux : utiliser des tokens CSS custom définis dans le `[data-theme]` block.

### Pitfall 3 : La confirmation inline désynchronise vote-ui.js

**Ce qui va mal :** Supprimer le HTML du bottom sheet overlay (`#confirmationOverlay`) que `vote-ui.js` appelle directement.

**Comment éviter :** Deux stratégies possibles :
- A) Garder l'overlay HTML mais le masquer via CSS (`display: none !important`) et ajouter le nouveau bouton inline géré par JS
- B) Modifier `vote-ui.js` pour utiliser le nouveau pattern inline (préféré — plus propre mais touche au JS)

Le plan doit choisir explicitement entre A et B.

### Pitfall 4 : FOUC non corrigé sur la page public (design-system.css direct)

**Ce qui va mal :** Ne corriger le FOUC que sur les pages qui chargent `app.css`, oubliant `public.htmx.html` qui charge `design-system.css` directement.

**Comment éviter :** Le inline CSS critique doit être ajouté à TOUTES les pages `.htmx.html`, pas seulement celles qui passent par `app.css`.

### Pitfall 5 : `clamp()` avec vw sur écrans moniteur standard (~1920px)

**Ce qui va mal :** `clamp(32px, 3.6vw, 56px)` donne 56px à 1555px+ de largeur d'écran. Sur un écran tablette de 768px, ça donne 27px — sous le minimum. Les navigateurs respectent le `min`, donc en dessous de 889px (~32/0.036) la valeur est fixée à 32px.

**Ce que ça veut dire :** Vérifier le rendu à 768px, 1024px, 1920px et "4m projector" (~3840px). Les valeurs du wireframe sont testées pour la projection — elles peuvent paraître gigantesques sur une petite tablette.

**Comment éviter :** Tester à plusieurs breakpoints. Ne pas ajuster mécaniquement tous les `clamp()` sans valider visuellement.

   - Ce qu'on sait : le wireframe PageVotant utilise `{l:"NSP", c:"#9CA3AF"}` comme 4e choix. Le HTML actuel utilise `btnBlanc` avec "Blanc".
   - Ce qui est flou : est-ce que "Blanc" et "NSP" sont fonctionnellement identiques, ou est-ce que renommer "Blanc" en "NSP" casserait le backend ?
   - Recommandation : Garder le libellé "Blanc" pour le backend, mais le planner doit vérifier `vote.js` et l'API pour confirmer. Le CSS de style peut être mis à jour sans toucher au libellé.

2. **Confirmation inline : modifier vote-ui.js ou garder l'overlay caché ?**
   - Ce qu'on sait : deux stratégies valides (A: CSS-only en masquant l'overlay, B: modifier vote-ui.js)
   - Ce qui est flou : l'overlay a de la logique d'accessibilité (aria-modal, aria-labelledby) et du CSRF logic dans `vote-ui.js`. Modifier le JS est plus propre mais plus risqué.
   - Recommandation : Le plan doit spécifier clairement laquelle choisir. La stratégie A (CSS-only) est plus sûre pour une phase UI pure.

3. **Pages settings/help/audit — étendue exacte**
   - Claude's Discretion dit "évaluer page par page". Ces pages peuvent nécessiter peu ou beaucoup de travail.
   - Recommandation : Le planner doit prévoir une task "audit visuel + corrections mineures" pour ces pages, avec une time-box explicite.
