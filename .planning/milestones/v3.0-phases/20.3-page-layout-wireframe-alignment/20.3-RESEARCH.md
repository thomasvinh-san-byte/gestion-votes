# Phase 20.3: Page Layout Wireframe Alignment - Research

**Researched:** 2026-03-17
**Domain:** HTML layout restructuring + CSS density alignment (all pages vs wireframe v3.19.2)
**Confidence:** HIGH

---

## Summary

Phase 20.3 is a systematic layout pass across all 28 pages of AG-VOTE to align each page's structure, density, and spacing with the wireframe v3.19.2 (`ag_vote_wireframe.html`). The goal is that cockpit pages (dashboard, operator, hub) fit entirely in 1920x1080 without unnecessary scroll. Pages with long data (tables) are handled case-by-case.

The wireframe is a self-contained React prototype with inline styles. Every layout decision (column widths, gap values, padding values, grid counts) is directly readable from its JSX. Four pages require significant HTML restructuring beyond CSS-only changes: operator (tabs → split panel), settings (tab panel → sidebar nav + content), hub (identity banner + 2-column layout already partially in place), and wizard (footer nav fix). The remaining pages primarily need spacing compaction in CSS.

Phases 20.1 and 20.2 already aligned tokens, components, and CSS definitions. Phase 20.3 exclusively touches HTML structure and page-level layout CSS — not JavaScript logic, not design tokens, not component internals.

**Primary recommendation:** Work page-by-page in order of structural complexity: operator first (largest HTML change), then settings, wizard, hub confirmation, then all tables, then dashboard, then remaining pages. Each page gets a targeted CSS override in its dedicated CSS file plus HTML changes where needed.

## Architecture Patterns

### Operator — Split Panel (biggest structural change)

The current operator page uses a full-width tabs UI inside `.app-main`. The wireframe replaces the execution tabs (vote, results, presences) with a permanent split panel when in execution mode.

**Current structure (simplified):**
```
app-shell
  app-sidebar
  meeting-bar
  lifecycle-bar
  tabs-nav
  app-main
    viewSetup (section)
    viewExec (section)
      operator-exec tabs (parole, vote, resultats)
```

**Wireframe target structure:**
```
app-shell
  app-sidebar
  meeting-bar (compact: header + actions row)
  kpi-strip (flex row, flex-shrink:0)
  progress-track (flex row, flex-shrink:0)
  app-main (flex: 1, display: flex, flex-direction: column, overflow: hidden)
    [setup mode] → existing hub-layout tabs unchanged
    [exec mode] → split panel:
      left panel (flex:1) → resolution card + sub-tabs (scrollable inside)
      right panel (width:200px) → ordre-du-jour list
    guidance bar (flex-shrink:0, at bottom)
```

**Key insight:** The operator page already handles this via CSS grid (`grid-template-rows: auto auto auto 1fr` in operator.css). The exec split panel needs to be added as a new HTML section toggled by JS (already done with `viewSetup`/`viewExec` pattern). The CSS needs:
- `.operator-split` container: `display: flex; gap: 10px; flex: 1; min-height: 0; overflow: hidden`
- `.operator-left`: `flex: 1; display: flex; flex-direction: column; min-width: 0`
- `.operator-right`: `width: 200px; flex-shrink: 0; display: flex; flex-direction: column`
- Inner content areas must use `overflow-y: auto` with `flex: 1; min-height: 0`

**CRITICAL — JS selector risk:** `operator-exec.js` and `operator.js` use DOM selectors that target existing HTML elements. Any HTML restructuring MUST preserve all existing IDs (`#tabVote`, `#tabResultats`, `#tabParole`, `#tabPresences`, etc.). The split panel replaces the TAB STRUCTURE visually but must keep the same content DIVs with their IDs intact.

### Settings — Sidebar Nav Layout

Current settings uses horizontal `.settings-tabs` + `.settings-panel` with a `.container` wrapper. Wireframe uses:

```jsx
<div style={{display:"flex", gap:14, minHeight:0}}>
  <div style={{width:200, flexShrink:0}}>  // nav vertical
    {tabs.map(t => <button>)}
  </div>
  <div style={{flex:1, minWidth:0}}>  // content area
    {tabContent}
  </div>
</div>
```

**Implementation:** Replace `.container > .settings-tabs + .settings-panel` with:
```html
<div class="settings-layout">
  <nav class="settings-sidenav">
    <button class="settings-sidenav-item active" data-stab="regles">...</button>
    <!-- etc -->
  </nav>
  <div class="settings-content">
    <!-- existing tab panels unchanged -->
  </div>
</div>
```

New CSS in `settings.css`:
```css
.settings-layout {

  min-height: 0;
}
.settings-sidenav {
  width: 200px;
  flex-shrink: 0;

  align-items: center;
```

### Wizard — Fixed Footer Nav

The wireframe uses `step-nav` (already defined in wizard.css) as the navigation between steps. The locked decision is that each step must fit without scroll, meaning the step-nav must be fixed or sticky at the bottom.

Current `step-nav`:
```css
.step-nav {
  display: flex;
  gap: 8px;
  justify-content: flex-end;
  margin-top: 16px;
  padding-top: 14px;
  border-top: 1px solid var(--color-border-subtle);
}
```

**Change needed:** Make `.app-main` in wizard use flex-column layout; the `.card.wiz-step` becomes flex with internal scroll, and `.step-nav` becomes `position: sticky; bottom: 0; background: var(--color-surface);` — or the `.card` itself is `display:flex; flex-direction:column; flex:1; overflow:hidden` with inner content scrollable and `.step-nav` pinned at bottom via `flex-shrink:0`.

### Dashboard — Density Reduction

Current paddings producing excess height:
- `.kpi-grid.dashboard-grid` has `margin-bottom: 1.5rem (24px)` → reduce to 14px
- `.grid-2.dashboard-grid` has `margin-bottom: 1.25rem (20px)` → reduce to 14px
- `.dashboard-panel` has `padding: 20px` → reduce to 16px
- `.kpi-card` padding is `var(--space-5)` = 20px → reduce to ~12px
- `.shortcut-card` padding is `20px 18px` → reduce to `14px`
- `.urgent-card` padding is `14px` ✓ already wireframe

The wireframe dashboard total height budget at 1080px (header 56px, footer ~32px, app-main padding 16px top+bottom):
- Usable content height: 1080 - 56 - 32 - 32 = ~960px
- urgent card: ~72px + 14px margin = ~86px
- KPI row: ~80px + 14px = ~94px
- grid-2 panels: ~180px + 14px = ~194px
- grid-3 shortcuts: ~72px
- Total: ~446px — fits comfortably

### Anti-Patterns to Avoid

- **Touching design-system.css:** Token and component CSS is locked from phase 20.2. Only page-specific CSS and HTML.
- **Removing IDs referenced by JS:** `operator.js` and `operator-exec.js` target specific element IDs. The split panel restructure must preserve all existing IDs on content sections.
- **Global `.card-body` padding reduction:** This would affect all pages simultaneously. Instead reduce per page via scoped overrides.
- **CSS-only structural changes:** The locked decisions say this phase touches HTML + CSS. Don't try to fake a sidebar nav for settings with CSS only on existing horizontal tabs.

| Problem | Don't Build | Use Instead | Why |

## Common Pitfalls

### Pitfall 1: JS Selector Breakage on Operator Restructure
**What goes wrong:** Restructuring `operator.htmx.html` to use split panel changes the DOM structure, breaking CSS selectors in `operator.js` and `operator-exec.js` that assume current tab/panel structure.
**Why it happens:** JS files use `getElementById`, `querySelector`, and `data-tab` selectors.
**How to avoid:** Before restructuring, audit all IDs and selectors in `operator-exec.js` and `operator.js`. Keep all existing element IDs intact. Only add new wrapper elements; don't remove or rename existing ones.
**Warning signs:** JS errors on page load after HTML changes.

### Pitfall 2: Hub Layout Already Done — Double Work
**What goes wrong:** Hub already has `.hub-layout-body` / `.hub-stepper-col` (220px) / `.hub-main-col` in `hub.css`. Blindly "implementing" the wireframe layout duplicates or conflicts.
**Why it happens:** Research found existing implementation matches wireframe spec.
**How to avoid:** For hub, VERIFY the existing layout matches wireframe dimensions (it does: 220px left, 24px gap, flex:1 right). Focus on COMPACTION: reduce `.hub-identity` padding, reduce `.hub-action` padding, verify `.hub-details-body` doesn't scroll unnecessarily.
**Warning signs:** If you're adding new `.hub-layout-body` CSS when it already exists.

### Pitfall 3: Dashboard Viewport Calculation
**What goes wrong:** Reducing card padding per wireframe values still doesn't make dashboard fit in 1080px because the total height of all sections adds up to more than available.
**Why it happens:** Dashboard has: header (56px) + app-main padding (16px top) + urgent card + KPIs + grid-2 + grid-3 + app-footer (~32px).
**How to avoid:** Calculate bottom-up. The KPI cards with `text-align: center` and `var(--space-5)` padding are the biggest culprits. Target `.kpi-card` padding reduction to `12px 10px`. Also check `.dashboard-grid` margins.
**Warning signs:** Dashboard still scrolls at 1080px after padding changes.

### Pitfall 4: Table Padding Creates Horizontal Overflow
**What goes wrong:** Reducing `td` padding to 8px vertical but keeping wide column widths causes columns to overflow horizontally on smaller screens.
**Why it happens:** Tables with many columns have minimum widths on `th`/`td` elements.
**How to avoid:** Apply `padding: 8px 12px` (8 vertical, 12 horizontal) — not `8px` all sides. Keep horizontal padding reasonable.
**Warning signs:** Tables overflow horizontally after padding change.

### Pitfall 5: Settings Layout Container Width
**What goes wrong:** The settings `.container` wrapper constrains max-width, but the new sidebar+content layout needs full width of `.app-main`.
**Why it happens:** Current `settings.htmx.html` wraps content in `<div class="container">` — a max-width container class.
**How to avoid:** Remove the `.container` wrapper or replace it with the `.settings-layout` flex container that fills the full `.app-main` width.
**Warning signs:** Settings sidebar appears narrow or misaligned.

### Pitfall 6: Wizard Fixed Footer Cuts Off Content on Short Steps
**What goes wrong:** Making `.step-nav` sticky/fixed on short wizard steps (step 4: récapitulatif) leaves a large gap below the step content before the nav.
**Why it happens:** Sticky positioning within a flex column creates visual awkwardness when content is short.
**How to avoid:** Use `margin-top: auto` on `.step-nav` within a flex column context to push it to the bottom naturally: `.card.wiz-step { display: flex; flex-direction: column; } .step-nav { margin-top: auto; }` — this achieves the wireframe's "footer always at bottom" behavior without fixed positioning artifacts.
**Warning signs:** Large visual gap between short step content and footer nav.

## Open Questions

1. **Operator: How many of the 7 execution tabs become the split panel?**
   - What we know: Wireframe shows `tab-vote` area (resolution + ODJ sidebar) as the primary execution view. Tabs for `parole`, `resultats`, `presences` exist in current HTML as hidden `tab-btn-live` tabs.
   - What's unclear: Whether the wireframe sub-tab pattern (Résultat/Avancé/Présences within the left panel) replaces the top-level execution tabs entirely.
   - Recommendation: Keep existing tab buttons (parole, vote, resultats) as they are for non-split-panel mode — but when in execution with the split panel visible, show sub-tabs WITHIN the left panel (Résultat/Avancé/Présences) per wireframe. This minimizes JS changes.

2. **Settings JS: Which class does settings.js use to activate tabs?**
   - What we know: `settings.js` exists and drives tab switching. The current classes are `.settings-tab` and `.settings-panel`.
   - What's unclear: Whether `settings.js` uses `querySelectorAll('.settings-tab')` or a data attribute.
   - Recommendation: Read `settings.js` early in the plan execution to determine what selector to bridge (either add class to new sidenav items, or update the JS selector).

3. **Hub scroll at 1080px: Does hub.htmx.html already fit or does it still scroll?**
   - What we know: Hub has the correct 2-col structure. Hub has `.hub-details-body` hidden by default (collapsed). The visible content is: header (56px) + hub-identity (~80px) + hub-status-bar (~12px) + hub-layout-body (stepper + action card + checklist).
   - What's unclear: Whether the action card + checklist in the right column fits without scroll at 1080px.
   - Recommendation: Verify in-browser at 1080px. If overflowing, reduce `.hub-identity` padding (from 1rem 1.5rem to ~10px 14px) and `.hub-action` padding.
