In hub.js, the existing meeting_id propagation (lines 419-427) filters with:
```javascript
if (s.dest && s.dest.indexOf('/operator.htmx.html') === 0)
```

This excludes the postsession step (dest: '/postsession.htmx.html'). Broaden the filter to also cover postsession.

Replace the single `indexOf` check with a condition that matches BOTH operator and postsession destinations. The cleanest approach:

```javascript
HUB_STEPS.forEach(function(s) {
  if (s.dest && (s.dest.indexOf('/operator.htmx.html') === 0 || s.dest.indexOf('/postsession.htmx.html') === 0)) {
    var u = new URL(s.dest, window.location.origin);
    u.searchParams.set('meeting_id', sessionId);
    s.dest = u.pathname + u.search;
  }
});
```

This preserves the URL.searchParams.set pattern (idempotent on retry per Phase 23 decision HUB-01). The postsession page must already read `meeting_id` from URL params — verify this by checking `postsession.js` for `URLSearchParams` or `meeting_id` usage. If the postsession page uses a different mechanism (dropdown), the URL param is still safe to pass (it will be ignored, not break anything).

Do NOT change any other code in hub.js. Do NOT change the render() call that follows. Only change the if-condition inside the forEach.
