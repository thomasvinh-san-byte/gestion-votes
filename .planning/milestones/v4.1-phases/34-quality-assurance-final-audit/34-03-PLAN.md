  <action>
For each file below, identify the element that sits visually "above" the card surface (the elevated sub-element) and set its background to `var(--color-surface-raised)`. Follow the three-depth model:
- Layer 1 (body background): `var(--color-bg)` — already applied globally via `body {}`
- Layer 2 (card surface): `var(--color-surface)` — already on most `.card` and panel elements
- Layer 3 (raised/elevated): `var(--color-surface-raised)` — needs to be added

**CRITICAL ANTI-PATTERN:** Do NOT add `--color-surface-raised` to every element. Only apply to elements that are visually elevated ABOVE their card container — sticky toolbars, card sub-headers with distinct background, embedded preview panels, floating action areas. If an element is inline content within a card, it stays at `var(--color-surface)` or inherits.

**File-by-file candidates (read each file first to confirm these elements exist and are appropriate):**

1. **app.css** — `.motion-card` header area or `.vote-result-item` summary bar if they have a distinct background. Look for elements using `var(--color-bg-subtle)` that should be `var(--color-surface-raised)` instead (bg-subtle = de-emphasis, surface-raised = elevation — choose based on visual intent).

2. **archives.css** — `.archive-card-header` if it has a gradient or distinct background. Replace `var(--color-bg-subtle)` with `var(--color-surface-raised)` if the header is meant to feel elevated, not de-emphasized.

3. **report.css** — The `.pv-preview` panel (created in plan 02) is elevated content. Add `background: var(--color-surface-raised)` to `.pv-preview` if it doesn't already have it.

4. **users.css** — Find the most appropriate elevated element (e.g., user detail panel, avatar area, or toolbar). If no natural elevated element exists, apply to the `.table-toolbar` or similar sticky header element.

5. **admin.css** — `.dash-kpi-card` or KPI summary area, or modal overlays.

6. **audit.css** — Detail panel or modal overlay if present.

7. **doc.css** — Table of contents rail (`.toc-rail` or similar) or the main content panel.

8. **help.css** — FAQ answer panel (the expanded content area inside an accordion — applying surface-raised makes it feel like it "opens above" the question).

9. **landing.css** — Feature cards or the hero box. The landing page hero could be `var(--color-surface-raised)` to pop against the body background.

10. **login.css** — The login card/form container itself. This is the single raised element on the page.

11. **members.css** — Import panel or member detail/edit panel.

12. **postsession.css** — Result card content areas or the validation confirmation panel.

13. **settings.css** — Setting section cards or the active tab content area.

14. **trust.css** — Chain-of-custody panels or verification result cards.

15. **validate.css** — Validation result panel or success/error state container.

16. **vote.css** — The ballot card on the voter screen. At mobile scale, depth is less critical but applying it to the ballot card still improves the feel.

17. **wizard.css** — The wizard form card or step content area.

**Approach for each file:**
1. Read the file, find elements with explicit `background` declarations
2. Identify which element is the "highest" in the visual hierarchy (above the card surface)
3. Set that element's background to `var(--color-surface-raised)`
4. If the element currently uses `var(--color-bg-subtle)`, evaluate: is the intent elevation (use surface-raised) or de-emphasis (keep bg-subtle)?
5. Verify the element also has `var(--color-surface)` at the card level — the 3-layer model requires all three to coexist

If a file has genuinely no elevated sub-element (e.g., it's a flat list page with no panels), add a comment `/* QA-02: Single-depth page — no elevated sub-element applicable */` and skip it. This is acceptable for validate.css and vote.css if those pages have no natural elevation.
