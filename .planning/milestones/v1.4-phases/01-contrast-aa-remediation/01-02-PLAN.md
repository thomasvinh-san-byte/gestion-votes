<objective>
Retirer tous les fallbacks hex `var(--color-*, #hex)` dans le Shadow DOM des 23 Web Components sous `public/assets/js/components/`. Pattern cible : `var(--color-foo)` (sans second argument).

Purpose: Éviter pitfall #1 — un shift oklch au niveau `:root` (Plan 01-01) laisse les fallbacks hex "stale" dans les shadow trees ; en mode dark ils deviennent des régressions visuelles invisibles au runtime (particulièrement sur première peinture / cold reload). La garantie de résolution des tokens est assurée par le load order `shell.js` établi en v1.3 Phase 14-02.

Output:
- Inventaire des fallbacks retirés
- Code des composants Shadow DOM nettoyé
- CI grep gate vert
</objective>

**Règles strictes :**
- Ne toucher QUE les `var()` contenant un token `--color-*` avec fallback hex.
- NE PAS toucher aux `var(--color-*, oklch(...))` (pattern autorisé par Pitfall #1 "si un fallback doit exister, utiliser oklch").
- NE PAS toucher aux `var(--size-*, ...)`, `var(--radius-*, ...)`, `var(--shadow-*, ...)` — seuls `--color-*` sont scope.
- Conserver le comportement JS inchangé : le shadowRoot templating, les event handlers, le tout reste identique.
- Ne pas réorganiser l'ordre des règles CSS dans les templates `.innerHTML` ou template literals.

Approche conseillée : sed ou édition manuelle fichier par fichier. Après chaque fichier, `php -l` n'est pas applicable (c'est du JS), mais vérifier la syntaxe en sourçant visuellement et via la commande `node -e "require('fs').readFileSync('<file>', 'utf8')"` qui doit réussir.
