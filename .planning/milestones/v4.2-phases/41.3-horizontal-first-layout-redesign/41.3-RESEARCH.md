## User Constraints (from CONTEXT.md)

### Locked Decisions
- Screens are horizontal — design with width as the primary axis, not height
- No more narrow centered forms — wizard's 680px track and 480px field cap are anti-patterns for desktop
- Multi-column by default — 2-3 column field layouts, not single column stacks
- Side-by-side sections — related sections next to each other, not stacked
- Fill the content area — use the full max-width (1200px for most pages) for form layouts
- **Wizard:** Remove 680px max-width track, remove 480px field cap, Step 1 3-column grid (Type | Date | Heure), Step 4 2-column recap
- **Settings:** Use full width with 2-3 column field grids inside each card
- **Hub:** Checklist items use full width; quorum bar + motions list could be side by side

- All multi-column layouts collapse to single column at 768px; at 1024px minimum 2 columns; at 1366px+ 3 columns where content allows

- Whether to use CSS Grid or Flexbox for each layout

## Summary

Every page has been directly audited by reading the CSS and HTML source files. Two files contain the primary offenders: `wizard.css` hosts the 680px `.wiz-content` centering track and the 480px field cap, and `settings.css` hosts the 720px `.settings-content` cap. The design system already has `.form-grid-2` (2-column, collapses at 480px) but lacks `.form-grid-3`. The Hub's main column is already fluid (220px + 1fr), but checklist items stack in a single column and the quorum/motions sections are vertically stacked when they could sit side by side.

The `app-main` container itself has no max-width — it uses `flex: 1` plus `padding: 16px 22px`. So content areas are already theoretically full-width; the problem is internal max-width constraints on `.wiz-content` and `.settings-content`, plus the absence of multi-column grid usage inside form cards.

**Primary recommendation:** Remove `.wiz-content { max-width: 680px }`, remove `.wiz-content … max-width: 480px`, remove `.settings-content { max-width: 720px }`, add `.form-grid-3` to the design system, and apply `form-grid-2`/`form-grid-3` to every form with more than two fields.

## Standard Stack

### Core
| Library/Pattern | Version/Value | Purpose | Why Standard |
|-----------------|---------------|---------|--------------|
| CSS Grid | Native | Multi-column form layout | Sub-pixel alignment, gap control, named areas |
| `form-grid-2` | Already in design-system.css | 2-column `1fr 1fr` with `gap: 16px 12px` | Already used in wizard and admin |
| `form-grid-3` | NEW — to be added to design-system.css | 3-column `repeat(3, 1fr)` | Missing today |
| CSS `@media` breakpoints | 768px collapse | Responsive fallback | Consistent with existing breakpoints |

### Supporting
| Pattern | Value | Purpose | When to Use |
|---------|-------|---------|-------------|
| `form-group-full` | `grid-column: 1 / -1` | Full-width field inside a grid | Textareas, rich text, titles |

| `.hub-live-pair` | New utility class | Side-by-side quorum + motions | Hub page only when both sections visible |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| CSS Grid `form-grid-3` | Flexbox with `flex: 1; min-width: 33%` | Grid is preferred — exact column alignment, no wrap ambiguity |
| Removing `margin: 0 auto` from `.wiz-content` | Keeping it | On very wide monitors (4K, 27") centering within a `max-width: 1200px` shell is acceptable; remove the 680px cap only |

## Architecture Patterns

### Recommended Change Matrix

```
wizard.css
  .wiz-content           max-width: 680px → REMOVE
  .wiz-content field cap max-width: 480px → REMOVE (entire rule block)

settings.css
  .settings-content      max-width: 720px → REMOVE
  media (max-width: 768px) .settings-content max-width: 100% → REMOVE (redundant)

design-system.css  (ADD)
  .form-grid-3           NEW utility class

wizard.htmx.html   (RESTRUCTURE Step 1 and Step 4)
hub.htmx.html or hub.css (ADD .hub-live-pair layout)
```

### Pattern 1: form-grid-3 (new utility)
**What:** 3-column equal grid, collapses to 2-column at 1024px, 1-column at 768px
**When to use:** Short fields — Type, Date, Heure in one row; SMTP host + port + TLS in one row
**Example:**
```css
/* Source: design-system.css — add after .form-grid-2 */
.form-grid-3 {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: var(--space-field) 12px;
}

.form-grid-3 .form-group-full {
  grid-column: 1 / -1;
}

@media (max-width: 1024px) {
  .form-grid-3 { grid-template-columns: 1fr 1fr; }
}

@media (max-width: 768px) {
  .form-grid-3 { grid-template-columns: 1fr; }
}
```

### Pattern 2: Wizard Step 1 — 3-column Identité row
**What:** Type, Date, and Heure group on one row; Title stays full-width above
**When to use:** Step 1 — Identité section
**Example:**
```html
<!-- Title: full width (no grid wrapper) -->
<div class="field">
  <label>Titre</label>
  <input class="field-input" id="wizTitle">
</div>

<!-- Type + Date + Heure: 3-column grid -->
<div class="form-grid-3">
  <div class="field">Type de séance</div>
  <div class="field">Date</div>
  <div class="field">
    <label>Heure</label>
    <div class="time-input-wrap">HH : MM</div>
  </div>
</div>
```

Note: `.time-input-wrap { max-width: 130px }` remains. The grid cell stretches but the time input stays narrow within it — this is visually fine (label aligns, input sits left-aligned at natural width).

### Pattern 3: Wizard Step 4 — 2-column recap
**What:** Session info in left column, participants + resolutions in right column
**When to use:** Review step only
**Example:**
```html
<div class="review-grid"> <!-- grid-template-columns: 1fr 1fr -->
  <div>
    <!-- Session info review-sections -->
  </div>
  <div>
    <!-- Participants review-section + Resolutions review-section -->
  </div>
</div>
```

```css

  gap: var(--space-card);
  align-items: start;

  .review-grid { grid-template-columns: 1fr; }

```

### Pattern 4: Hub live pair
**What:** Quorum hero + motions section side by side
**When to use:** Hub main column when both quorum and motions are visible (post-quorum-check state)
**Example:**
```css
/* hub.css — add */
.hub-live-pair {
  display: grid;
  grid-template-columns: 1fr 1fr;
  gap: var(--space-card);
  align-items: start;
  margin-bottom: var(--space-card);
}

@media (max-width: 1024px) {
  .hub-live-pair { grid-template-columns: 1fr; }
}
```

```html
<!-- hub.htmx.html — wrap quorum + motions in .hub-live-pair -->
<div class="hub-live-pair" id="hubLivePair" style="display:none;">
  <div class="hub-quorum-hero" id="hubQuorumSection">...</div>
  <div class="hub-motions-section" id="hubMotionsSection">...</div>
</div>
```

Note: hub.js currently controls `hubQuorumSection` and `hubMotionsSection` visibility individually. With the wrapper `hub-live-pair`, JS shows/hides the wrapper instead (or continues showing children — the wrapper itself stays always rendered, children toggle). The simpler approach is to keep individual JS control and add `hub-live-pair` as a layout-only wrapper with no visibility logic.

### Anti-Patterns to Avoid
- **Removing `max-width: 130px` on `.time-input-wrap`:** The custom HH:MM spin input physically cannot stretch — it is 2×48px inputs with a separator. Removing this constraint breaks the visual.
- **Converting `.settings-toggle-row` to multi-column:** Toggle rows (label + description + switch) are informational — not data entry fields. They belong as single-column rows with description below.
- **Applying `form-grid-3` to textarea-only forms:** Textareas need full width. Any form that is just one textarea does not benefit from a grid.
- **Setting explicit max-width on `.wiz-content` to "1200px" instead of removing it:** The CONTEXT.md says remove the constraint — use full available content width. App shell with sidebar handles the viewport constraint already.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| 3-column grid breakpoints | Custom per-component media queries | Single `.form-grid-3` utility in design-system.css | Consistency; one breakpoint change affects all 3-col forms |
| Column ratio variants | `.form-grid-2-weighted { grid-template-columns: 2fr 1fr }` etc. | Inline `grid-template-columns` style or separate utility only if used 3+ times | Premature abstraction for one-off cases |
| Responsive wrapper logic | JS-based column toggling | CSS Grid + `@media` | Zero JS overhead |

## Common Pitfalls

### Pitfall 1: The 480px Field Cap Selector
**What goes wrong:** The cap rule in wizard.css uses a complex `:not()` selector:
```css
.wiz-content .form-group:not(.form-grid-2 .form-group) input,
.wiz-content .form-group:not(.form-grid-2 .form-group) select,
.wiz-content .form-group:not(.form-grid-2 .form-group) textarea {
  max-width: 480px;
}
```
If you only remove the `max-width: 680px` from `.wiz-content` but forget to delete this rule, standalone inputs (title, heure) still cap at 480px.
**How to avoid:** Delete the entire rule block, not just the `.wiz-content` max-width line.

### Pitfall 2: `.time-input-wrap` Inside a Grid Cell
**What goes wrong:** When `.time-input-wrap` is placed inside a `form-grid-3` cell, the cell stretches to 1fr but the input inside only uses 130px. This creates a wide empty cell. On narrow (1024px) 2-column fallback, the Heure cell has large blank space.
**How to avoid:** This is acceptable — the label aligns correctly and users understand time pickers are small. Alternatively, pair Date and Heure in a `form-grid-2` and keep Type above as full width. Either approach is valid.
**Warning signs:** If the time group cell looks disproportionately empty at 1366px, switch to the `form-grid-2` pairing for Date+Heure and make Type full-width.

### Pitfall 3: Hub JS Visibility and Wrapper Element
**What goes wrong:** hub.js calls `document.getElementById('hubQuorumSection').style.display = 'none/block'` and `document.getElementById('hubMotionsSection').style.display = 'none/block'` independently. Adding a `hub-live-pair` wrapper that uses `display: grid` means when one child is hidden, the other child takes up the full row as a single column — which looks fine. But if hub.js is also hiding the wrapper itself, the children never show.
**How to avoid:** Do NOT add `id="hubLivePair"` or hide the wrapper from JS. Keep the wrapper as a pure CSS layout container. JS continues to control individual children. The grid simply reflows to 1 column if only one child is visible.
**Warning signs:** Motions section never appears even after JS shows it.

### Pitfall 4: Settings Mobile Responsive Override
**What goes wrong:** There is a media query in settings.css:
```css
@media (max-width: 768px) {
  .settings-content { max-width: 100%; }
}
```
After removing `max-width: 720px` from `.settings-content`, this rule becomes a no-op. It does no harm but leaves dead CSS.
**How to avoid:** Delete both the 720px rule and the mobile override together.

### Pitfall 5: Stepper Progress Bar Width
**What goes wrong:** `.wiz-progress-wrap` is a sticky bar at the top of `.app-main`, not inside `.wiz-content`. It already fills full width. Removing `.wiz-content` max-width does not affect the stepper. No action needed.

### Pitfall 6: form-grid-3 Collapse at Wrong Breakpoint
**What goes wrong:** If `.form-grid-3` collapses to 1-column at 768px (same as `form-grid-2`), a 3-column form looks identical to a 2-column form at tablet. Adding a 2-column intermediate at 1024px gives a smoother degradation.
**How to avoid:** Use the 3-step breakpoint pattern: 3col → 2col at 1024px → 1col at 768px.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Single-column centered forms (Bootstrap card pattern) | Full-width multi-column grids | Industry shift ~2020 | SaaS apps now default to horizontal form layouts at desktop |
| `max-width: 680px` centered track | Full-width content area | This phase (41.3) | Eliminates ~50% horizontal space waste on 1366px screens |
| `form-grid-2` only | `form-grid-2` + `form-grid-3` | This phase (41.3) | Enables 3-field rows for Type/Date/Heure and SMTP config |

**Deprecated/outdated in this codebase:**
- `max-width: 680px` on `.wiz-content`: A mobile-first carryover from early wizard design. Explicitly named as anti-pattern in CONTEXT.md.
- `max-width: 480px` field cap: Was added to prevent inputs looking stretched in the 680px track. Becomes unnecessary once the track is removed.
- `max-width: 720px` on `.settings-content`: Copied from the original settings sidenav pattern. With a fluid grid column, this cap is unnecessary.

## Open Questions

1. **Wizard Step 2 — member list at wide widths**
   - What we know: `.member-row` is a flex row (name | votes | actions). At 1300px+ each row becomes a very wide flex container with huge name truncation area.
   - What's unclear: Should member list become a proper table with fixed columns (name, email, votes, actions) at wide widths? The CONTEXT.md says "Member list as a wide table/grid."
   - Recommendation: Convert `.member-row` to a CSS Grid with fixed column widths: `grid-template-columns: 1fr 180px 80px 40px` (name | email | votes | delete). This is a behavior decision — acceptable for Claude's Discretion.

2. **Hub `.hub-live-pair` — when to activate**
   - What we know: The quorum and motions sections are toggled by `hub.js` via individual `style.display`.
   - What's unclear: The JS code needs minimal change — wrapping the two elements in a layout-only div is safe. But if hub.js uses `document.getElementById('hubQuorumSection').closest('.hub-main-col')` for any logic, adding an intermediate wrapper may break traversal.
   - Recommendation: Read `hub.js` to verify before adding wrapper. If unsafe, apply the 2-column via a CSS `.hub-main-col > .hub-quorum-hero + .hub-motions-section` adjacent-sibling approach instead (though grid adjacent-sibling is not directly possible in CSS — would require a wrapper).
