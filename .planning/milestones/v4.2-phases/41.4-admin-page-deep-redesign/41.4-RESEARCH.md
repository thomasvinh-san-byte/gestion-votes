## Summary

Admin.htmx.html is 1190 lines of HTML across 7 tab panels plus a full dashboard overview section. The page contains two distinct visual systems that never converged: a `.dash-kpi` pattern for the top overview KPIs and a `.kpi-card`/`.kpi-grid` pattern for the users KPI strip — neither displays contextual detail text. Tab navigation already has the correct underline-border skeleton in app.css but the icons are 16px and static; the active state is color-only with no animation.

The users section has a structural split: the HTML renders a `<table class="users-table">` with `<tbody id="usersTableBody">` but the CSS defines both `.user-row` (flex row pattern) and `.user-avatar-initials` (table circle pattern). The JS exclusively renders `<tr>` rows with the table pattern. The `.user-status-badge` badges are explicitly set to `0.65rem` — confirmed cramped. Pagination exists but uses plain text "Précédent/Suivant" buttons with no chevron icons and page number buttons are only 28x28px.

Policy cards have structure (`.policy-card`, `.policy-info`, `.policy-actions`) but no icon, no status badge, and action buttons are `btn-ghost btn-xs` text links rendered inline. The state diagram JS renders `<span class="state-arrow-visual">→</span>` Unicode arrows between nodes — a literal `→` character at `font-size: var(--text-2xl)`. The system health strip is six items in a single flex row sharing a `0.95rem` value display — serviceable but visually lightweight compared to the full-card treatment needed.

**Primary recommendation:** Work section by section — dashboard KPIs first (highest visual impact, above the fold), then tabs, then users panel, then policies, then state diagram, then system health. CSS changes stay in admin.css; JS changes only where rendering functions need updating (renderUsersTable, renderVoteList, renderQuorumList, state flow JS).

### Anti-Patterns to Avoid
- **Switching to a table for user rows:** The `.user-row` flex pattern enables hover-reveal actions; `<table>` `<tr>` hover doesn't compose well with absolutely-positioned hover overlays.
- **Adding new JS components:** No new Web Components. Use CSS-only patterns for hover states, transitions.
- **Animating the tab indicator with JS:** CSS-only transition on `border-bottom-color` is sufficient and already present in app.css. JS animation adds complexity for no benefit.
- **Removing the `inline-form` class from meeting-roles assign form:** The `inline-form` layout (stacked form-groups in a row) should become `.form-grid-3` — but verify the `ag-searchable-select` components stretch correctly inside a grid cell.

## Common Pitfalls

### Pitfall 1: JS References `.btn-edit-user` by Class in Event Delegation
**What goes wrong:** The `usersTableBody` delegated click handler searches for `.btn-edit-user`, `.btn-toggle-user`, `.btn-password-user`, `.btn-delete-user` on `<tr>` rows. If the HTML structure changes from table to flex, the event delegation target must update from `usersTableBody` to the new container ID.
**How to avoid:** When switching from `<table>` to `.users-list`, update the event delegation target from `document.getElementById('usersTableBody')` to `document.getElementById('usersListContainer')` (or new ID).
**Warning signs:** Click handlers silently fail — no modal opens when clicking action buttons.

### Pitfall 2: `filterRole` Change Handler Uses `loadUsers`
**What goes wrong:** `document.getElementById('filterRole').addEventListener('change', loadUsers)` — if the filterRole select becomes filter-tab pills (as done in users.htmx.html), the change event won't fire.
**How to avoid:** Keep the filterRole as a `<select>` for now, or if converting to pills, add click handlers explicitly.
**Warning signs:** Role filter stops working silently.

### Pitfall 3: `ag-tooltip` Wrapper Inside `.kpi-card` Grid Layout
**What goes wrong:** STATE.md documents this pitfall explicitly: "Users KPI strip uses ag-tooltip inside kpi-card wrapping kpi-label only — wrapping the whole card would break the grid." Adding `ag-tooltip` as a wrapper around an entire `.kpi-card` breaks grid layout.
**How to avoid:** Wrap only the `.kpi-label` span, not the outer `.kpi-card` element.

### Pitfall 4: Inline Styles on State Nodes Override Class
**What goes wrong:** `transitionsBody` JS renders state nodes with `style="padding:0.5rem 0.75rem;font-size:0.8rem"` inline. If admin.css increases `.state-node-visual` padding, the inline style overrides it for table rows.
**How to avoid:** Remove inline styles from `transitionsBody` renderer; add a `.state-node-sm` modifier class instead.

### Pitfall 5: `admin-panel.active` + `animation` Conflict
**What goes wrong:** If `.admin-panel` uses `display: none` and `.admin-panel.active` uses `display: block` + `animation`, the fadeIn animation works on entry but there's no exit animation (CSS can't animate `display: none`). The user sees a flash on first load if all panels try to animate.
**How to avoid:** Apply the animation only. The first active panel (default) should not animate on page load. Use `@starting-style` if needed, or accept the clean snap on initial load.

### Pitfall 6: User Row vs User Table — CSS Duplication
**What goes wrong:** Both `.user-row` and `.user-avatar-initials` CSS exist. After migration, dead CSS remains in admin.css.
**How to avoid:** When migrating to `.user-row` pattern, comment out or remove the `.users-table`, `.user-avatar-initials`, `.table-col-avatar`, `.table-col-actions`, `.table-col-actions-sm` CSS blocks. These are ~40 lines of now-dead code.

### Pitfall 7: `filterRole` Event — `loadUsers` Resets Role Filter
**What goes wrong:** The current `loadUsers` function re-fetches from API and calls `filterAndRenderUsers`. But `filterAndRenderUsers` reads `searchUser` input, NOT the role filter select. The role filter change handler calls `loadUsers` but loadUsers doesn't apply role filtering.
**How to avoid:** This is a pre-existing bug. When redesigning the filter UI, ensure `filterAndRenderUsers` reads the role filter value and applies it alongside the text search.
