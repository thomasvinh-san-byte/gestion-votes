Pour CHACUN des 6 fichiers :

1. **Remplacer chaque bloc `<div class="modal ...">...</div>` par `<ag-modal id="..." ...>...</ag-modal>`** en respectant l'API du composant. Préserver le contenu fonctionnel (titre, body, footer, actions, attributs `hx-*`, IDs référencés par JS).
   - Si la modale legacy a un `id="XYZ"`, garder le même `id` pour ne pas casser les triggers.
   - Si le composant impose des slots nommés (ex: `<span slot="title">`, `<div slot="body">`), restructurer le contenu en conséquence.
   - Préserver les data-attributes lus par HTMX (`hx-target`, `hx-swap`, etc.).

2. **Adapter les triggers** :
   - Tout `<button>` ou `<a>` qui ouvrait la modale legacy (via `data-bs-toggle="modal"`, `onclick`, ou autre mécanisme) doit :
     a. Ouvrir la `<ag-modal>` via la méthode publique du composant (probablement `document.getElementById('XYZ').open()` ou un attribut `data-modal-target="XYZ"` selon l'API). Adapter au contrat lu dans `ag-modal.js`.
     b. Recevoir `aria-haspopup="dialog"` (MODAL-03).
     c. Recevoir un signifiant visuel : suffixe `…` (ellipsis), icône `<svg>`/`<i>`, ou suffixe textuel explicite (ex: "Voir détail…"). Le choix doit être cohérent avec le style existant du fichier — privilégier l'ellipsis `…` pour les liens textuels et une icône pour les boutons-icônes.
   - Exemple :
     ```html
     <!-- AVANT -->
     <button data-bs-toggle="modal" data-bs-target="#exportModal">Exporter</button>
     <!-- APRÈS -->
     <button type="button"
             aria-haspopup="dialog"
             onclick="document.getElementById('exportModal').open()">
       Exporter…
     </button>
     ```

3. **LEX-02 transverse (rappel)** : si un trigger contient le verbe `Approuver` dans le copy de finalisation (séance/PV), le remplacer par `Confirmer` ou `Valider` selon contexte (cf. plan 04.1). Cas-par-cas.
