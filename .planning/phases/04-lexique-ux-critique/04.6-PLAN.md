2. **Heuristique prévention UI** :
   - **`missing_meeting_id`** → si un trigger UI nécessite un meeting_id, le bouton doit être DÉSACTIVÉ tant que le contexte n'est pas connu. Faisable v2.3 (HTMX `disabled` conditionnel ou CSS `pointer-events: none`).
   - **`invalid_meeting_id`** → si l'ID vient toujours d'une liste UI (pas de saisie manuelle), il sera toujours valide. Faisable v2.3 si on confirme l'absence totale de saisie manuelle. Si saisie manuelle existe (CLI / admin form), garder le rattrapage côté serveur.
   - **`meeting_not_found`** / **`motion_not_found`** → ne peut pas être prévenu UI car la suppression peut survenir entre l'affichage et le clic. Race condition fondamentale → rattrapage serveur indispensable. Améliorer le message (déjà fait par 04.4) suffit.

   - **`missing_meeting_id`** : trouver les triggers HTML qui peuvent être déclenchés sans contexte meeting. Exemple : un bouton "Voir séance" dans la nav globale → l'enrouler dans une condition `{{#if meeting_id}}` ou ajouter `disabled` jusqu'à ce que le contexte soit défini.
   - **`invalid_meeting_id`** : si saisie manuelle inexistante, ce code peut être marqué "prévenu par design v2.3" sans fix nécessaire (juste rationale écrite).
