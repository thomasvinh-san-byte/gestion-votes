# Phase 04 Plan 02 : Migration modales legacy → `<ag-modal>` (MODAL-01 + MODAL-03) — Summary

Migration des 7 modales legacy `.modal` vers le web component `<ag-modal>` sur 6 fichiers HTML, avec adaptation des 6 fichiers JS associés (open/close API), et ajout systématique de `aria-haspopup="dialog"` + signifiant visuel `…` sur les triggers d'ouverture (MODAL-03 : Norman — un focus trap est inutile si l'utilisateur ne sait pas qu'il s'apprête à entrer dans une dialog).

### Mapping legacy → ag-modal

Pour chaque modale, conservation du `id` (référencé par le JS) + mappage de la structure :

| Modale (id)             | Title                       | Size | Slots utilisés          | Bouton X header (legacy) |
| ----------------------- | --------------------------- | ---- | ----------------------- | ------------------------ |
| `launchModal`           | Lancer la séance            | lg   | default + footer        | Supprimé (ag-modal)      |
| `auditDetailModal`      | Détail de l'événement       | lg   | default + footer        | Supprimé (ag-modal)      |
| `editMeetingModal`      | Modifier la séance          | md   | default + footer        | Supprimé (ag-modal)      |
| `deleteMeetingModal`    | Supprimer la séance         | sm   | default + footer        | Supprimé (ag-modal)      |
| `exportsModal`          | Exports                     | lg   | default + footer        | Supprimé (ag-modal)      |
| `auditEventModal`       | Détail de l'événement       | md   | default seul (pas de footer) | Supprimé (ag-modal) |
| `validateModal`         | Confirmation de validation  | md   | default + footer        | Supprimé (ag-modal)      |

`<ag-modal>` fournit :
- son propre header avec `title` + bouton X close (par défaut `closable="true"`)
- son propre backdrop avec click-to-close + `prefers-reduced-motion` respect
- focus trap natif (Tab/Shift+Tab cycliques sur les éléments focusables du shadow DOM + slotted)
- Escape close
- Restore focus sur le `previouslyFocused` (capturé dans `open()`)

### Adaptation JS — pattern par fichier

Tous les JS suivent le même pattern post-migration :

```js
// AVANT
modal.hidden = false;        // ou modal.style.display = 'flex'/'block'
// listeners backdrop click + Escape + bouton X header
modal.hidden = true;

// APRÈS
if (typeof modal.open === 'function') modal.open();
// Plus aucun listener backdrop/Escape/X header (gérés en interne)
if (typeof modal.close === 'function') modal.close();
```

Cas spécial **validate.js** : le reset de l'état UI (checkbox + bouton confirm disabled) doit s'exécuter quel que soit le mode de fermeture (Annuler / X / Escape / backdrop click). Solution : listener sur l'événement custom `ag-modal-close` que le composant dispatch dans `close()`.

```js
validateModal.addEventListener('ag-modal-close', () => {
  if (confirmCheckbox) confirmCheckbox.checked = false;
  if (btnModalConfirm) btnModalConfirm.disabled = true;
});
```

## Deviations from Plan

### Rule 3 — Auto-fix blocking issue : extension de scope aux fichiers JS

**Trigger** : sans adaptation des 6 fichiers JS associés (`audit.js`, `meetings.js`, `archives.js`, `trust.js`, `validate.js`, `operator-tabs.js`), les modales ne s'ouvrent plus du tout. Le code legacy fait `modal.hidden = false` ou `modal.style.display = 'flex'`, mais `<ag-modal>` ne contrôle pas son `hidden` natif — il pilote un overlay interne via `.open()` / `.close()` qui synchronise `aria-hidden` et l'animation de backdrop.

**Action** : 6 commits ont touché HTML + JS associé en duo. Documenté dans chaque commit message + ici.

**Frontmatter** : le plan listait 6 HTML dans `files_modified`, je liste explicitement les 12 fichiers (6 HTML + 6 JS) dans `affects` du SUMMARY pour traçabilité.

### Rule 1 — Bug fixé en passant : meetings.js openEditModal préremplissage type

**Found during:** migration de meetings.htmx.html

**Issue** : `openEditModal` cherchait `modal.querySelector('input[name="editMeetingType"][value="..."]')` (radios). Le markup, pourtant, est un `<select id="editMeetingType">` depuis longtemps. Le `querySelector` retournait `null`, le pré-remplissage du type de séance ne fonctionnait jamais — bug silencieux préexistant.

**Fix** : remplacé par `var typeSelect = document.getElementById('editMeetingType'); if (typeSelect) typeSelect.value = typeVal;`. Inclus dans le commit `506655e`.

**Files modified** : `public/assets/js/pages/meetings.js`
**Commit** : `506655e`

### Décision out-of-scope : `op-quorum-modal`

L'élément `<div class="op-quorum-modal">` (operator.htmx.html L1613) a été **conservé** malgré sa suggestivité de "modale" :
- Sa CSS class est `.op-quorum-modal` (custom), pas `.modal` legacy.
- Le grep d'acceptance `grep -F 'class="modal'` ne matche que les classes commençant par `modal` (préfixe), donc `class="op-quorum-modal"` passe. ✓
- Le wrapper parent `<div class="op-quorum-overlay" ... role="dialog" aria-modal="true">` fournit déjà la sémantique a11y de dialog — pas de regression.
- Migrer ce composant nécessiterait de revoir le CSS sur-mesure de la quorum warning (background, layout — différent du pattern `<ag-modal>`).

Suivi : si une harmonisation totale est souhaitée v2.4, créer un plan dédié `op-quorum-modal-migration` et migrer aussi le CSS associé.

### Décision out-of-scope : `.session-menu-btn` (meetings)

Ce bouton ouvre un `<ag-popover>` (menu de 4 actions : Voir / Modifier / Supprimer / Archiver), pas une `<ag-modal>` directement. La modale est ouverte via la sélection d'un item du popover (logique JS `handlePopoverAction`). 

Sémantiquement, `aria-haspopup="menu"` est plus juste qu'`aria-haspopup="dialog"`. Le composant `<ag-popover>` peut ou non porter cet attribut nativement (pas inspecté ici, hors-scope). Pas de modification.
