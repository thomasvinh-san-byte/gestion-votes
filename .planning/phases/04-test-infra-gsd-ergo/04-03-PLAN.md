  <how-to-verify>
**Préalable** : être sur une branche / commit qui contient des modifications v2.6 substantielles. Idéal : tous les commits v2.6 Phase 1+2+3 (≥30 fichiers modifiés au total entre Heartbeat, ErrorEvents, et Tokens cleanup).

**Choix du scope pour la review** :
- Option A — review focalisée sur **un scope** (la plus utile pour valider le flag) :
  ```
  /gsd:code-review --scope=php --timeout-min=45
  ```
  ou
  ```
  /gsd:code-review --scope=tests --timeout-min=30
  ```

- Option B — review complète multi-scope avec timeout étendu :
  ```
  /gsd:code-review --scope=all --timeout-min=90
  ```

**Procédure** :

1. Vérifier le diff range : `git diff --name-only main..HEAD | wc -l` doit retourner ≥30 fichiers (ajuster diff_base si nécessaire pour atteindre le seuil v2.6 commits).

2. Lancer la review (option A recommandée — exerce un scope split, plus diagnostic qu'un `all`) :
   ```
   /gsd:code-review --scope=php --timeout-min=45
   ```

3. Observer :
   - L'agent doit consommer le flag `--scope=php` (filtrer aux fichiers `app/**/*.php` + `public/api/**/*.php`)
   - L'agent doit respecter `--timeout-min=45` (pas de timeout brutal au-dessous)
   - Si les fichiers PHP en scope dépassent 50 ou 500KB : l'agent **doit chunker** automatiquement (pattern documenté lignes 407-421 de l'agent .md) — vérifier qu'un `.planning/.review-progress.json` apparaît temporairement et est nettoyé en fin
   - Le REVIEW.md doit être produit dans `.planning/phases/04-test-infra-gsd-ergo/` (ou phase courante du dispatch)

4. Capturer pour archivage :
   - La commande exacte invoquée
   - Le scope effectivement reviewed (`grep files_reviewed_list` dans REVIEW.md → liste fichiers)
   - Le nombre de fichiers (`grep files_reviewed:`)
   - La durée totale (depuis l'invocation jusqu'à l'écriture de REVIEW.md)
   - Le path du REVIEW.md généré
   - Le statut (`status: clean | issues_found`)
   - Si chunking déclenché : noter le nombre de chunks

1. Aucun timeout (l'agent termine avant la limite imposée par `--timeout-min`)
