  <action>
Read the following files to gather all data needed for the audit:

1. **Route inventory** -- Read `app/routes.php` and extract every route that handles mutations:
   - All explicit POST, PATCH, PUT, DELETE routes (from `mapMulti` and `map`)
   - All `mapAny` routes (these accept any HTTP method including POST)
   - Include HTML page routes that handle POST (e.g., `/vote`, `/reset-password`, `/setup`, `/account`)
   - Exclude: GET-only routes, page shell routes (GET only), bootstrap routes (email tracking pixel/redirect)

2. **IdempotencyGuard usage** -- Search all controller files for `IdempotencyGuard` references:
   ```bash
   grep -rn "IdempotencyGuard\|idempotency" app/Controller/ --include="*.php"
   ```
   Document which controller methods use IdempotencyGuard.

3. **DB UNIQUE constraints** -- Read `database/schema-master.sql` and extract all UNIQUE constraints and unique indexes that prevent duplicate inserts. Key ones from CONTEXT.md:
   - `ballots(motion_id, member_id)`
   - `attendances(tenant_id, meeting_id, member_id)`
   - `invitations(tenant_id, meeting_id, member_id)`
   - `proxies(tenant_id, meeting_id, giver_member_id)`
   Verify these and find any additional UNIQUE constraints.

4. **Rate limiting** -- Note which routes have `rate_limit` middleware config (already visible in routes.php).

5. **Produce the audit document** at `.planning/phases/01-audit-et-classification/01-IDEMPOTENCY-AUDIT.md` with:

   **Header section:**
   - Title: "Audit Idempotence -- Inventaire des Routes Mutantes"
   - Date, phase reference
   - Summary counts: total mutating routes, protected by IdempotencyGuard, protected by UNIQUE constraint, CSRF only, by risk level

   **Main audit table** with columns per D-02:
   | Route | Method | Controller::method | Current Protection | Risk Level | Notes |

   **Protection values:**
   - `IdempotencyGuard` -- uses the guard middleware
   - `UNIQUE constraint` -- DB prevents duplicates
   - `CSRF only` -- only CSRF token protection, no idempotency
   - `Rate limit + CSRF` -- has rate limiting but no idempotency
   - `Upsert` -- operation is inherently idempotent (INSERT ON CONFLICT UPDATE)

   **Risk classification per D-03/D-04/D-05:**
   - **Critique** -- duplicate has direct business impact: double vote, double creation of meeting/member/email-template/export-template, double email send, double import creating duplicates
   - **Moyen** -- duplicate creates noise: duplicate reminder, duplicate attachment upload, duplicate group assignment
   - **Bas** -- duplicate is harmless or DB-protected: update operations (idempotent by nature), delete operations (idempotent), UNIQUE-constraint-protected inserts

   **Phase 2 targets section:**
   - List all Critique-risk routes that lack IdempotencyGuard AND lack UNIQUE constraint
   - These are the exact routes Phase 2 must protect

   Order routes grouped by controller for readability.
  </action>
