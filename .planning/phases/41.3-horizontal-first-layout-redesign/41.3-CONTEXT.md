# Phase 41.3: Horizontal-First Layout Redesign - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign all page layouts and forms to exploit horizontal screen space. Screens are 1920x1080 (or 1366x768 minimum) — forms and content sections must use the full available width, not stack vertically in narrow columns. This affects every page with forms, card grids, and content sections.

</domain>

<decisions>
## Implementation Decisions

### Core Principle: Horizontal First
- **Screens are horizontal** — design with width as the primary axis, not height
- **No more narrow centered forms** — wizard's 680px track and 480px field cap are anti-patterns for desktop
- **Multi-column by default** — 2-3 column field layouts, not single column stacks
- **Side-by-side sections** — related sections next to each other, not stacked
- **Fill the content area** — use the full max-width (1200px for most pages) for form layouts

### Wizard (Primary Target)
- **Remove 680px max-width track** — form fills the full content width (up to 1200px with shell sidebar)
- **Remove 480px field cap** — fields use .form-grid-2 or .form-grid-3 to fill available width
- **Step 1 (Infos):** 3-column grid — Type | Date | Heure on one row. Lieu | Adresse on second row. Title full width above
- **Step 2 (Participants):** Member list as a wide table/grid, not a narrow stacked list. Upload zone full width
- **Step 3 (Résolutions):** Resolution list as horizontal cards in a grid, not vertical stack. Add form as side panel or inline wide
- **Step 4 (Récapitulatif):** Summary as 2-column layout — left: session info, right: participants + resolutions
- **Stepper stays horizontal at top** — already compact from Phase 36

### Dashboard
- **KPI row + session list already horizontal** — verify density is good
- **Quick actions aside** — already 280px sidebar — verify it's well utilized

### Settings/Admin
- **Form sections** — use full width with 2-3 column field grids inside each card
- **Sidenav + content** — already 220px + fluid — verify content fills width

### Hub
- **Stepper sidebar (220px) + main content** — checklist items should use full width
- **Quorum bar + motions list** — could be side by side instead of stacked

### Data Tables
- **Already horizontal** — tables use full width. Verify toolbar/filter layout uses width well

### Operator Console
- **Already horizontal** — 280px sidebar + fluid main. Verify live panel uses width

### All Forms Across App
- **Any form with >2 fields** should use .form-grid-2 or .form-grid-3
- **Short fields (date, time, number, select)** should NEVER be full width — always paired
- **Only textareas and rich text** should be full width
- **Labels inline with fields** where space allows, above fields only on mobile

### Responsive Consideration
- All multi-column layouts collapse to single column at 768px
- At 1024px: minimum 2 columns
- At 1366px+: 3 columns where content allows

### Claude's Discretion
- Exact column ratios for each form (equal columns vs weighted)
- Whether to use CSS Grid or Flexbox for each layout
- Exact breakpoint behavior for each page
- Which sections are candidates for side-by-side treatment beyond forms

</decisions>

<specifics>
## Specific Ideas

- Think of every form like a spreadsheet row — fields flow horizontally, not vertically
- The wizard should feel like filling out a Stripe checkout — everything visible at once, no scrolling
- Settings should feel like Notion's settings — wide, clean, fields use available space
- If a user has a 27" monitor, the app should look good on it — not leave 60% of screen blank

</specifics>
