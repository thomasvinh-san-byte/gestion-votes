  <action>
Rewrite `MeetingsController::createMeeting()` to:

1. **Field mapping (BEFORE validation):** Read raw wizard fields from `$data`:
   - `$data['type']` -> `$meetingType` (default `'ag_ordinaire'`)
   - `$data['date']` + `$data['time']` -> `$scheduledAt` (combined as `"YYYY-MM-DD HH:MM:00"`, null if date empty)
   - `$data['place']` -> `$location`
   - `$data['quorum']` -> `$quorumPolicyId` (pass as-is, may be UUID or empty)
   - `$data['defaultMaj']` -> `$votePolicyId` (pass as-is, may be UUID or empty)
   - `$data['members']` -> `$members` (cast to array, default `[]`)
   - `$data['resolutions']` -> `$resolutions` (cast to array, default `[]`)
   - Keep `$data['title']` validated via manual check (min 3 chars) — do NOT run `ValidationSchemas::meeting()->validate($data)` on raw wizard payload because field names mismatch

2. **Wrap entire body in `api_transaction()`:**
   - Generate meeting UUID inside the transaction closure
   - `$repo->meeting()->create(...)` with mapped fields
   - Member loop: for each member, validate `nom` (required) and `email` (required, `filter_var` FILTER_VALIDATE_EMAIL). On invalid: `api_fail()` with 422 and `{ error: true, details: [{ index, field, message }] }`. For valid: `findByEmail()` — if found, increment `$membersLinked`; if not found, `generateUuid()` + `create()` + increment `$membersCreated`. Then `upsertMode($meetingId, $memberId, 'present', $tenantId)`.
   - `voix` field defaults to `1` if not provided.
   - Resolution loop: for each resolution, validate `title` (required, trimmed non-empty). On invalid: `api_fail()` with 422 and details. For valid: `generateUuid()` + `$motionRepo->create()` with `agendaId=null`, `description` from resolution or empty string, `secret=false`, `votePolicyId=null`, `quorumPolicyId=null`.
   - Return result array from transaction closure: `{ meeting_id, title, members_created, members_linked, motions_created }`

3. **After transaction:** `IdempotencyGuard::store($result)` with the FULL expanded result (not just meeting_id + title). Then `api_ok($result, 201)`.

4. **Policy defaults:** Keep existing logic that assigns default vote_policy_id and quorum_policy_id — but move it inside the transaction. If `$data['quorum']` or `$data['defaultMaj']` are non-empty UUIDs, use those instead of the list-first-policy fallback.

5. **Audit log:** Keep `audit_log('meeting_created', ...)` after the transaction — it is non-critical and should not roll back the transaction if it fails.

**Anti-patterns to avoid:**
- Do NOT call `ValidationSchemas::meeting()->validate($data)` on the raw wizard payload (field name mismatch will reject valid data)
- Do NOT put member/motion inserts outside the `api_transaction()` closure
- Do NOT call `clearDraft()` equivalent on the backend — that is frontend-only
- Do NOT generate UUIDs outside the PDO connection context
