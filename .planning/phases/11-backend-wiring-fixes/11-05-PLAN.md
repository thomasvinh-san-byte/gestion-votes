<objective>
Close DEBT-01. The method `MeetingStatsRepository::getDashboardStats(string $meetingId, string $tenantId): array` was written in Phase 2 (v1.0) to consolidate 11 separate COUNT queries into one SQL round-trip with scalar subqueries. It currently is NOT called by `DashboardController::index()` — a direct grep confirms only one hit in the whole `app/` tree: its own definition at `MeetingStatsRepository.php:154`.

The controller currently issues separate queries via `$attRepo->dashboardSummary()`, `$this->repo()->proxy()->countActive()`, `$statsRepo->countOpenMotions()`, etc. (~5-6 distinct calls, and more inside `$motionRepo->listClosedWithManualTally()`). Wiring `getDashboardStats()` reduces the DB round-trips for the core metrics, unlocking the v1.0 perf win.

Purpose: Realize the v1.0 optimization. Prove via test that the method is called.

Output: DashboardController index() uses getDashboardStats(); a PHPUnit test asserts the call and asserts the response payload carries the stats.
</objective>
