# Phase 2: Classes CSS et Inline Cleanup - Context

**Gathered:** 2026-04-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix three categories of CSS/HTML hygiene issues: (1) wizard uses wrong form classes, (2) 42 inline styles in HTML files, (3) shell.js drawer renders with hardcoded inline styles. No visual design changes — just class normalization.

</domain>

<decisions>
## Implementation Decisions

### Wizard Class Migration (UI-04)
- **D-01:** Replace all `field-input` with `form-input` in wizard.htmx.html
- **D-02:** Replace all `field-label` with `form-label` in wizard.htmx.html
- **D-03:** Replace `field-input` on `<select>` elements with `form-select`
- **D-04:** Replace `field-error-msg` with the design-system equivalent if it exists, or keep as-is
- **D-05:** No changes to wizard.js — only HTML class attributes

### Inline Style Cleanup (UI-05)
- **D-06:** Replace `style="display:none"` / `style="display: none;"` with `hidden` attribute (15+ occurrences in members, report, validate, docs, email-templates)
- **D-07:** Move complex inline styles to CSS classes: settings.htmx.html preview div (full card as inline style), analytics donut legend dots, hub popover paragraph
- **D-08:** Keep JS-managed inline styles that are dynamically updated (e.g., width percentages on progress bars in public.htmx.html) — these are runtime values, not static styling

### Shell.js Drawer Refactoring (UI-06)
- **D-09:** Create `.drawer-section`, `.drawer-label`, `.drawer-value`, `.drawer-alert`, `.drawer-badge` classes in design-system.css
- **D-10:** Replace inline style strings in shell.js renderContext/renderReadiness/renderInfos/renderAnomalies with these classes
- **D-11:** Keep the HTML structure generated by JS — only replace style="..." with class="..."

### Claude's Discretion
- Exact class names for new drawer CSS classes
- Whether to create one utility class per inline style or group related ones
- How to handle the settings.htmx.html font-size inline styles on radio labels

</decisions>

## Existing Code Insights

### Wizard Classes to Replace
- `field-input` → `form-input` (defined in design-system.css)
- `field-label` → `form-label` (defined in design-system.css)
- `field-input` on `<select>` → `form-select`
- `field-hint` → keep (no equivalent exists, harmless)
- `field-error-msg` → keep (error display class)

### Inline Style Categories
1. `display:none` → `hidden` attribute (simple, 15+ occurrences)
2. `style="color:var(--color-*)"` → utility classes like `.text-primary` (3 in analytics)
3. `style="cursor:pointer"` → `.cursor-pointer` utility (1 in operator)
4. Complex card-like styles → new CSS class (1 in settings)
5. Font-size overrides → CSS class (2 in settings)

### Shell.js Drawer Pattern
- 4 render functions: renderContext, renderReadiness, renderInfos, renderAnomalies
- Each builds HTML strings with inline styles for layout, colors, borders
- Styles reference design tokens via var() but as inline attributes
