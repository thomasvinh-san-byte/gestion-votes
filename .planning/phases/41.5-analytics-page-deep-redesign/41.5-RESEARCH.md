## Summary

Phase 41.5 is a pure CSS/HTML visual redesign of `analytics.htmx.html`, the statistics dashboard page. No new functionality, no new API calls, no new JS logic — everything is already wired up in `analytics-dashboard.js`. The goal is to transform the current basic vertical layout into a Stripe-quality analytics dashboard with horizontal KPI cards, a featured hero chart, a larger horizontal donut, sticky filter bar, striped tables, card-style anomaly indicators, and a modern sliding tab indicator.

The current page uses `kpi-card` (design-system.css, text-align:center vertical layout) for KPIs, equal-sized 280px chart containers in a 2-col auto-fit grid, a 180px centered donut with legend below, a basic border-bottom tab system, and list-item anomaly indicators with colored left borders. All of these need to be replaced.

The Phase 41.4 admin page established the horizontal KPI pattern (`dash-kpi` in admin.css) which is the exact prototype to port into analytics. All CSS changes live in `analytics.css`; HTML changes in `analytics.htmx.html`; JS in `analytics-dashboard.js` only where class names change.

**Primary recommendation:** Port the `dash-kpi` horizontal KPI pattern from admin.css into analytics.css as `an-kpi` classes, implement the featured hero chart with `chart-card--hero` modifier, expand the donut to 240px with flex horizontal legend, add a sliding tab indicator via CSS position:relative + ::after pseudo-element, convert anomaly items to a 2-col card grid, and add alternating rows to data tables.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sliding tab indicator | Custom JS scrolling indicator | CSS `::after` + `transform:scaleX` | Pure CSS, no element measurement needed for static tabs |
| Tab badge | Dynamic JS count computation | Simple JS textContent assignment | Already have data from `loadAnomalies()` |
| Donut hover highlight | Custom SVG event listeners | CSS `:hover` on `.donut-segment` + stroke-width change | SVG supports :hover natively |
| Sticky filter bar | Intersection observer JS | CSS `position:sticky` | No JS needed, browser handles |
| Responsive grid | JS breakpoint listeners | CSS Grid auto-fit / media queries | Already established pattern |

**Key insight:** All visual changes are CSS-only (layout, spacing, color) except: (1) the anomaly badge count update in JS, (2) `col-numeric` class additions to table cells, (3) the `btn` class change on the refresh button, (4) HTML restructuring for the donut horizontal layout and hero chart extraction from the grid. No new Chart.js logic, no new API calls.

## Common Pitfalls

### Pitfall 1: ag-tooltip wrapper breaks kpi-grid flex layout
**What goes wrong:** Wrapping `.kpi-card` in `ag-tooltip` makes the tooltip element a grid item, not the card.
**Why it happens:** `ag-tooltip` is a Web Component that inserts an outer element between the grid and the card.
**How to avoid:** Already solved in Phase 35 with `analytics.css` rule `analytics-kpi-grid ag-tooltip { display: contents; }` — this rule must be preserved.
**Warning signs:** KPI cards with unexpected widths or wrapping.

### Pitfall 2: Removing .kpi-card text-align:center breaks existing centering
**What goes wrong:** `design-system.css .kpi-card` has `text-align: center` globally. Overriding with `text-align: left` in the analytics context must be scoped.
**How to avoid:** Scope override to `.analytics-kpi-grid .kpi-card { text-align: left; }` — do not modify design-system.css.

### Pitfall 3: Donut SVG math is based on viewBox, not CSS pixel size
**What goes wrong:** Enlarging `.donut-wrapper` to 240px does not change SVG coordinate space. The circumference is 2*PI*50=314 (r=50 in viewBox "0 0 120 120"). No JS changes are needed.
**How to avoid:** Only change CSS `width/height` on `.donut-wrapper`. Leave SVG `viewBox`, `cx`, `cy`, `r` attributes untouched.

### Pitfall 4: Chart hero vs grid — participationChart currently inside .charts-grid
**What goes wrong:** Moving `participationChart` outside `.charts-grid` into a standalone `chart-card--hero` div requires HTML restructuring. The canvas ID `participationChart` must remain unchanged — `analytics-dashboard.js` uses `document.getElementById('participationChart')`.
**How to avoid:** Move the entire `.chart-card` div for participationChart outside the `.charts-grid`. Only the containing structure changes, the `<canvas id="participationChart">` stays identical.

### Pitfall 5: Sticky filter bar top value depends on app-header height
**What goes wrong:** If `--header-height` CSS variable is not defined, `top: var(--header-height, 64px)` falls back to 64px. The app shell uses a fixed header — verify the actual pixel value.
**How to avoid:** Use the fallback `64px` unless a CSS variable is confirmed. Alternatively, use `top: 0` within an `app-main`-scoped sticky context to avoid dependency on header height.

### Pitfall 6: `charts-grid--2col` conflicts with `@media (min-width: 768px)` auto-fit rule
**What goes wrong:** The existing `.charts-grid` already has `grid-template-columns: repeat(2, minmax(0, 1fr))` at 768px+. Adding a `--2col` modifier is redundant. The 3-col breakpoint at 1400px needs to be a modifier, not the default.
**How to avoid:** Use a `.charts-grid--3col` modifier for the 1400px-3-column behavior. The base `.charts-grid` stays 2-col above 768px by default.

### Pitfall 7: Anomaly JS `anomalyItem()` function builds HTML — must match new CSS card classes
**What goes wrong:** The JS renders `.anomaly-item` with specific class names. If CSS changes the structure (e.g., removes `.anomaly-icon` emoji font-size assumptions), but JS still sets `font-size: var(--text-xl)` via CSS, there's a conflict.
**How to avoid:** The CSS override `anomaly-icon { font-size: unset; }` handles the emoji case. The `icon()` helper already returns SVG — the only change needed in JS is ensuring the icon container size is controlled by CSS, not inline styles.

### Pitfall 8: Tab sliding indicator requires gap:0 on tab container
**What goes wrong:** If `.analytics-tabs` keeps `gap: 0.5rem` between tabs, the `::after` underline extends only to the button width but there's a gap in the border. The tab bar border-bottom visual break is acceptable but the underline must be flush.
**How to avoid:** Set `gap: 0` on `.analytics-tabs` and instead use `padding: 0.5rem 1.25rem` on each tab to provide separation. The `border-bottom: 1px solid` on the container forms the continuous line.

## Open Questions

1. **`--header-height` CSS variable existence**
   - What we know: The sticky filter bar needs `top: Npx` to clear the fixed app-header
   - What's unclear: Whether `--header-height` is defined in design-system.css or app.css
   - Recommendation: Use `top: 64px` as fallback (standard header height). Check at implementation time with a quick grep.

2. **Donut segment hover interaction**
   - What we know: CONTEXT.md says "hover segments to highlight + show value"
   - What's unclear: The SVG segments have CSS `transition: stroke-dasharray`. SVG `stroke-width` change on `:hover` is straightforward CSS.
   - Recommendation: Implement as CSS `.donut-segment:hover { stroke-width: 20px; }` — simple, no JS event listeners needed.

3. **Tab badge for non-anomaly tabs**
   - What we know: CONTEXT.md says "Tab badge counts showing number of items per tab"
   - What's unclear: What counts make sense for Participation (meeting count?), Résolutions (resolution count?), Timing (vote count?)
   - Recommendation: Anomalies badge is high value (shows issue count). For other tabs, populate badge with total item count from API data only if Claude's discretion applies. The Anomalies badge is the primary one to implement.
