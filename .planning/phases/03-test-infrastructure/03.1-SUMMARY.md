The plan instructed reactivating F-4 in `modal-focus-trap.spec.js`. After
reading the file, **no F-4 skip exists there** — the modal-focus-trap suite has
2 describe blocks (synthetic + integration `exportsModal`) that already run
unconditionally without any `seedMeeting` dependency.

The actual F-4 skipped test lives in `cockpit-keyboard-shortcuts.spec.js`
(lines 142–163), tagged `@integration`, with this conditional skip:

```js
let helper;
try { helper = require('../helpers/seed-meeting'); } catch (_) { helper = null; }
test.skip(!helper, 'seed-meeting helper unavailable — F-4 smoke test deferred to gap-closure plan');
```

This was the test pending TEST-V24-01. Reactivating it required:
1. Removing the `try/catch + test.skip` gate
2. Updating the call from `helper.seedRunningMeeting()` (0-arg) to
   `seedRunningMeeting(request, { motionsCount: 1 })` (Playwright `request`
   fixture passed in)
3. Exporting a `seedRunningMeeting()` shorthand from the helper that returns
   `{ id }` for backward compatibility with the consuming spec

`modal-focus-trap.spec.js` was left untouched — its tests already pass without
fixtures (synthetic modal injected at runtime, integration test uses static
`#btnExportsModal` on `archives.htmx.html`).

## Triple-Guard Defence-in-Depth

The dev-only seed-meeting endpoint is gated at three layers:

1. **Route registration gate** (`app/routes.php` line ~157) — the
   `$router->map('POST', '.../test/seed-meeting', ...)` call lives inside
   `if (!in_array($appEnv, ['production', 'prod'], true)) { ... }`. In production
   the route is never registered.
2. **EnvGuardMiddleware** (`'env_guard' => true` in route middleware config) —
   even if a developer accidentally moved the route outside the conditional,
   the middleware returns 404 silently when `APP_ENV ∉ {development, dev, test,
   testing, demo}`.
3. **Controller `guardProduction()`** — a final `api_fail('endpoint_disabled', 403)`
   if `APP_ENV ∈ {production, prod}` even when both prior gates are bypassed
   (e.g. someone instantiates the controller directly from a CLI script).

Plus `audit_log('test_seed_meeting', ...)` records every successful call so a
production leak would leave a forensic trail.

### Auto-fixed Issues

**1. [Rule 3 — Blocking] guardProduction() env source**
- **Found during:** Task 8 (writing TestSeedControllerTest)
- **Issue:** Initial implementation used `config('env', 'dev')` which returned
  `'dev'` in tests because `tests/bootstrap.php` stubs `config()` with a
  hardcoded match block, ignoring any reflection write to `Application::$config`.
- **Fix:** `guardProduction()` now reads `$_ENV['APP_ENV'] ?? getenv('APP_ENV') ?? config('env', ...)`,
  letting tests drive production via `$_ENV` while keeping the production code
  path intact (production deployments still set APP_ENV via .env loading).
- **Files modified:** app/Controller/TestSeedController.php
