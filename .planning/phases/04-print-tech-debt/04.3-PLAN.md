# Plan 04.3 — Push to 95% strict (follow-up to 04.2)

## Goal

Compléter le travail de Plan 04.2 (qui a livré 83.2 %) pour atteindre la cible stricte **≥95 %** de TECH-V24-02. Méthode : migration agressive des 24 borders + 16 shadows résiduels + 14 @keyframes shadows via création de **tokens 1-site** dans `design-system.css`. **D-08 ("pas de tokens jetables") est explicitement amendé pour ce plan** — décision utilisateur post-04.2 audit, documentée.

## Context

Plan 04.2 a atteint 83.2 % combiné (Borders 89 %, Shadows 71 %) avec :
- 24 borders `keep-justified` (1.5px widths, 3px, dashed-danger, single-site)
- 16 shadows `keep-justified` (color-mix per-site, glassmorphism, 1-site widths)
- 14 @keyframes shadows (animation geometry)

L'audit complet est dans `.planning/phases/04-print-tech-debt/04.2-AUDIT.md`. Cette source de vérité est obligatoire pour la migration.

## Acceptance Criteria

1. **Ratio combiné final ≥95 %** — vérifié par recalcul grep post-migration. Si pas atteint, REPORT-BLOCKER avec analyse pourquoi.
2. **Tous les 24 borders keep-justified** migrés via tokens (peut nécessiter 1-site tokens nommés sémantiquement, ex: `--border-glassmorphism-thin`, `--border-emphasis-1.5px`, etc.).
3. **Tous les 16 shadows résiduels** migrés via tokens (idem, 1-site OK).
4. **@keyframes shadows** : extraction des shadow values en tokens utilisés DANS les keyframes. Ex:
   ```css
   /* AVANT */
   @keyframes pulse {
     0% { box-shadow: 0 0 0 0 rgba(220, 38, 38, 0.6); }
     50% { box-shadow: 0 0 0 8px rgba(220, 38, 38, 0); }
   }
   /* APRÈS */
   --keyframe-pulse-from: 0 0 0 0 rgba(220, 38, 38, 0.6);
   --keyframe-pulse-to: 0 0 0 8px rgba(220, 38, 38, 0);
   @keyframes pulse {
     0% { box-shadow: var(--keyframe-pulse-from); }
     50% { box-shadow: var(--keyframe-pulse-to); }
   }
   ```
   Counts as "tokenized" pour la métrique.
5. **Tous nouveaux tokens** ajoutés dans `design-system.css` (extension de la section démarrée par `f4da267`).
6. **Aucune nouvelle valeur littérale** introduite — tous les nouveaux tokens composent depuis `--color-*` existants ou capturent verbatim des valeurs hardcoded existantes (préservation visuelle stricte).
7. **Atomic commits** : 1 commit nouveau tokens (`feat(04-3): add residual 1-site tokens for borders/shadows`) + 1 commit per fichier CSS migré + 1 commit SUMMARY métrique finale.
8. **Régression visuelle nulle** : tokens capturent verbatim les valeurs originales — visuellement identique pre/post.

## Task Breakdown

### Task 1 — Lire audit existant

Lire `.planning/phases/04-print-tech-debt/04.2-AUDIT.md` complètement. C'est la source de vérité de TOUS les sites résiduels (24 + 16 + 14 = 54 sites). Sans cet audit, pas de migration ciblée.

### Task 2 — Concevoir naming tokens 1-site

Pour chaque catégorie résiduelle, proposer un schéma de naming :
- **Borders 1-site** : `--border-<context>-<width>` (ex: `--border-glassmorphism-thin`, `--border-cta-emphasis-2px`)
- **Shadows 1-site** : `--shadow-<context>-<role>` (ex: `--shadow-login-glow`, `--shadow-quorum-pulse-mid`)
- **Keyframes shadows** : `--keyframe-<animation>-<step>` (ex: `--keyframe-pulse-from`, `--keyframe-quorum-missed-mid`)

Documenter le schéma dans le commit message + 04.3-SUMMARY.

### Task 3 — Ajouter tokens dans design-system.css

1 commit groupé : `feat(04-3): add 1-site tokens for residual borders/shadows`. Tous les nouveaux tokens en une fois, organisés par section (borders / shadows / keyframes).

### Task 4 — Migrer fichiers résiduels

Pour chaque fichier CSS contenant un site résiduel :
1. Remplacer la valeur hardcoded par `var(--<new-token>)`
2. Atomic commit : `feat(04-3): tokenize residual <category> in <file>.css`

Ordre suggéré : commencer par les fichiers avec le plus de sites résiduels (impact ratio max).

### Task 5 — Métrique finale + SUMMARY

Re-run grep audit. Calculer ratio final. Si ≥95 % → DONE. Si <95 %, reporter REPORT-BLOCKER avec count exact résiduels.
