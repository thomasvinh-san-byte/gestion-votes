# Plan 04.2 — borders/shadows tokens migration

## Goal

Atteindre un **ratio ≥95 %** des `border` et `box-shadow` CSS utilisant les design tokens (`--shadow-*`, `--border-*`) plutôt que des valeurs hardcodées (`1px solid #abc`, `0 1px 2px rgba(...)`). Audit + migration cas-par-cas avec atomic commits per fichier.

Source : TECH-V24-02 (REQUIREMENTS.md), CONTEXT.md décisions D-05..D-09.

## Acceptance Criteria

1. **`04.2-AUDIT.md` livré** : grep complet `border:` et `box-shadow:` hardcoded dans `public/assets/css/` (excluant `design-system.css`). Pour chaque match :
   - File:line
   - Current value (raw CSS)
   - Classification : `migrate-token` (token existant suffit) | `migrate-new-token` (nouveau token nécessaire — exception) | `keep-justified` (raison spécifique documentée)
   - Token cible si migrate
2. **Métrique baseline** : `tokenized_count / (tokenized_count + hardcoded_count)` calculée avant et après migration. Documentée dans `04.2-SUMMARY.md`.
3. **Cible ratio ≥95 %** atteinte. Si <95 % après migration normale, identifier les `keep-justified` résiduels et leur ratio résiduel — ≤5 % du total.
4. **Atomic commits per fichier CSS** : 1 commit = 1 fichier migré, message explicite avec count avant/après.
5. **Tokens utilisés** = tokens existants livrés v2.3 TECH-01 quick : `--shadow-xs`, `--shadow-sm`, `--shadow-md`, `--shadow-lg`, `--border-default`, `--border-subtle`, `--border-strong`, `--border-dashed`, `--border-focus`. Pas de nouveaux tokens créés sauf justification explicite.
6. **Régression visuelle nulle** : vérifier que les pages affectées rendent visuellement identique avant/après (smoke check). Si différence visible, choix de token incorrect → re-classifier.
7. **`design-system.css` UNTOUCHED** : c'est la source de tokens, pas une cible.
8. **Test gardien optionnel** : si pertinent, ajouter `tests/Security/CssTokensTest.php` qui assert ratio ≥95 % via grep/preg_match. Sinon, skipped (la métrique chiffrée du SUMMARY est le test).

## Task Breakdown

### Task 1 — Audit grep ciblé

```bash
# Borders hardcoded (excluding design-system.css and border-radius/border-color)
grep -rnE "(^|[^a-z-])border:\\s+[0-9]+(px|rem|em)\\s+[a-z]+\\s+(#[0-9a-f]+|rgba?\\(|var\\(--color-)" \
  public/assets/css/ \
  | grep -v "design-system.css" \
  > /tmp/borders-audit.txt

# Box-shadow hardcoded
grep -rnE "box-shadow:\\s+[^v]" public/assets/css/ \
  | grep -v "design-system.css" \
  | grep -v "var(--shadow" \
  > /tmp/shadows-audit.txt

# Tokenized counts (baseline)
grep -rcE "border:\\s+var\\(--border-|box-shadow:\\s+var\\(--shadow-" public/assets/css/ \
  | grep -v ":0$"
```

EXPLORE-PATTERNS.md (créé v2.4 P3) recommande : pour CSS BEM hyphen-safe, utiliser `<token>($|[^a-zA-Z_-])` plutôt que `\b`. Adapter regex si faux-positifs.

Produire `04.2-AUDIT.md` avec :
- Section 1 : Borders inventaire (file:line:value:classification:target-token)
- Section 2 : Shadows inventaire (idem)
- Section 3 : Métriques baseline (`tokenized=N1, hardcoded=N2, ratio=N1/(N1+N2)*100`)
- Section 4 : Plan migration (groupé par fichier, ordre exec)

### Task 2 — Migrer borders, fichier par fichier

Pour chaque fichier listé dans audit :

```css
/* AVANT */
.foo {
  border: 1px solid #d4d4d4;
}

/* APRÈS */
.foo {
  border: var(--border-default);
}
```

(Le token `--border-default` est défini comme `1px solid var(--color-border-default)` ou similaire dans design-system.css — vérifier.)

Atomic commit : `feat(04-2): migrate borders in public/assets/css/foo.css (5 → 0 hardcoded)`

### Task 3 — Migrer shadows, fichier par fichier

```css
/* AVANT */
.bar {
  box-shadow: 0 1px 2px rgba(0,0,0,0.05);
}

/* APRÈS */
.bar {
  box-shadow: var(--shadow-xs);
}
```

Atomic commit per fichier.

### Task 4 — Smoke regression visuel

Pour les fichiers les plus modifiés, identifier les pages utilisatrices :
- `vote.css` → `/vote/...`
- `editorial.css` → `/audit`, `/trust`, etc.
- `app.css` → toutes pages

Si dev-machine accessible : ouvrir 2-3 pages avant/après commit → visuellement identique ?
Sinon : compare les classes + visualiser via les valeurs des tokens (ex: `--shadow-xs` est défini comme `0 1px 2px rgba(0,0,0,0.05)` → visuellement identique au hardcoded migré).

### Task 5 — Métrique finale + SUMMARY

Re-run grep audit après toutes migrations. Calculer ratio final :
```
tokenized = grep tokenized count
hardcoded = grep hardcoded residuel count
ratio = tokenized / (tokenized + hardcoded) * 100
```

Si ratio ≥ 95 % → DONE.
Si ratio < 95 % → identifier les hardcoded résiduels, justifier ou migrer.

Documenter dans `04.2-SUMMARY.md` avec table baseline → final.

## Tests à créer

## Risks & Mitigations

| Risk | P | I | Mitigation |
|---|---|---|---|
| Token cible visuel ≠ hardcoded original | M | M | Smoke compare avant/après par fichier. Si différence : re-classifier, garder hardcoded justified ou créer nouveau token (exception) |
| Cible 95 % impossible vu cas légitimes | L | M | Identifier en audit AVANT migration. Si > 5 % légitimes, ouvrir discussion avant migration (gris pour produit) |
| Régression visuelle non détectée sandbox | M | L | Documenter procédure dev-machine smoke check. Si bug remonté terrain → revert atomic commit du fichier concerné |
| Faux-positifs grep (border-radius, border-color) | L | L | Regex strict : `border:\\s+[0-9]+` exclut `border-radius:`, `border-color:`, `border-bottom:`, etc. Tester regex sur sample. |

## Atomic Commit Strategy

N+2 commits où N = nombre de fichiers CSS migrés :
1. `docs(04-2): audit borders/shadows hardcoded (X → Y migrate, Z keep-justified)`
2. `feat(04-2): migrate borders in <file1>.css`
3. `feat(04-2): migrate borders in <file2>.css`
... (1 par fichier)
N+1. `feat(04-2): migrate shadows in <fileX>.css`
N+2. `docs(04-2): SUMMARY plan complete (ratio: 70% → 95%+)`
