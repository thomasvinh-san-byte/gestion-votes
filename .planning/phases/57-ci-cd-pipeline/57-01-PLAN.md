**Job 2: `coverage`** (CI-02 — needs PHP + pcov)
```yaml
coverage:
  runs-on: ubuntu-latest
  steps:
    - name: Checkout
      uses: actions/checkout@v6
    - name: Setup PHP with pcov
      uses: shivammathur/setup-php@v2
      with:
        php-version: "8.4"
        extensions: intl, mbstring, zip, gd, pdo_pgsql, pcov
        coverage: pcov
        tools: composer:v2
    - name: Cache Composer dependencies
      uses: actions/cache@v5
      with:
        path: vendor
        key: composer-${{ hashFiles('composer.lock') }}
        restore-keys: composer-
    - name: Install dependencies
      run: composer install --optimize-autoloader --no-interaction --no-progress
    - name: Run coverage check (Services >= 90%, Controllers >= 60%)
      run: bash scripts/coverage-check.sh
```
The `shivammathur/setup-php` action with `coverage: pcov` installs pcov natively — no manual .deb extraction needed. The `coverage-check.sh` script auto-detects pcov when it is a loaded extension (PCOV_SO stays empty).

**Job 3: `e2e`** (CI-01 — Playwright against Docker stack)
```yaml
e2e:
  needs: [build]
  runs-on: ubuntu-latest
  steps:
    - name: Checkout
      uses: actions/checkout@v6

    - name: Set up Docker Buildx
      uses: docker/setup-buildx-action@v3

    - name: Load Docker image from GHA cache
      uses: docker/build-push-action@v6
      with:
        context: .
        load: true
        push: false
        tags: agvote:ci
        cache-from: type=gha

    - name: Start Docker stack
      run: |
        # Override app image to use the one we just built (not rebuild from Dockerfile)
        export DB_PASS=ci-test-password
        docker compose up -d db redis
        # Wait for postgres + redis healthy
        docker compose exec -T db sh -c 'until pg_isready -U vote_app; do sleep 1; done'
        docker compose exec -T redis sh -c 'until redis-cli -a "agvote-redis-dev" ping | grep -q PONG; do sleep 1; done'
        # Start app using pre-built image
        docker run -d --name agvote-app \
          --network $(docker network ls --filter name=backend -q | head -1) \
          -p 8080:8080 \
          -e APP_ENV=demo \
          -e APP_AUTH_ENABLED=1 \
          -e CSRF_ENABLED=1 \
          -e RATE_LIMIT_ENABLED=1 \
          -e DB_DSN="pgsql:host=agvote-db;port=5432;dbname=vote_app" \
          -e DB_HOST=agvote-db \
          -e DB_PORT=5432 \
          -e DB_DATABASE=vote_app \
          -e DB_USER=vote_app \
          -e DB_PASS=ci-test-password \
          -e REDIS_HOST=agvote-redis \
          -e REDIS_PORT=6379 \
          -e REDIS_PASSWORD=agvote-redis-dev \
          -e LOAD_SEED_DATA=1 \
          agvote:ci
        # Wait for app to be healthy
        for i in $(seq 1 30); do
          if curl -sf http://localhost:8080/api/v1/health.php > /dev/null 2>&1; then
            echo "App healthy after ${i}s"
            break
          fi
          sleep 1
        done
      env:
        DB_PASS: ci-test-password

    - name: Setup Node.js
      uses: actions/setup-node@v6
      with:
        node-version: "22"
        cache: "npm"

    - name: Install npm dependencies
      run: npm ci

    - name: Install Playwright Chromium
      run: npx playwright install chromium --with-deps

    - name: Run Playwright E2E tests (chromium only)
      run: npx playwright test --project=chromium
      working-directory: tests/e2e
      env:
        CI: true
        BASE_URL: http://localhost:8080

    - name: Upload Playwright report on failure
      if: failure()
      uses: actions/upload-artifact@v4
      with:
        name: playwright-report
        path: tests/e2e/playwright-report/
        retention-days: 7

    - name: Cleanup Docker stack
      if: always()
      run: |
        docker rm -f agvote-app 2>/dev/null || true
        docker compose down -v 2>/dev/null || true
      env:
        DB_PASS: ci-test-password
```

IMPORTANT: The `docker compose up` command needs the network name. Use `docker network ls` to find the backend network created by compose. The app container runs with the pre-built `agvote:ci` image to avoid rebuilding. The auth.setup.js `clearRateLimit()` uses `docker exec agvote-redis` which matches the container name from compose.

**Job 4: `integration`** (CI-04 — PHPUnit Integration suite against real DB)
```yaml
integration:
  needs: [build]
  runs-on: ubuntu-latest
  services:
    postgres:
      image: postgres:16.8-alpine3.21
      env:
        POSTGRES_DB: agvote_test
        POSTGRES_USER: agvote
        POSTGRES_PASSWORD: agvote
      ports:
        - 5432:5432
      options: >-
        --health-cmd "pg_isready -U agvote -d agvote_test"
        --health-interval 10s
        --health-timeout 5s
        --health-retries 5
    redis:
      image: redis:7.4-alpine3.21
      ports:
        - 6379:6379
      options: >-
        --health-cmd "redis-cli ping"
        --health-interval 10s
        --health-timeout 5s
        --health-retries 5
  steps:
    - name: Checkout
      uses: actions/checkout@v6

    - name: Setup PHP
      uses: shivammathur/setup-php@v2
      with:
        php-version: "8.4"
        extensions: intl, mbstring, zip, gd, pdo_pgsql, redis
        tools: composer:v2

    - name: Cache Composer dependencies
      uses: actions/cache@v5
      with:
        path: vendor
        key: composer-${{ hashFiles('composer.lock') }}
        restore-keys: composer-

    - name: Install dependencies
      run: composer install --optimize-autoloader --no-interaction --no-progress

    - name: Load base schema
      run: PGPASSWORD=agvote psql -h localhost -U agvote -d agvote_test -f database/schema-master.sql

    - name: Apply migrations
      run: |
        for f in database/migrations/*.sql; do
          echo "Applying: $(basename $f)"
          PGPASSWORD=agvote psql -h localhost -U agvote -d agvote_test -f "$f"
        done

    - name: Run Integration tests
      run: vendor/bin/phpunit --testsuite Integration --no-coverage
      env:
        DATABASE_URL: "pgsql:host=localhost;dbname=agvote_test;user=agvote;password=agvote"
        REDIS_HOST: localhost
        REDIS_PORT: 6379
```

The `integration` job uses GitHub Actions `services:` to spin up PostgreSQL and Redis containers. It loads the schema + migrations before running the Integration test suite. This is cleaner than using docker-compose since Actions services are managed by the runner.

Note on E2E docker network: The compose network name depends on the project directory name. In CI the checkout is in the workspace root, so the network will likely be named something like `gestion_votes_php_backend` or similar. The safest approach is to use `docker compose` directly and override the app service image. Let me refine the E2E approach:

REVISED E2E approach — instead of manual `docker run`, create a `docker-compose.ci.yml` override OR use `docker compose` with image override. Actually, the simplest CI approach: use `docker compose` with the existing `docker-compose.yml` and set `DB_PASS` env var. The `build: .` in compose will use the GHA cache from the earlier `build` job. But this rebuilds — wasteful.

Better approach: after building `agvote:ci` from cache, tag it as what compose expects:
1. Build from GHA cache as `agvote:ci`
2. Tag it: `docker tag agvote:ci $(docker compose config --images | head -1)` or simpler — set `image: agvote:ci` via compose override.
3. `DB_PASS=ci-test-password docker compose up -d`

Simplest: Write a minimal `docker-compose.ci.yml` override inline that sets `image: agvote:ci` and removes `build: .` for the app service. Then: `docker compose -f docker-compose.yml -f /tmp/ci-override.yml up -d`.

Actually simplest: Just let compose build from Dockerfile. The GHA cache (`cache-from: type=gha`) means this is nearly instant (all layers cached from the `build` job). So just:
```bash
DB_PASS=ci-test-password docker compose up -d --build
```
Wait for health, run tests, done. The `--build` uses the local Docker cache which was populated by the `build` job via GHA cache. This is the pragmatic approach.

BUT: the `build` job uses Buildx with GHA cache backend. `docker compose up --build` uses the standard Docker builder, NOT Buildx GHA cache. So layers won't be shared.

FINAL approach: Use `docker compose up -d` but override the image. Write inline compose override:

```bash
cat > /tmp/ci-compose.yml << 'CIEOF'
services:
  app:
    image: agvote:ci
    build: !reset null
CIEOF
DB_PASS=ci-test-password docker compose -f docker-compose.yml -f /tmp/ci-compose.yml up -d
```

Actually `build: !reset null` is not valid YAML for compose. Simpler: just set the image env and use the pre-built tag. Docker compose respects `image:` even when `build:` is present — if the image already exists locally, `docker compose up -d` (without `--build`) will use it.

So:
1. Build and load as `agvote:ci` from GHA cache
2. Re-tag: `docker tag agvote:ci gestion_votes_php-app:latest` (or whatever compose names it)

Actually compose names images as `{project}-{service}`. The project name defaults to directory name. In CI the directory is the repo checkout root. We can control this:

```bash
COMPOSE_PROJECT_NAME=agvote DB_PASS=ci-test-password docker compose up -d
```
Then compose expects image `agvote-app`. So:
```bash
docker tag agvote:ci agvote-app
COMPOSE_PROJECT_NAME=agvote DB_PASS=ci-test-password docker compose up -d --no-build
```

This uses the pre-built image without rebuilding. Container names become `agvote-app-1`, `agvote-db-1`, `agvote-redis-1` (unless `container_name` is set in compose — which it IS: `agvote-app`, `agvote-db`, `agvote-redis`). So `docker exec agvote-redis` will work for the auth.setup.js rate-limit clearing.
