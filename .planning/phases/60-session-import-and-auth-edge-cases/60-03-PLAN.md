    **AuthController::login() — add rate limit check at the very beginning of login(), after the POST method check (line 23), before raw body parsing:**

    ```php
    // ── Rate limiting with audit (AUTH-02) ──
    $ip = $_SERVER['REMOTE_ADDR'] ?? 'unknown';
    $maxAttempts = (int) (getenv('APP_LOGIN_MAX_ATTEMPTS') ?: 5);
    $windowSeconds = (int) (getenv('APP_LOGIN_WINDOW') ?: 300);

    if (RateLimiter::isLimited('auth_login', $ip, $maxAttempts, $windowSeconds)) {
        try {
            audit_log('auth_rate_limited', 'security', null, [
                'ip' => $ip,
                'attempt_count' => $maxAttempts,
                'window' => $windowSeconds,
            ]);
        } catch (Throwable) { /* best effort */ }
        $retryMinutes = (int) ceil($windowSeconds / 60);
        throw new \AgVote\Core\Http\ApiResponseException(
            new \AgVote\Core\Http\JsonResponse(429, [
                'ok' => false,
                'error' => 'rate_limit_exceeded',
                'detail' => "Trop de tentatives. R\u00e9essayez dans {$retryMinutes} minutes.",
                'retry_after' => $windowSeconds,
            ], [
                'Retry-After' => (string) $windowSeconds,
            ]),
        );
    }
    ```

    Add the `use AgVote\Core\Security\RateLimiter;` import at the top of the file if not already present.
    Add `use AgVote\Core\Http\ApiResponseException;` and `use AgVote\Core\Http\JsonResponse;` if not already present.

    Note: Use `ApiResponseException` directly (not `api_fail`) because `api_fail()` does not support custom HTTP headers. The `Retry-After` header is required per user decision.

    Also add `RateLimiter::check('auth_login', $ip, $maxAttempts, $windowSeconds, false);` AFTER the isLimited check to increment the counter for this attempt. Place it right before the credential extraction (before `$rawBody = ...`):

    ```php
    // Increment attempt counter (non-strict — we handle blocking above)
    RateLimiter::check('auth_login', $ip, $maxAttempts, $windowSeconds, false);
    ```

    Do NOT remove the existing route-level rate_limit guard — it provides defense-in-depth.
