**Researched:** 2026-03-18
**Domain:** CSS architecture cleanup, wireframe alignment, JS bug fixes

## Summary

Phase 20.4 addresses the visual inconsistency accumulated across phases 20.1-20.3, where piecemeal CSS patches created contradictory rules across 25 CSS files totaling ~23,600 lines. The wireframe (ag_vote_wireframe.html v3.19.2) defines the canonical design language with specific tokens, spacing, typography, and component patterns that the implementation has drifted from.

The core problem is a layered CSS cascade where the same properties are set in multiple files with conflicting values. The load order is: design-system.css (via app.css @import) -> pages.css (via app.css @import) -> page-specific CSS (via HTML <link>). When pages.css overrides design-system.css globally, and page-specific files re-override locally, the result is unpredictable visual output.

**Primary recommendation:** Clean CSS contradictions in a single "Step 0" pass before touching any page HTML, then work through pages in critical-path order comparing each against the wireframe component.

## Common Pitfalls

### Pitfall 1: Cascade Order Blindness
**What goes wrong:** Fixing a style in design-system.css but pages.css overrides it later, so the fix is invisible.
**Why it happens:** app.css imports design-system.css then pages.css -- pages.css always wins on equal specificity.
**How to avoid:** When consolidating, always check if the property is also set in pages.css before modifying design-system.css.
**Warning signs:** "I changed it but nothing happened."

### Pitfall 2: Global Selector Leaking to Page-Specific Contexts
**What goes wrong:** A global change to `.card` or `.grid-2` breaks a specific page that relied on the old value.
**Why it happens:** 14 pages share `.grid-2`, `.card`, `.btn` classes with different contextual needs.
**How to avoid:** Use scoped selectors (`.dashboard-grid .kpi-card`, `.meetings-main .section-header`) for any override that shouldn't be global.
**Warning signs:** Fixing dashboard layout breaks wizard or hub.

### Pitfall 3: Wireframe Token Mismatch
**What goes wrong:** Using wireframe shorthand tokens (`--bg`, `--accent`) in implementation CSS.
**Why it happens:** Copy-pasting from wireframe source without translating to design-system tokens.
**How to avoid:** Always use `--color-*` prefixed tokens from design-system.css, never wireframe shorthands.

### Pitfall 4: Horizontal Overflow on Dense Tables
**What goes wrong:** Reducing table cell padding causes text overflow on narrow columns.
**Why it happens:** Already documented in pages.css line 1252 -- minimum 12px horizontal padding prevents this.
**How to avoid:** Keep `padding: 8px 12px` minimum for table cells, never go below 12px horizontal.

### Pitfall 5: Wizard Import Button (.xlsx not supported)
**What goes wrong:** User selects an .xlsx file but the parser reads it as text/CSV, producing garbage data.
**Why it happens:** `handleCsvFile()` in wizard.js (line 322) uses `FileReader.readAsText()` and splits by `\n`/`,` -- this is CSV-only parsing but the file input accepts `.xlsx`.
**How to avoid:** Either (a) remove `.xlsx` from the accept attribute and button label, or (b) add SheetJS/xlsx parsing. Option (a) is simpler and within scope.

## JS Bugs Identified

### Bug 1: Wizard Import Button - .xlsx Files Not Parsed (HIGH)
**Location:** `public/assets/js/pages/wizard.js` line 322-351
**Problem:** `handleCsvFile()` function reads the file as plain text (`reader.readAsText(file)`) and splits by newlines/commas. The button label says "Importer (.csv, .xlsx)" and the file input accepts `.xlsx`, but `.xlsx` is a binary ZIP format that cannot be parsed this way.
**HTML:** `wizard.htmx.html` line 198-207: button says "Importer (.csv, .xlsx)", input accepts ".csv,.xlsx"
**Fix options:**
  - **Simple (recommended):** Change accept to `.csv` only and update button label to "Importer (.csv)"
  - **Complex (out of scope):** Add xlsx.js library for binary parsing
**Severity:** HIGH -- user-facing broken feature

## CSS Cleanup Strategy

### Step 0: Resolve Contradictions

**Priority A -- Remove duplicate definitions in pages.css:**
1. `.section-header` defined twice (lines 916 and 1277) -- keep the 14px version (wireframe density)
2. `.tab-toolbar` defined twice (lines 181 and 1284) -- keep the 14px version
3. `.page-header` in pages.css overrides app.css -- consolidate to one location

**Priority B -- Align gap values with wireframe:**
- Wireframe uses `gap: 10px` for grids consistently
- pages.css `.grid-2` uses `var(--space-5, 1.25rem)` = 20px -- too wide
- wizard.css `.grid-2` uses `12px` -- closer but still wrong
- Should either: (a) set pages.css to 10px to match wireframe, or (b) use a CSS var

**Priority C -- Remove wizard.css local re-definitions:**
- `.field-label`, `.field-input`, `.grid-2`, `.grid-4`, `.flex-between` are all defined in both wizard.css and other files
- These should use the global definitions unless wizard has specific needs
