## Summary

The wizard has four steps rendered as full `.wiz-step` cards inside `.wiz-content` (max-width 680px). Each step's visible content is measured from the source; Steps 1 and 3 are the most vertical-space-heavy and will require the most work. The `.step-nav` footer is already sticky (position: sticky; bottom: 0) inside the card and does NOT consume viewport scroll area — it sticks to the bottom of the card, not the viewport. This means the card itself must fit within the available viewport height.

**Viewport budget:** 1080px total − 56px app-header − ~50px stepper (padding + items) − 16px top margin − 16px bottom margin = approximately 942px for the card itself. The `.step-nav` footer is ~72px (padding: 1.25rem 2.5rem = 20px top + bottom + ~32px button height). The `.wiz-step-body` must therefore not exceed ~870px of rendered content. Currently Step 1 and Step 3 exceed this.

**Primary recommendation:** Apply 4 targeted interventions in wizard.css + wizard.htmx.html: (1) reduce `.wiz-step-body` padding from 2rem 2.5rem to 1.25rem 1.5rem, (2) reduce `.wiz-section` padding from 1.5rem 0 to 0.875rem 0, (3) convert identity + voting-rules rows to `.form-grid-2`, (4) collapse the "Règles de vote" section behind an advanced toggle. No JS architecture changes needed.

## Architecture Patterns

### Pattern 1: Viewport-fit Card Body

The `.wiz-step` card must be configured so it never overflows the viewport. The approach: set a `max-height` on `.wiz-step` and let `.wiz-step-body` scroll internally if content truly overflows (fallback), while the primary goal is that content fits without needing to scroll.

```css
/* wizard.css — updated .wiz-step */
.wiz-step {
  max-height: calc(100vh - 56px - 58px - 40px); /* header + stepper + wiz-content padding */
  display: flex;
  flex-direction: column;
}

.wiz-step-body {
  flex: 1;
  overflow-y: auto; /* already set — keeps as fallback */
  padding: 1.25rem 1.5rem; /* was: 2rem 2.5rem — saves 28px top+bottom */
}
```

### Pattern 2: Section Padding Reduction

```css
/* wizard.css — updated .wiz-section */
.wiz-section {
  padding: 0.875rem 0; /* was: 1.5rem 0 — saves ~13px per section × 3 = ~39px on Step 1 */
}

.wiz-section-title {
  margin-bottom: 0.75rem; /* was: 1.25rem — saves ~8px */
}
```

### Pattern 3: 2-Column Field Groups Using .form-grid-2

The `.form-grid-2` utility already exists in design-system.css with `grid-template-columns: 1fr 1fr; gap: var(--space-field) 12px`. Use it for:

**Step 1 — "Identité" section:** Type + Date side by side, Heure below (or Type + Date + Heure via a 3-col grid). Since `.row` already exists but has alignment issues with the time input, convert to a wrapper approach.

Recommended groupings (Claude's discretion):
- Pair 1: `wizType` (select) + `wizDate` (date input) — both compact width, natural pairing
- Pair 2: `wizTimeHH/MM` stays in its own row (time input already has custom markup)
- Pair 3 (Lieu section): `wizPlace` + `wizAddr` already in `.row` — convert to `.form-grid-2`
- Pair 4 (Règles section): `wizQuorum` + `wizDefaultMaj` — the biggest win, saves ~80px

**Step 3 — "Add resolution" panel:** `resoMaj` + "Vote secret" chips already in `.row`.

### Pattern 6: Smart Defaults in wizard.js

Add to `init()` after `restoreDraft()` check:

```javascript
function applySmartDefaults() {
  // Only apply if no draft exists (first visit)
  if (localStorage.getItem(DRAFT_KEY)) return;

  // Today's date
  var dateEl = document.getElementById('wizDate');
  if (dateEl && !dateEl.value) {
    var today = new Date();
    var y = today.getFullYear();
    var m = String(today.getMonth() + 1).padStart(2, '0');
    var d = String(today.getDate()).padStart(2, '0');
    dateEl.value = y + '-' + m + '-' + d;
  }

  // Default time: 18:00 (typical evening AG)
  var hhEl = document.getElementById('wizTimeHH');
  var mmEl = document.getElementById('wizTimeMM');
  if (hhEl && !hhEl.value) hhEl.value = '18';
  if (mmEl && !mmEl.value) mmEl.value = '00';

  // Default title: "AG Ordinaire du [date]" (auto-fill on date change)
  // Don't pre-fill title — user sees blank and knows to fill it
}
```

## Common Pitfalls

### Pitfall 1: `.step-nav` Position Sticky and Max-Height Interaction

**What goes wrong:** If `.wiz-step` gets `max-height` without `overflow: hidden`, and `.wiz-step-body` does not have `overflow-y: auto`, the sticky `.step-nav` may not stick correctly inside a constrained parent.

**How to avoid:** Keep `overflow-y: auto` on `.wiz-step-body`. Set `overflow: hidden` on `.wiz-step` (not `auto`). The sticky footer works because it is a sibling of `.wiz-step-body`, not inside it. The existing structure is correct — only change padding values.

**Warning sign:** Footer disappears or stops sticking.

### Pitfall 2: `.form-grid-2` Inside `.wiz-step-body` Width Constraint

**What goes wrong:** `.wiz-content` has `max-width: 680px`. Inside `.wiz-step-body`, there is a rule: `.wiz-content .form-group:not(.form-grid-2 .form-group) input, select, textarea { max-width: 480px; }`. If `.form-grid-2` is used with `.field` (not `.form-group`), the `:not()` selector does not apply and fields may get restricted.

**Context:** The wizard uses `.field` class (not `.form-group`). The `max-width: 480px` rule targets `.form-group`. No conflict — `.form-grid-2` can be used with `.field` children freely.

**How to avoid:** Use `.form-grid-2` as a wrapper around `.field` elements (not `.form-group`). This matches existing Phase 41.1 patterns.

### Pitfall 3: `<details>/<summary>` Animation

**What goes wrong:** `<details>` has no native open/close animation — it snaps.

**How to avoid:** If snap is acceptable (it is, for an advanced-options section), use as-is. For a minimal slide-down, add:

```css
.wiz-advanced-toggle .wiz-section-inner {
  overflow: hidden;
  animation: wizFadeIn 150ms ease-out; /* reuse existing keyframe */
}
```

The `wizFadeIn` keyframe already exists in wizard.css (opacity 0→1, translateY 8px→0). This gives a subtle fade-in without JavaScript.

### Pitfall 4: Field Class Naming (`.field` vs `.form-group`)

**What goes wrong:** wizard.htmx.html uses `.field` + `.field-label` + `.field-input` (wizard-local classes). design-system.css uses `.form-group` + `.form-label` + `.form-input`. Mixing them causes mismatched styles.

**How to avoid:** Keep all wizard fields as `.field` / `.field-label` / `.field-input`. Only use `.form-grid-2` as the layout wrapper (it is layout-only, no child-class assumptions beyond gap and grid).

This was already established in Phase 41.1: "wizard `.field-input` class name preserved in HTML — token values aligned in wizard.css instead of renaming HTML classes."

### Pitfall 5: Draft Restoration and Smart Defaults

**What goes wrong:** Smart defaults run after `restoreDraft()`. If a draft exists with empty values (user previously cleared fields), defaults would NOT apply (correct). But if defaults check `!value`, they incorrectly skip when the draft has an empty string.

**How to avoid:** In `applySmartDefaults()`, guard with `if (localStorage.getItem(DRAFT_KEY)) return;` — this skips defaults entirely if any draft exists, regardless of field values. Only a truly fresh visit (no draft) gets defaults.

### Pitfall 6: `.wiz-member-add-form` Regression

**What goes wrong:** The member add form uses `.row` for layout with a `.field-action` at the end. If the form is compacted with `.form-grid-2` instead of `.row`, the "Ajouter" button loses its `align-self: flex-end` alignment.

**How to avoid:** Keep `.wiz-member-add-form` using `.row` layout (not `.form-grid-2`). Only use `.form-grid-2` for two-equal-width fields. The member form has 3 fields + 1 button, which is a flex-row use case, not a 2-column grid.

## Hub UX Improvements

### Current Hub Flow Analysis

The hub (`hub.htmx.html`) uses a two-column layout: left stepper + right action card. The "launch session" flow is: Hub → checklist satisfied → click "Ouvrir la séance" (in `meetingBarActions`) → redirect to operator.

**Identified friction:**
1. The `#hubDetailsToggle` is expanded to see KPIs — always-hidden details reduces information density
2. The convocation send button (`#btnSendConvocations`) is buried in `hubConvocationSection` which is `style="display:none"` until JS shows it
3. The "Modifier" button in the header links to `/wizard.htmx.html` without preserving the meeting ID — user must find the meeting again

**Improvements (within scope):**
- Remove the `hub-details-toggle` collapsing behavior — always show KPI strip (it's only 4 small cards, ~120px)
- Change the details toggle from "show details" to only collapsing the 2e convocation warning and documents section (less important content)
- Hub guidance panel: the `.ctx-panel`-style guidance should be more scannable — use an icon-row checklist pattern instead of paragraph text

### Operator Quick Actions

The operator page uses a `mode-switch` between "Préparation" and "Exécution" modes. Key workflow improvement: the primary CTA button text is dynamically set. No structural changes needed here — the current design is already optimized. The main gap is visual feedback on SSE connection state (already present via `op-sse-indicator`).

**One improvement:** The `btnBarRefresh` button should be hidden when SSE is live (no manual refresh needed). This is a JS-only change in operator.js that is out of scope for this phase's CSS/HTML focus.

### Settings Page

Settings uses a sidenav tab pattern. The main density issue is that each section card has `padding: var(--space-card) = 24px` internally. No structural changes needed — the Phase 40 redesign already handled settings density.

## Open Questions

1. **Whether to keep the `alert-warn` (convocation deadline) on Step 1**
   - What we know: The alert is ~56px and provides legal guidance (21 days — art. 9)
   - What's unclear: Is this essential on first fill, or better shown in Hub after creation?

2. **Whether Step 2 needs a compacted form by default**
   - What we know: Step 2 is ~618px which is borderline but fits the budget
   - What's unclear: With members added (rows rendered), it overflows
   - Recommendation: The member list renders below the fold — this is acceptable scroll behavior for Step 2. The initial state must fit; scrolling after data entry is expected UX for list management. Keep Step 2 as-is structurally; only apply padding reductions.

   - What we know: Each `.ctx-panel` consumes ~80-92px
   - Recommendation: Remove from initial view. Move guidance into the step-level help popover already present in the header. The `ag-popover` on the "Aide" button in the header is per-step and already contains step guidance.
