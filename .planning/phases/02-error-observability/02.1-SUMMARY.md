# Phase 2 Plan 02.1 — Migrate `business_error` Summary

## Audit findings

Initial CONTEXT D-02 hypotheses (`quorum_not_met`, `meeting_state_invalid`, `vote_already_closed`) were **rejected** by the audit:

- `quorum_not_met` already exists in the dictionary (line 89) and is unrelated to `business_error` callers.
- No `business_error` caller relates to a quorum or to a closed vote — every single caller comes from meeting controllers, not ballots/motions.

The 10 migratable callers grouped cleanly into three semantic buckets centred on the **meeting** domain:

| Bucket | Callers | Code | HTTP |
|---|---|---|---|
| A — Workflow / transition failure | 4 (transition, readyCheck, resetDemo, validate) | `meeting_transition_failed` | 400 |
| B — Lifecycle CRUD failure | 2 (update, deleteMeeting) | `meeting_operation_failed` | 409 / 400 |
| C — State / read query failure | 4 (status, statusForMeeting, summary, stats) | `meeting_state_read_failed` | 400 |
| D — Generic fallback (kept) | 1 (`AbstractController::handle` catch-all) | `business_error` | 400 |

See `02.1-AUDIT.md` for full caller-by-caller mapping.
