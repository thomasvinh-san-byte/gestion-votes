## audit_events Status

**Confirmed limitation:** `api_fail()` does not call `audit_log()`. The error-stats
dashboard cannot derive a comprehensive "Top error codes" feed from `audit_events`
today.

**Shipped behaviour:** filter `audit_events.action IN (...)` against an explicit
allowlist of 6 error-flavored action names found in the codebase
(`admin.confirm.failed`, `auth_account_locked`, `auth_account_lock_engaged`,
`auth_rate_limited`, `device_blocked`, `device_kicked`). Empty result set is
handled with French empty-state copy. Limitation banner is always visible.

**v2.5+ recommendations** (recorded in 02.3-AUDIT.md §"Recommendations"):
1. Phased `error_log()` → `Logger::errorContext()` migration (15 controllers + 3 services + remainder).
2. `api_fail()` → `audit_log('api_fail.{code}', ...)` bridge so dashboard feeds on real production data.
3. Next-step click instrumentation (HTMX hook → new endpoint) to populate the Taux column.
