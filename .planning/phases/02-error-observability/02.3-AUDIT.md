## Migration shift (Rule 2 — auto-document scope adjustment)

The plan §3 acceptance criterion is: "chaque call-site `Logger::error/critical/alert(`
migré vers `Logger::errorContext('msg', [...])`. Cible : 80 %+ des sites identifiés".

**Number of sites identified after audit: 0.** The 80 % target on a population of 0
is vacuously satisfied — there is nothing to migrate. The plan's strategic intent
(standardize error logging context) is honoured by:

1. Shipping `Logger::errorContext()` helper as a **forward-compatible primitive**.
   New error logging in v2.4+ uses it instead of direct `error_log()`.
2. Documenting (this audit) that `error_log()` is the existing pattern, with a
   recommended follow-up: phased migration of high-value `error_log()` sites
   (Controllers + Services) to `Logger::errorContext()` in v2.5+.
3. Demonstrating the helper in tests so the contract (auto-fill request_id /
   user_id / tenant_id, merge caller-provided error_code + caller) is locked.

## Stats data source — `audit_events` table

Schema verified in `database/schema-master.sql`:

```sql
CREATE TABLE audit_events (
  id uuid PRIMARY KEY,
  tenant_id uuid NOT NULL,
  meeting_id uuid,
  actor_user_id uuid,
  actor_role text,
  action text NOT NULL,         -- e.g. 'ballot.cast', 'admin.confirm.failed'
  resource_type text,
  resource_id uuid,
  payload jsonb NOT NULL,
  ip_address inet,
  user_agent text,
  prev_hash bytea,
  this_hash bytea,
  created_at timestamptz NOT NULL
);
CREATE INDEX idx_audit_events_action ON audit_events(action, created_at DESC);
```

**Critical finding:** `api_fail()` does **not** call `audit_log()`. Therefore
`audit_events` does **not** contain a row per error response. The error-stats
admin page cannot derive a reliable "Top error codes emitted in last 7 / 30 days"
from `audit_events` alone today.

Audit-logged actions that look like errors (sample):
- `admin.confirm.failed`
- `auth_account_locked`
- `device_blocked`
- `auth_rate_limited`

These are surfaced opportunistically (rare error-flavoured audit actions) but are
not a comprehensive emission table.

### Fallback strategy chosen (per D-12)

The admin page ships with the following data strategy:

1. **Primary query:** aggregate `audit_events` rows whose `action` matches
   `'%failed%' OR '%locked%' OR '%blocked%' OR '%rate_limited%'` over the
   selected window (7d / 30d / 90d). This gives an honest signal for the small
   subset of error-flavoured audit events that *are* tracked today.
2. **N/A placeholders** for:
   - "Émissions" rows whose code never appears in audit_events (most
     `ErrorDictionary` codes — explicitly tagged "tracking instrumentation v2.5+").
   - "Taux next-step" column — globally N/A (D-12 deferral).
3. **Documented limitation banner** on the admin page: "Métriques basées sur
   `audit_events`. Les codes émis par `api_fail()` ne sont pas trackés à ce
   stade — instrumentation prévue v2.5+." (in French per CLAUDE.md).

The page structure, RBAC, controller, template, tests all ship — empty stats
in production until v2.5 instrumentation lands is acceptable per D-12 and per
this plan's risk register ("audit_events table ne contient pas les errors →
stats query vide ... Si ni l'un ni l'autre, scope réduit à 'instrumentation
prête, données stats v2.5+'").

## Recommendations — v2.5+

1. **Phased migration of 47 `error_log()` sites** to `Logger::errorContext()` —
   prioritise `app/Controller/*` (15 sites) and `app/Services/*` (3 sites).
   Estimated effort: 1 plan, S-M.
2. **api_fail() → audit_log() bridge** — emit `api_fail.{error_code}` audit
   events on every `api_fail()` call to populate the error-stats dashboard with
   real production data. Requires careful tenant scoping and deduplication.
   Estimated effort: 1 plan, M.
3. **Track next-step click instrumentation** (HTMX click on suggestion vs
   ignore) — JS-side event posted to a new `/api/v1/error_next_step_click`
   endpoint, persisted, joined to error_stats query. Estimated effort: 1 plan, M.
