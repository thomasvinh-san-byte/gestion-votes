# Phase 2 Plan 02.2 — Debounce SSE empty-state handlers Summary

Guard the `<ag-integrity-modal>` `attributeChangedCallback` re-render path with a >=250ms debounce via a reusable `window.AgSseDebounce` utility, so a burst of attribute mutations (the channel an SSE emitter would use to push live integrity-chain events) produces exactly one render. The dashboard hero card live arm of the requirement is deferred to v2.5+ because no SSE emitter is wired to it today (see scout discovery).

## Implementation summary

### sse-debounce utility

`public/assets/js/utils/sse-debounce.js` exposes:

```js
window.AgSseDebounce.create(target, renderFn, getMs)
```

- Vanilla `setTimeout` / `clearTimeout`, single timer per closure (one per element instance).
- `target` (HTMLElement) receives `data-render-count` incremented after each effective render — single read/write avoids DOM mutation observer flake.
- `getMs` is invoked **on every event** (not captured at creation time). This makes the override live: `setAttribute('data-sse-debounce-ms', '500')` takes effect on the next event without re-creating the handler.
- ES module + classic script + Node compatible (`module.exports` + `globalThis.AgSseDebounce`).
- Verified manually: 5 calls within 100ms (debounce 50ms) produce `renderCount === 1` and `data-render-count === '1'` (run in node before commit).

### ag-integrity-modal changes

- `attributeChangedCallback` now distinguishes initial render (synchronous, preserves first paint) from subsequent mutations (debounced burst channel).
- Lazy-created `_debouncedRender` per instance, captured on first post-initial mutation. Uses `_debounceMs` getter that re-reads `data-sse-debounce-ms` live.
- Graceful fallback path if `window.AgSseDebounce` is unavailable at upgrade time (synchronous render + manual `data-render-count` increment) — keeps the contract stable even if the utility script fails to load.
- `audit.htmx.html` and `report.htmx.html` now include `<script src="/assets/js/utils/sse-debounce.js" defer></script>` **before** the modal script so the namespace is ready at element upgrade.

### E2E spec — sse-burst-idempotency.spec.js

Test 1 (`5 attribute mutations in 100ms produce exactly 1 render`):
- Logs in as admin, opens `/audit.htmx.html`, asserts the modal element + utility are present.
- Resets `data-render-count` to `'0'` (initial connectedCallback render is synchronous, not debounced).
- Schedules 5 `setAttribute('data-events', JSON.stringify([...]))` calls 20ms apart.
- Waits 450ms (>= 100ms burst + 250ms debounce + 50ms margin) and asserts `data-render-count === 1`.

Test 2 (`data-sse-debounce-ms override 500ms`):
- Sets the override to `'500'`, fires one mutation, asserts `count === 0` at 300ms (between default 250 and override 500), then `count === 1` at 600ms.

Both tagged `@regression` so they join the standard regression run.

## Reusable pattern

**API**

```js
const debounced = window.AgSseDebounce.create(
  /* target */ this,             // HTMLElement that receives data-render-count
  /* renderFn */ () => this._render(),
  /* getMs */ () => this._debounceMs   // re-read each event for live tuning
);
debounced(); // call on each SSE event / attribute mutation
```

**Where it is used today**

1. `<ag-integrity-modal>` — `attributeChangedCallback` post-initial path (`public/assets/js/components/ag-integrity-modal.js`).

**Where it should be used next (v2.5+ candidates)**

- Dashboard hero card live channel, when an SSE emitter is wired to push live meeting status to `#heroCard`. Add `data-sse-debounce-ms` on the section, route the SSE handler through `AgSseDebounce.create(heroCard, renderHero, () => parseInt(heroCard.getAttribute('data-sse-debounce-ms') || '250', 10))`.
- Any future custom element rebound to a high-frequency SSE topic where a single render per burst is desirable.

**When NOT to use it**

- Bas frequence updates (cockpit `<ag-health-bar>` quorum-state mirror in `operator-realtime.js`) — already proven OK without debounce in v2.3 P1, adding it would only delay user feedback.
- Real-time inputs (typing, sliders) where 250ms feels laggy. Debounce there should be tuned per UX, not via this helper.

## Deviations from Plan

### Auto-adjusted scope (Rule 3 — auto-fix blocking issues)

**1. [Rule 3 - Scope] Hero card live debounce deferred to v2.5+**

- **Found during:** Task 1 (audit / scout).
- **Issue:** The plan assumed a hero card live SSE handler exists in `pages/dashboard.js` or inline in `dashboard.htmx.html`. Audit found none — the dashboard uses a single `Utils.apiGet('/api/v1/dashboard')` with no streaming channel. There is no SSE handler to debounce.
- **Fix:** Documented the discovery in `02.2-SCOUT.md`; reduced AC 2 to deferred and AC 4 from 3 tests to 2 tests. Kept the reusable utility in place so wiring the debounce in v2.5+ stays a small change (one line + one attribute).
- **Files modified:** scope of this plan only — no code change beyond what is documented as in-scope.
- **Commit:** `fa029da` (scout) documents the discovery; subsequent commits respect the reduced scope.

**2. [Rule 2 - Defensive] Initial render kept synchronous**

- **Found during:** Task 2 (debounce implementation).
- **Issue:** Naively routing every `attributeChangedCallback` through the debounce would delay the first paint by 250ms — a visible regression on the audit page where the modal is mounted with empty initial state and `connectedCallback` triggers a render through the same path.
- **Fix:** Guarded the debounced branch with `if (!this.hasAttribute('data-rendered')) { this._render(); return; }`. Initial render is synchronous; only post-initial mutations (the realistic SSE burst channel) flow through the debounce.
- **Files modified:** `public/assets/js/components/ag-integrity-modal.js`.
- **Commit:** `0e38e0b`.

**3. [Rule 2 - Defensive] Fallback path when utility script missing**

- **Found during:** Task 2.
- **Issue:** If `sse-debounce.js` failed to load (CSP / network / cache miss), the modal would silently never re-render after first paint — worse than no debounce.
- **Fix:** Added a synchronous-render fallback that still increments `data-render-count` so the contract observed by E2E and consumers remains stable.
- **Commit:** `0e38e0b`.

If either test fails:
- Test 1 fails with `count === 5` → debounce is not wired (utility not loaded, or modal bypasses it). Check `<script src="/assets/js/utils/sse-debounce.js" defer>` is present **before** `ag-integrity-modal.js` in `audit.htmx.html`.

- Test 2 mid-assertion fails with `count === 1` at 300ms → override not honored, `getAttribute('data-sse-debounce-ms')` returning `null` or stale. Check the `_debounceMs` getter is re-read per event (not captured).
