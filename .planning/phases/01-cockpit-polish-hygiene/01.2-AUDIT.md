## Contrast a11y considerations

Migration targets:

| Token | Light mode (light bg) | Dark mode (dark bg) | WCAG AA text 4.5:1 | WCAG AA UI 3:1 |
|-------|-----------------------|---------------------|--------------------|----------------|
| `--color-warning` (oklch ~0.62 0.16 80) on `--color-surface` | ratio ~4.7:1 (text-text small) | ratio ~5.2:1 | ‚úÖ text & UI | ‚úÖ |
| `--color-warning-subtle` background + `--color-warning-text` foreground | ratio ~7:1 | ratio ~6.5:1 | ‚úÖ | ‚úÖ |
| `--color-text-muted` (oklch ~0.55 0.012 260) on surface | ratio ~4.6:1 | ratio ~5.0:1 | ‚úÖ borderline | ‚úÖ |
| `--color-text` on surface | ratio >=10:1 | ratio >=10:1 | ‚úÖ | ‚úÖ |

For dot/segment backgrounds (lines 422, 692, 949) we choose `--color-text-muted` rather than `--color-text` to avoid creating a visual hierarchy of "lots of black blobs" that draws attention back to the rejected state. The rejected state should be **factually present, not visually loud**.

For numeric counters (`.op-vote-against .op-vote-count`, `.exec-kpi.against`, `.vote-results-live .against`) we use `--color-text` because these are foreground readable text; muted would be too dim for live numerical data.

Contrast values above are computed from OKLCH lightness deltas; manual verification on dev machine recommended once palette is shipped.
