# Plan 01.1 — Declutter cockpit ≤25 cliquables

## Goal

Réduire le nombre d'éléments cliquables visibles dans le cockpit opérateur en mode `exec` de **~91 baseline** à **≤25 stricts** sur viewport ≥1024px, en appliquant 3 techniques combinées (disclosure + contextual-only + sub-tab collapse), tout en préservant 100 % des actions critical-path en accès direct.

Source : COCKPIT-V24-01 (REQUIREMENTS.md), Schoger S-1 (Refactoring UI Ch.10), CONTEXT.md décisions D-01..D-06.

## Context anchors

- **Baseline mesurée** : `public/operator.htmx.html` contient 82 `<button>` + 9 anchors stylisés btn = **91 cliquables** (avant audit fin)
- **Cible** : ≤25 sur viewport ≥1024px en mode exec
- **Critical-path sacrés** (D-06) : Lancer/Fermer scrutin (`#opBtnLaunchVote`, `#opBtnCloseVote`, `#opBtnToggleVote`), Proclamer (`#opBtnProclaim`), Quorum override (à identifier), liens sécurité racine (logout, retour dashboard)

## Acceptance Criteria (what must be TRUE)

1. **Audit baseline livré** dans `01.1-AUDIT.md` (artifact dédié) : grep + classification de chaque cliquable visible mode exec en 3 buckets : `critical-path` (always-visible), `contextual-only` (apparait selon état), `disclosed` (masqué derrière "Plus d'actions"). Total = baseline exact.
2. **Compte ≤25** vérifié sur 3 états UI principaux : `idle` (aucun vote actif), `voting` (vote en cours), `proclaiming` (post-fermeture). Mesure via `page.locator(...).count()` Playwright.
3. **Disclosure pattern** implémenté avec `<details class="op-action-disclosure">` natif (pas de JS custom — pattern accessible Tab+Enter par défaut) ou `<ag-action-disclosure>` custom element si besoin de state cross-page. État ouvert/fermé persisté via `localStorage('agOperatorDisclosure')`. Bouton "Plus d'actions" comporte un compteur (`(N+)`).
4. **Contextual-only** : matrice action × état UI documentée dans `01.1-AUDIT.md`. Au moins ces règles appliquées :
   - `#opBtnCloseVote` invisible quand `data-vote-state !== "open"`
   - `#opBtnProclaim` invisible quand `data-vote-state !== "closed"` ou pas de résultat à proclamer
   - `#opBtnLaunchVote` invisible quand un vote est en cours (`data-vote-state === "open"`)
   - Boutons "Délégations / Procurations import" invisibles en mode `exec` (mode `setup` only)
5. **Sub-tab collapse** : l'audit identifie les top 3 sub-tabs visibles + le reste replié derrière disclosure "Plus d'onglets". Top 3 nommés explicitement dans le PLAN avec rationale (fréquence d'usage estimée).
6. **Critical-path préservé** : tous les boutons sacrés (D-06) restent visibles dans tous les états UI. Vérifié explicitement par 1 test Playwright dédié.
7. **Régression nulle** sur `cockpit-keyboard-shortcuts.spec.js` + `critical-path-operator.spec.js` — restent verts.
8. **Capture before/after** : 2 screenshots `cockpit-before.png` + `cockpit-after.png` (mode exec, viewport 1280×720) dans `01.1-SCREENSHOTS/`. Diff visuel commenté dans `01.1-SUMMARY.md`.
9. **Test gardien permanent** : `tests/e2e/specs/cockpit-button-count.spec.js` créé. Compte ≤25 sur 3 états UI. Échoue si régression future.

### Task 2 — Implémenter contextual-only (HTML data-attributes + CSS)

Sur `public/operator.htmx.html` :
- Ajouter `data-vote-state="idle|open|closed"` sur container racine `#viewExec`
- Wrapper boutons contextuels avec attribut conditionnel : `data-show-when="vote-state:closed"`
- CSS rules dans `public/assets/css/operator.css` ou nouveau `operator-declutter.css` :
  ```css
  [data-show-when="vote-state:open"] { display: none; }
  [data-vote-state="open"] [data-show-when="vote-state:open"] { display: revert; }
  /* etc. pour closed, idle */
  ```
- Mise à jour `data-vote-state` faite par `operator-realtime.js` quand SSE event arrive (pattern déjà établi pour quorum-state mirror v2.3 P1)

### Task 3 — Implémenter disclosure "Plus d'actions"

Dans `public/operator.htmx.html` mode exec, regrouper les boutons `disclosed` :

```html
<details class="op-action-disclosure" data-persist="agOperatorDisclosure">
  <summary class="op-action-disclosure__trigger" aria-haspopup="true">
    Plus d'actions <span class="op-action-disclosure__count">(N)</span>
  </summary>
  <div class="op-action-disclosure__content">
    <!-- boutons disclosed ici -->
  </div>
</details>
```

CSS dans `public/assets/css/components/op-action-disclosure.css` :
- `summary` stylisé comme bouton secondaire
- `[open]` state visible (pas de transition initiale, instant — D-discrétion)
- Compteur calculé en JS au load

JS dans `public/assets/js/pages/operator-disclosure.js` (nouveau, sentinel pattern) :
- Persist open/closed dans localStorage
- Restore on page load
- Compte enfants dynamiquement → injecte dans `.op-action-disclosure__count`

## Risks & Mitigations

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Régression sur Lancer/Fermer (sacred path) | M | H | Test critical-path explicit + manual smoke avant commit |
| `data-vote-state` mirror SSE casse le quorum-state mirror | M | M | Pattern identique v2.3 P1 (operator-realtime.js `_setHb()`) — réutiliser, pas réécrire |
| Compte ≤25 atteint en trichant (cacher en CSS sans `display:none`) | L | M | Test compte `:visible` Playwright (true visibility, pas `visibility:hidden`) |
| Disclosure inaccessible clavier | L | H | `<details>` natif est Tab+Enter accessible par défaut — pas de risque si on évite custom JS |
| Sub-tab collapse casse `operator-tabs.js` localStorage de tab actif | M | M | Lire `operator-tabs.js` avant modif, vérifier que les tabs cachés ne sont pas re-sélectionnés au reload |
