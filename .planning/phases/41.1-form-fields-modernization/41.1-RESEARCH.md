## Summary

The codebase uses two parallel field-styling systems that must be reconciled: `.field-input` / `.field-label` (wizard.css, scoped under `.wiz-step-body`) and `.form-input` / `.form-label` / `.form-group` (design-system.css, used on settings, admin, members). The design-system base is technically correct (36px height, 6px radius, double-ring focus shadow) but several polish details are missing: placeholders are not italic, select dropdowns have no custom chevron, `.form-label` uses `color: --color-text` instead of `--color-text-secondary`, and the form-group gap is `--space-2` (8px) not the canonical `--space-field` (16px). Wizard fields have their own more-polished scoped styles (1.5px border, 8px radius, surface background instead of surface-raised) that diverge from the design-system contract.

The wizard `max-width: 480px` cap on fields inside `.wiz-content` is a blunt layout constraint that prevents proper 2-column grids for short fields (date + time). Settings uses a `.settings-form-grid` with `auto-fill minmax(220px, 1fr)` but no explicit 2-column enforcement. Members' create-form uses `grid-template-columns: 2fr 1.5fr 1fr auto auto` which is correct in structure but has no gap harmonization. Admin forms use plain `.form-group` stacks without grid helpers.

**Primary recommendation:** Patch design-system.css for global field polish (placeholder italic, label color, select chevron, textarea min-height, form-group gap), then add scoped 2-column grid utilities and align wizard `.field-input` to the design-system contract.

### Anti-Patterns to Avoid

- **Using `.form-input` class on `<textarea>` directly:** The `.form-input` rule lacks `resize: vertical` and has no `min-height` override. Use `.form-textarea` class on `<textarea>` elements, or add `select.form-input, textarea.form-input` alias rules to `.form-textarea` block.
- **Overriding focus with `outline: none` without replacement:** Never remove outline without the double-ring shadow replacement. The DS already handles this ŌĆö don't add `outline: none` in page CSS.
- **`.wiz-content` max-width cap on form groups:** `max-width: 480px` on `.wiz-content .form-group input/select/textarea` prevents 2-col grids. Remove the cap from the flex/grid children ŌĆö apply max-width to the grid container instead.
- **Hardcoded gap values:** All field spacing must use `--space-field` (16px) and the 12px horizontal constant must be documented, not repeated.

## Common Pitfalls

### Pitfall 1: `.form-input` on `<select>` Has No Chevron
**What goes wrong:** Browser-default arrow appears on some OS/browsers; looks inconsistent and not branded.
**Why it happens:** `.form-input` has no `appearance: none` or custom background-image.
**How to avoid:** Add `appearance: none` + SVG chevron background to a combined `select.form-input, .form-select` rule in design-system.css.
**Warning signs:** Native select arrow visible on Chrome/Windows or Safari/macOS.

### Pitfall 2: Two Parallel Field Class Systems
**What goes wrong:** Wizard uses `.field-input` / `.field-label` (1.5px border, 8px radius, surface bg). Settings/admin/members use `.form-input` / `.form-label` (1px border, 6px radius, surface-raised bg). If you change design-system.css, wizard fields are unaffected unless explicitly updated.
**Why it happens:** Wizard was styled first with scoped custom classes; DS classes were standardized later.
**How to avoid:** After patching DS classes, update wizard.css to point `.field-input` styles at the same token values, or add `.field-input { @extend .form-input }` equivalents.
**Warning signs:** Wizard fields look different from settings fields after DS patch.

### Pitfall 3: `form-group` Gap Still 8px After Patch
**What goes wrong:** `.form-group { gap: var(--space-2) }` = 8px. Changing to `--space-field` (16px) affects label-to-input gap (label + input inside same form-group), not the spacing between form-groups.
**Why it happens:** `gap` inside a flex-column form-group controls labelŌåÆinput spacing, not form-groupŌåÆform-group spacing. The inter-group spacing is controlled by the parent container's gap or margin.
**How to avoid:** Keep `form-group` internal gap at `var(--space-2)` (8px, correct label-to-input distance). Add a new rule for the form-group stack: `.form-group + .form-group { margin-top: var(--space-field); }` OR ensure parent grid/flex uses `gap: var(--space-field)`.
**Warning signs:** All fields collapse together OR labels appear too far from their inputs.

### Pitfall 4: Dark Mode Select Chevron Invisible
**What goes wrong:** SVG chevron data-URI is encoded with a light-theme stroke color (`#857F72`). In dark mode the parchment color disappears against a dark background.
**Why it happens:** CSS `background-image` data-URIs cannot use CSS custom properties.
**How to avoid:** Add a `[data-theme="dark"]` override with a lighter stroke color (`#BCB7A5`).
**Warning signs:** Select dropdowns show no arrow in dark mode.

### Pitfall 5: `textarea.form-input` Misses Textarea Rules
**What goes wrong:** Settings and admin use `<textarea class="form-input">` not `<textarea class="form-textarea">`. The DS `.form-textarea` rule at line 1878 sets `min-height: 100px; resize: vertical` but `.form-input` does not.
**Why it happens:** Inconsistent class application in HTML templates.
**How to avoid:** Add `textarea.form-input { min-height: 100px; resize: vertical; }` to design-system.css, OR change HTML to use `.form-textarea` class (HTML changes are more surgical per-page).

### Pitfall 6: Wizard `max-width: 480px` Blocks 2-Column Grids
**What goes wrong:** `.wiz-content .form-group input, .wiz-content .form-group select, .wiz-content .form-group textarea { max-width: 480px; }` caps all fields. A 2-column grid inside wiz-content will have both columns max at 480px instead of splitting proportionally.
**Why it happens:** Phase 36 added the max-width to prevent fields stretching too wide in the 680px centered track.
**How to avoid:** Remove the selector targeting `input/select/textarea` individually and replace with a max-width on the `.form-group` wrapper inside non-grid contexts, or scope the cap to `.form-group:not(.form-grid-2 .form-group)`.

## Open Questions

1. **Label color: `--color-text-secondary` vs `--color-text-muted`**
   - What we know: CONTEXT says "13px semibold, --color-text-secondary" but in the DS, `--color-text-secondary` = `#151510` (near black) and `--color-text-muted` = `#857F72` (grey). Stripe-style forms use grey labels.
   - What's unclear: Did the user intend the grey (#857F72) or near-black (#151510)?
   - Recommendation: Use `--color-text-muted` for form-group uppercase labels (grey = recedes behind input value). Use `--color-text` for inline labels in toggle rows. This produces the Stripe-like hierarchy.

2. **Wizard `.field-input` vs `.form-input` ŌĆö full unification or alias?**
   - What we know: Wizard uses `.field-input` throughout HTML; changing HTML class names on ~30 wizard inputs is high-risk for JS selectors.
   - What's unclear: Whether any JS references `.field-input` class directly.
   - Recommendation: Keep `.field-input` class name in HTML. In wizard.css, rewrite `.field-input` to use the same token values as `.form-input` (same border width, radius, background, font-size) rather than renaming HTML. Lower-risk, same visual result.

3. **Textarea `field-sizing: content` (CSS auto-grow)**
   - What we know: `field-sizing: content` is supported in Chrome 123+ and Firefox 128+ (2024+), but Safari support is partial as of early 2026.
   - What's unclear: Whether the project targets Safari users.
   - Recommendation: Use `field-sizing: content` as progressive enhancement with `min-height` fallback: `min-height: 80px; field-sizing: content;` ŌĆö degrades gracefully to fixed min-height on Safari.
