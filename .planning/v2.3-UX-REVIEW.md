## 1. Ce que le milestone réussit déjà

- **Persistance > toast pour le quorum** (COCKPIT-02) : *"visibility of system state"* (Norman). Corrige une dette UX réelle de v2.0.
- **Sérif sur contenu / sans-sérif sur contrôles** (EDITORIAL-01/02) : signifiant fort de "document légal". À ne pas diluer.
- **Suppression de l'orbe login** (LOGIN-01) : la décoration marketing sur un outil crée de la dissonance. Bon réflexe.
- **"Next-step" sur ErrorDictionary** (ERR-01) : Norman pur — *"un bon message d'erreur dit comment récupérer"*.
- **North Star "screenshot rassurant"** : le bon test ultime pour un milestone visuel.

## 2. Trous critiques identifiés

### A. Phase 1 — Cockpit : 4 indicateurs égaux ≠ hiérarchie

COCKPIT-01 traite Quorum / SSE / Votants connectés / Résolution comme un seul niveau visuel. Or :

- **Quorum + Résolution** = info de décision (l'opérateur agit dessus).
- **SSE state + Votants connectés** = télémétrie système (info ambient).

Si tout est primaire, rien ne l'est. → amender COCKPIT-01 pour spécifier deux niveaux (`primary` typo dominante, `ambient` pill 12-13px discret).

### B. Phase 1 — Mental model "votants connectés"

"Connectés" répond à une question technique. Sous stress, l'opérateur veut savoir **"qui n'a pas encore voté ?"**. → amender COCKPIT-02 : indicateur principal = `Votes restants : 23 / 142`, "connectés" en survol/secondaire.

### C. Phase 1 — Anticipation manquante (prévention > détection)

COCKPIT-03 réagit *quand* le quorum bascule. Norman : un bon design préviendrait avant. → ajouter **COCKPIT-07** : état intermédiaire "quorum à risque" (ratio < 110 %) avec couleur warning douce.

### D. Phase 1 — Raccourcis clavier absents

Aucun requirement clavier sur le cockpit. Sous stress, l'opérateur veut `L` (Lancer), `F` (Fermer), `→` (motion suivante). → ajouter **COCKPIT-06**.

### E. Phase 2 — "Vérifier l'intégrité" doit enseigner

EDITORIAL-05 ouvre un modal sur la chaîne `audit_events`. 99 % des utilisateurs ne lisent pas un hash chain. Sans pédagogie, on montre du jargon — l'inverse de "rassurant". → amender EDITORIAL-05 : préambule français obligatoire.

### F. Phase 2 — Pill mono inline = piège typographique

EDITORIAL-04 met les numéros en pill monospace. OK en en-tête/tableau. Inline dans un paragraphe serif, ça casse le rythme. → préciser EDITORIAL-04 : pill *seulement* dans listes/headers, pas en flux.

### G. Phase 2 — `@media print` hors scope

`/audit`, `/report`, PV vont être imprimés. Aucune mention de styles print, page-break, masquage des contrôles. → ajouter **EDITORIAL-07**.

### H. Phase 3 — DASHBOARD-01 spécifie le combien, pas le quoi

"4 KPI → 3" sans dire lequel saute. → amender DASHBOARD-01 : le plan doit décider et justifier quel KPI a la moindre charge décisionnelle.

### I. Phase 3 — Imminence a des gradients (3 états, pas 2)

DASHBOARD-02 binaire (live ou <1h) écrase la nuance. → amender pour 3 états : ambient (<60min) / urgent (<5min) / live.

### J. Phase 3 — Empty state dashboard absent

Que voit l'opérateur quand aucune séance n'est planifiée ? Trou classique. → ajouter **DASHBOARD-04**.

### K. Phase 4 — ERR-02 teste la forme, pas le sens

"Ponctuation + verbe" laisse passer *"Erreur survenue, veuillez réessayer"*. → ajouter **ERR-03** : liste de phrases interdites (regex) dans le test.

Phrases bannies suggérées :

- `/réessayer\.?$/i`
- `/contactez (le|l')admin/i`
- `/erreur survenue/i`
- `/une erreur est survenue/i`

### L. Phase 4 — Prévention totalement absente

v2.3 traite *récupération* (ERR-01) mais jamais *prévention*. Exemple : `already_voted` ne devrait pas se produire si le bouton vote se désactive après soumission. → ajouter **ERR-04** : audit "5 erreurs les plus fréquentes — laquelle disparaît par contrainte UI ?".

### M. Phase 4 — Modal trigger affordance

MODAL-01/02 traitent le focus *interne*. Mais les boutons qui ouvrent les modales annoncent-ils qu'ils ouvrent un dialog ? → ajouter **MODAL-03** : `aria-haspopup="dialog"` + signifiant visuel (ellipsis ou icône) sur tous les triggers.

## 3. Méta — la story manque

Les 4 phases sont ordonnancées par "plus haute valeur d'abord" (logique d'exécution). Pour un milestone *design*, l'ordre par parcours utilisateur fait mieux ressortir le test "screenshot avant/après" :

```
Login (Phase 3) → arrive calme
  ↓
Dashboard (Phase 3) → s'oriente
  ↓
Cockpit (Phase 1) → conduit sous contrôle
  ↓
PV éditorial (Phase 2) → remet un document sérieux
  ↓
Erreurs/Modales (Phase 4) → récupère sans panique
```

**Décision retenue (à valider par l'utilisateur)** : on ne ré-ordonne pas les phases — l'ordre exécution reste pertinent et re-planifier ferait perdre du temps. Mais on ajoute un **panel screenshot transversal** comme gate de fin de milestone (success criterion explicite).

## 4. Success criterion ajouté au milestone

Le test ultime "screenshot rassurant" était dans la *goal* sans être encodé. → ajouté à ROADMAP.md milestone v2.3 :

> 3 observateurs internes notent les screenshots avant/après sur 5 dimensions sur échelle 1-5 :
> 1. *"Inspire confiance"*
> 2. *"Je comprends sans aide"*
> 3. *"Je ne suis pas surchargé"*
> 4. *"Je sais quoi faire ensuite"*
> 5. *"Ça a l'air légal"*
>
> Médiane doit gagner ≥ +1 point sur chaque dimension entre snapshot v2.2-merge et snapshot v2.3-merge. Capture obligatoire en fin de Phase 4.
